**Q1: Should we use a template engine (like Handlebars, EJS) or simple string replacement?**  
- [x] **a)** Use simple string replacement with `{{PLACEHOLDER}}` syntax ✅ **TRON DECISION**
  - **Refined:** Use Radical OOP pattern from UcpComponentSupport - Native `${this.attribute}` template literals with `this` context
  - Pro: No external dependencies, easier to maintain, faster runtime
  - Con: Limited to simple substitution, no loops/conditionals
- [ ] **b)** Use template engine (Handlebars/EJS)
  - Pro: More powerful, industry standard, handles complex logic
  - Con: External dependency, more complex, potential security issues
  
**Q2: Should HTML files be in `src/view/html/` or `dist/view/html/`?**  
- [x] **a)** Place in `src/view/html/` (source directory) ✅ **TRON DECISION**
  - Pro: HTML is source code, version-controlled, consistent with Web4
  - Con: TypeScript must read from src at runtime
- [ ] **b)** Place in `dist/view/html/` (build output)
  - Pro: Served from build directory, cleaner separation
  - Con: HTML is not build output, shouldn't be in dist
  
**Q3: Should we load HTML synchronously or asynchronously?**  
- [x] **a)** Load synchronously when needed ✅ **TRON DECISION**
  - **Refined:** No startup loading - templates loaded synchronously when methods are called
  - **Architecture:** Layers 2/3 are synchronous, async only in layer 4
  - Pro: Simpler code, fail fast if missing, templates ready immediately
  - Con: Blocks startup (negligible for small files)
- [ ] **b)** Load asynchronously
  - Pro: Non-blocking startup
  - Con: More complex, need to handle template not ready
  
**Q4: What placeholder syntax should we use?**  
- [ ] **a)** `{{PLACEHOLDER_NAME}}` (Handlebars/Mustache style)
  - Pro: Industry standard, readable, no HTML conflicts
- [x] **b)** `${PLACEHOLDER_NAME}` (Template literal style) ✅ **TRON DECISION**
  - **Refined:** Radical OOP with `this` context - templates access properties via `${this.property}`
  - Pro: Familiar to JavaScript developers, native JS feature
  - Con: Conflicts with actual template literals (not an issue in our use case)
- [ ] **c)** `%PLACEHOLDER_NAME%` (Environment variable style)
  - Pro: Simple, clear
  - Con: Less common, harder to spot
  
**Q5: Should we cache loaded templates or reload on every request?**  
- [ ] **a)** Cache at server startup (load once, reuse)
  - Pro: Better performance, simpler code
  - Con: Cannot update templates without restart
- [x] **b)** Reload on every request ✅ **TRON DECISION**
  - **Status:** Current implementation
  - Pro: Can update templates without restart
  - Con: Repeated file I/O, slower performance

**✅ What Worked Well:**
1. **Radical OOP Renderer Pattern:** UcpComponentSupport's `new Function('return \`...\`').call(this)` pattern is elegant and powerful
2. **Native Template Literals:** Using `${this.property}` instead of custom syntax leverages JavaScript's native features

4. **Synchronous Loading:** Keeping layers 2/3 synchronous simplifies code and follows Web4 architecture
5. **Source File Reading:** Templates are source code, not build artifacts - correct architectural decision

**⚠️ Challenges Overcome:**
1. **Path Resolution:** Initially read from `dist/view/html`, corrected to `src/view/html`
   - Learning: Runtime paths must account for compiled output location (`dist/ts/layer2`)
   - Solution: Use `../../../src/` to navigate from dist back to src

   - Learning: Radical OOP = leverage `this` context, not custom placeholder syntax
