## **🎯 HEADER**

### **Context:**
ONCE currently saves scenarios in a **legacy flat format** with `uuid`, `objectType`, `version`, `state`, and `metadata` at top level. The User component already uses the **Web4 Standard Format** with `ior`, `owner`, and `model`. To achieve **consistency across all Web4 components**, ONCE scenarios must be migrated to the same format.

### **Problem Statement:**
**Current ONCE Scenario Format** (Legacy):
```json
{
  "uuid": "11afd49a-cf3f-4451-9092-f51df374cde7",
  "objectType": "ONCE",
  "version": "0.3.21.1",
  "state": { ...serverModel... },
  "metadata": { ...created, modified, creator, tags... }
}
```

**Target Web4 Standard Format** (Used by User component):
```json
{
  "ior": {
    "uuid": "7e65139d-38cf-4f34-b769-0a86dd3a94e3",
    "component": "User",
    "version": "0.3.21.1"
  },
  "owner": "eyJ1c2VyIjoic3lzdGVtIiwiaG9zdG5hbWU...==",
  "model": { ...component-specific data... }
}
```

**Gap**: 
- ONCE scenarios lack `ior` (Interoperable Object Reference)
- ONCE scenarios lack `owner` (base64-encoded ownership metadata)
- ONCE scenarios mix component data (`state`) with metadata
- ONCE scenarios use `objectType` instead of `component`

### **Goal:**
Migrate ALL ONCE scenario files to Web4 Standard Format while:
1. ✅ Preserving all existing data (no data loss)
2. ✅ Maintaining backward compatibility during transition
3. ✅ Using `toScenario()` method (not manual JSON construction)
4. ✅ Implementing proper owner tracking
5. ✅ Following TRUE Radical OOP patterns

### **Success Criteria:**
1. ✅ All new ONCE scenarios use `{ior, owner, model}` format
2. ✅ `DefaultONCE.toScenario()` returns Web4 Standard Format
3. ✅ Existing scenarios can still be loaded (backward compatibility)
4. ✅ All tests pass without modification
5. ✅ Housekeeping works with new format
6. ✅ Git state clean after migration

**Phase 4: Update `ScenarioManager` with Backward Compatibility**
```typescript
// File: src/ts/layer2/ScenarioManager.ts

import { Scenario } from '../layer3/Scenario.interface.js';
import { LegacyONCEScenario } from '../layer3/Scenario.js';
import { isLegacyONCEScenario } from '../layer3/ScenarioTypeGuard.js';

async loadScenario(scenarioPath: string): Promise<Scenario> {
  const data = JSON.parse(readFileSync(scenarioPath, 'utf-8'));
  
  // ✅ Detect and migrate legacy format
  if (isLegacyONCEScenario(data)) {
    console.log(`🔄 Migrating legacy scenario: ${data.uuid}`);
    return this.migrateLegacyScenario(data);
  }
  
  // ✅ New format - return as-is
  return data as Scenario;
}

private migrateLegacyScenario(legacy: LegacyONCEScenario): Scenario {
  // Generate owner from metadata
  const ownerMetadata = {
    user: legacy.metadata.creator || 'system',
    hostname: legacy.metadata.host || 'localhost',
    utcTimestamp: legacy.metadata.created || new Date().toISOString(),
    uuid: legacy.uuid
  };
  
  const ownerBase64 = Buffer.from(JSON.stringify(ownerMetadata)).toString('base64');
  
  return {
    ior: {
      uuid: legacy.uuid,
      component: legacy.objectType,
      version: legacy.version
    },
    owner: ownerBase64,
    model: legacy.state  // Migrate state → model
  };
}
```

### **QA Decisions**

**Decision D1: Migration Timing**
When should existing scenario files be migrated?

**Option A:** Migrate all existing files immediately (rewrite on startup)
- ✅ **Pro**: All files in new format immediately
- ❌ **Con**: Risk of data loss if migration fails
- ❌ **Con**: Large filesystem operation on every startup

**Option B:** Migrate on-load (lazy migration)
- ✅ **Pro**: Safe - only migrates when accessed
- ✅ **Pro**: No bulk operation
- ❌ **Con**: Mixed format in filesystem during transition

**Option C:** Leave old files unchanged, only new saves use new format
- ✅ **Pro**: Zero risk - no file modifications
- ✅ **Pro**: Simple implementation
- ✅ **Pro**: Backward compatibility indefinitely
- ❌ **Con**: Mixed format persists

**RECOMMENDATION**: **Option C** - Safest approach, no data risk.

**Decision D2: Owner Metadata Source**
Where should owner data come from?

**Option A:** Use User component's `toScenario()` (like CLI does)
- ✅ **Pro**: Consistent with CLI pattern
- ✅ **Pro**: Proper User component integration
- ❌ **Con**: Requires User component to be initialized
- ❌ **Con**: Server scenarios might not have User context

**Option B:** Generate minimal owner from environment
- ✅ **Pro**: Always available (no dependencies)
- ✅ **Pro**: Simple implementation
- ❌ **Con**: Less rich owner data
- ❌ **Con**: Not using User component

**Option C:** Hybrid - Use User if available, fallback to environment
- ✅ **Pro**: Best of both worlds
- ✅ **Pro**: Graceful degradation
- ❌ **Con**: Two code paths to maintain

**RECOMMENDATION**: **Option C** - Hybrid approach with graceful fallback.

**Decision D3: Backward Link for toScenario Access**
How should `ServerHierarchyManager` access `DefaultONCE.toScenario()`?

**Option A:** Add backward link `this.once: DefaultONCE` to ServerHierarchyManager
- ✅ **Pro**: Clean OOP pattern (already used for ScenarioManager)
- ✅ **Pro**: Direct method access
- ❌ **Con**: Circular reference (but acceptable in OOP)

**Option B:** Inject `toScenario` function as parameter
- ✅ **Pro**: Functional approach, no circular ref
- ❌ **Con**: Less OOP, more procedural
- ❌ **Con**: Function injection is less type-safe

**Option C:** Move scenario construction entirely to ScenarioManager
- ✅ **Pro**: Single Responsibility - ScenarioManager owns all scenario logic
- ✅ **Pro**: Already has component backward link
- ❌ **Con**: Requires refactoring ServerHierarchyManager

**RECOMMENDATION**: **Option A** - Backward link, consistent with existing ScenarioManager pattern.
