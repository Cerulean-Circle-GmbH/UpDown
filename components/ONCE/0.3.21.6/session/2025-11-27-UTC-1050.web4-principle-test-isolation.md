## **📊 SUMMARY**

**Test Isolation** is a fundamental Web4 principle that ensures tests NEVER contaminate production data or interfere with each other. It uses Path Authority and scenario-based state management to create completely isolated test environments.

---

## **🎯 THE PRINCIPLE**

### **Test Isolation**

**Definition**: Each test runs in a completely isolated environment with its own:
- Component instances (in `test/data/components/`)
- Scenario storage (in `test/data/components/ONCE/0.3.21.6/scenarios/`)
- Project root (`test/data` becomes `PROJECT_ROOT`)
- CLI tools (symlinked to `test/data/scripts/`)

**Key Insight**: Production files in `/Users/Shared/Workspaces/2cuGitHub/UpDown/` are NEVER touched during tests.

## **🎯 SURGICAL TESTING**

Web4 provides surgical test execution at three levels using **single string tokens**:

### **Level 1: File-Level Testing**

**Command**: `once test file N`

**Purpose**: Run all tests in a specific file

**Example**:
```bash
once test file 7          # Run test file #7 (server-initialization.test.ts)
```

**Token Format**: Single number `"7"`

### **Level 2: Describe-Block Testing**

**Command**: `once test describe NM`

**Purpose**: Run a specific `describe()` block within a file

**Example**:
```bash
once test describe 7a     # Run describe block 'a' in file #7
once test describe 7b     # Run describe block 'b' in file #7
```

**Token Format**: File number + letter `"7a"` (NOT `"7 a"`)

### **Level 3: Individual Test Case (itCase)**

**Command**: `once test itCase NMP`

**Purpose**: Run a single `it()` test case

**Example**:
```bash
once test itCase 7a3      # Run test case #3 in describe 'a' in file #7
once test itCase 7b1      # Run test case #1 in describe 'b' in file #7
```

**Token Format**: File + describe letter + test number `"7a3"` (NOT `"7 a 3"` or `"7" "a" "3"`)

**How It Works**:
1. File number identifies test file (e.g., `7` = file #7)
2. Letter identifies describe block (e.g., `a` = first describe, `b` = second)
3. Number identifies it case within that describe (e.g., `3` = third test)
4. Parser uses `TestFileParser` (Layer 4) for AST parsing
5. Token like `"7a3"` is parsed as: file=7, describe=a, itCase=3

## **📋 TEST DISCOVERY & COMPLETION**

### **Tab Completion for Surgical Testing**

Web4 provides **intelligent tab completion** with single-string tokens:

**File Completion**:
```bash

# Shows: 1:once.test.ts  2:lifecycle-management.test.ts  3:...
```

**Describe Completion**:
```bash
once test describe 7<TAB>
# Shows: 7a:ONCEEnvironmentDetection  7b:ONCEBootstrapping  7c:...
```

**ItCase Completion**:
```bash
once test itCase 7a<TAB>
# Shows: 7a1:should-detect-nodejs  7a2:should-detect-browser  7a3:...
```

**Token Format Examples**:
- `"7"` - File #7
- `"7a"` - File 7, describe block 'a'
- `"7a3"` - File 7, describe 'a', it case #3
- `"17b5"` - File 17, describe 'b', it case #5

### **Implementation** (Radical OOP):

All completion methods are in `DefaultCLI`:

1. **`testFileReferenceParameterCompletion()`**:
   - Scans `test/` directory
   - Returns numbered list of test files
   - Format: `"N:filename.test.ts"`

2. **`testDescribeReferenceParameterCompletion()`**:
   - Parses selected test file
   - Extracts describe blocks
   - Returns numbered list: `"N:describe-name"`

3. **`testItCaseReferenceParameterCompletion()`**:
   - Parses selected describe block
   - Extracts it cases
   - Returns numbered list: `"N:it-case-name"`

**Key**: All use `TestFileParser` (Layer 4) for AST parsing

## **🚨 CRITICAL BUG DISCOVERED**

**Date**: 2025-11-27 UTC 11:00  
**Severity**: HIGH

### **The Bug**

Currently, scenarios are being saved in BOTH locations:
1. ✅ `test/data/scenarios/` (CORRECT)
2. ❌ `test/data/components/ONCE/0.3.21.6/scenarios/` (WRONG!)

**Evidence**:
```bash
$ find test/data/scenarios -name "*.json" | wc -l
4  # ✅ Correct location

$ find test/data/components/ONCE/0.3.21.6/scenarios -name "*.json" | wc -l
4  # ❌ Wrong! These shouldn't exist!
```

### **Root Cause**

The component copy includes a `scenarios/` directory that shouldn't be there. When tests run, some code is writing to the component's scenarios directory instead of PROJECT_ROOT/scenarios/.

### **Correct Behavior**

**ALWAYS**:
- Scenarios go to `${PROJECT_ROOT}/scenarios/`
- In production: `/Users/Shared/Workspaces/2cuGitHub/UpDown/scenarios/`
- In test isolation: `test/data/scenarios/`

**NEVER**:
- Scenarios in `components/ONCE/0.3.21.6/scenarios/`
- This violates the Path Authority principle

### **Fix Required**

1. Verify all scenario save/load code uses `PROJECT_ROOT/scenarios/`
2. Remove `scenarios/` directory from component copies in test/data
3. Add validation to prevent this in future

### **Impact on Iteration 1.16**

This bug MUST be fixed during test suite restoration. All tests must be verified to:
- ✅ Save scenarios to `test/data/scenarios/`
- ❌ Never save to `test/data/components/ONCE/0.3.21.6/scenarios/`

```typescript

const productionScenarios = '/Users/Shared/Workspaces/.../scenarios/';
fs.readFileSync(productionScenarios);
```

```typescript
// ✅ GOOD
const testScenarios = path.join(
  process.env.PROJECT_ROOT || testDataDir,

);

```

```typescript
// ❌ BAD
const componentPath = '/Users/Shared/Workspaces/2cuGitHub/UpDown/components/ONCE/0.3.21.6';
```

```typescript
// ✅ GOOD
const componentPath = once.model.componentRoot;  // Respects PROJECT_ROOT
```

```typescript

  });
});
```

```typescript

describe('My Tests', () => {
```

## **🎓 LESSONS LEARNED**

### **What I Re-Learned**:

1. **Test Isolation is NOT Just a Nice-to-Have**:
   - It's a FUNDAMENTAL Web4 principle
   - Without it, tests corrupt production data
   - It enables fearless refactoring

2. **Path Authority Enables Test Isolation**:
   - Semantic versions work identically in test/data
   - Component discovery works the same
   - No special test code needed

3. **Surgical Testing Saves Time**:
   - `once test itCase 7 2 3` runs ONE test in seconds
   - No need to run 100+ tests to debug one
   - Fast iteration = better code

4. **`once test shell` is the Secret Weapon**:
   - Provides interactive test environment
   - Can run commands exactly as tests do
   - Perfect for debugging failing tests

5. **Scenario-Based State Makes Testing Natural**:
   - Tests create scenarios in `test/data`
   - Production scenarios never touched
   - Easy to verify test state
