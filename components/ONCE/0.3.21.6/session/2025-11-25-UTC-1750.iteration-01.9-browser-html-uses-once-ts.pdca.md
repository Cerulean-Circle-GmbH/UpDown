## **P - PLAN**

### **Current State Analysis**

**Iteration 1.7 Implementation** (`once-client.html`):
```html
<script type="module">
    import { BrowserONCEKernel } from '../js/once-browser-kernel.js';
    const kernel = new BrowserONCEKernel();
    await kernel.init({ peerHost: peerHost }); // ❌ Top-level await
    // ... event handlers with arrow functions ❌
</script>
```

**Issues**:
1. ❌ Direct import of `BrowserONCEKernel` (bypasses `ONCE.ts`)
2. ❌ Top-level `await` (browser compatibility issue)
3. ❌ Arrow functions in event handlers (Radical OOP violation)
4. ❌ Doesn't use Unified Kernel Architecture from Iteration 1.8

**Iteration 1.8 Architecture**:
- `ONCE.ts` (Layer 1) - Environment detector
- `ONCE.start(scenario)` - Dynamically loads kernel based on environment
- `BrowserONCEKernel.ts` (Layer 2) - Currently a stub that needs to delegate to JS

---

### **Design Decision**

**What Needs to Happen**:

1. **HTML Update**: Load `ONCE.ts` (compiled) instead of `once-browser-kernel.js`
2. **TypeScript Wrapper**: Make `BrowserONCEKernel.ts` delegate to the working `once-browser-kernel.js`
3. **Scenario Extraction**: `ONCE.ts` should extract browser config (peer host) from URL params

**Why This Design**:
- ✅ Minimal risk: Working JS kernel stays unchanged
- ✅ Proper layering: `ONCE.ts` → `BrowserONCEKernel.ts` → `once-browser-kernel.js`
- ✅ Type safety: TypeScript wrapper provides type contracts
- ✅ Future-proof: Easy to migrate JS to TS later
