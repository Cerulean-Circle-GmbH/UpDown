## **🏛️ MAJOR ARCHITECTURAL DECISIONS** (2025-12-22)

### File/Folder JsInterface Architecture (P35)

**Problem**: Need runtime polymorphism for `File`/`Folder` but TypeScript interfaces erased at runtime.

**Solution**: Created `File` and `Folder` as JsInterface classes extending `JsInterface`:

```typescript
// layer3/File.ts — EXISTS AT RUNTIME
export abstract class File extends JsInterface {
  abstract get model(): Model;
  abstract get path(): string;
  abstract get isFolder(): boolean;
  abstract parentSet(folder: File | null): void;
  abstract pathSet(newPath: string): void;
  // ...
}

// layer3/Folder.ts — EXTENDS FILE
export abstract class Folder extends File {
  abstract get children(): File[];
  abstract childAdd(child: File): void;
  // ...
}

// layer2/DefaultFile.ts
export class DefaultFile extends UcpComponent<FileModel> implements File {
  static start(): void {
    File.implementationRegister(DefaultFile);
  }
}

// layer2/DefaultFolder.ts
export class DefaultFolder extends UcpComponent<FolderModel> 
  implements Container<File>, File, Folder {
  static start(): void {
    File.implementationRegister(DefaultFolder);
    Folder.implementationRegister(DefaultFolder);
  }
}
```

**Key Changes**:
- `FolderModel.children: Collection<File>` (not FileSystemNode union)
- `FolderModel.parent: LazyReference<Folder>` (typed parent)
- `linkTarget: LazyReference<File/Folder>` (IOR reference, not string path)
- **DELETED**: `FileSystemNode.type.ts`, `ContainerModel.interface.ts`
- **NEW**: `layer3/File.ts`, `layer3/Folder.ts`

**Web4 Principles Applied**:
- **P35**: JsInterface for Runtime Interfaces (NEW)
- **P34**: IOR as Unified Entry Point (`linkTarget` as IOR)
- **DRY**: One `File` interface for all file-like operations

**Usage Pattern**:
```typescript
// Register related file on any component
image.relatedObjectRegister(File, imageFile);

// Lookup later
const file = image.relatedObjectLookupFirst(File);

// Type-safe collections
children: Collection<File>;  // Can hold DefaultFile | DefaultFolder
```
