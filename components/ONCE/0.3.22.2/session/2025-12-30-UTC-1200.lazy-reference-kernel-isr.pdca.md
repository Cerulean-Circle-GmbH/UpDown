## Context

**Goal**: Build a Unit-based Web4 radical OOP P2P file browser as a PWA.

**Problem**: Options A and B both have drawbacks:
- **Option A**: Four-Case Callback Pattern pollutes Layer 2 with type narrowing
- **Option B**: Per-component orchestrators (FileOrchestrator, FolderOrchestrator) duplicate kernel functionality

**This PDCA**: Option C uses the **kernel's universal ISR mechanism** (Self-Replacing IOR objects) so Layer 2 is pure AND no per-component orchestrators are needed.

### The Universal ISR Mechanism (from URP PDCA)

**Key Insight**: The kernel already has a plan for universal lazy reference handling via **Self-Replacing IOR Objects**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    KERNEL ISR (SELF-REPLACING IOR)                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  model.children = ['ior:scenario:file-1', 'ior:scenario:folder-2']  │
│                                                                     │
│  Step 1: First Access                                               │
│  ─────────────────────                                              │
│  const child = folder.model.children[0];                            │
│    │                                                                │
│    ├── UcpModel.get() intercepts                                    │
│    ├── Detects 'ior:...' string                                     │
│    ├── Creates IOR object: new IOR().initRemote(iorString)          │
│    ├── Sets parent reference: ior.initWithParent(model, 'children', 0)│
│    ├── Replaces string with IOR in array: children[0] = ior        │
│    ├── Triggers resolution: ior.resolveAndReplace() (fire-and-forget)│
│    └── Returns IOR object (sync)                                    │
│                                                                     │
│  Step 2: Background Resolution                                      │
│  ─────────────────────────────                                      │
│  ior.resolveAndReplace():                                           │
│    │                                                                │
│    ├── const instance = await this.resolveInstance()                │
│    ├── SELF-REPLACEMENT: children[0] = instance                     │
│    └── UcpModel.set triggers view re-render                         │
│                                                                     │
│  Step 3: Subsequent Access                                          │
│  ─────────────────────────                                          │
│  const child = folder.model.children[0];                            │
│    │                                                                │
│    └── Returns DefaultFile instance directly (already resolved)     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### How Option C Differs from Option B

**Option B**: Per-component orchestrators hold IOR strings and resolve them:

```typescript
// Option B: FolderOrchestrator (separate class)
class FolderOrchestrator {
  private childIORs: string[] = [];  // IORs stored HERE
  private folder: DefaultFolder;
  
  async load(path: string): Promise<void> {
    const scenario = await IOR.load(path);
    this.childIORs = scenario.model.childIORs;
    this.folder = new DefaultFolder().init(scenario);
    
    // Orchestrator resolves each child
    for (const ior of this.childIORs) {
      const child = await IOR.resolve(ior);
      this.folder.childAdd(child);
    }
  }
}
```

**Option C**: Kernel handles resolution universally via UcpModel proxy. Components use standard OOP patterns:

```typescript
// Option C: Web4 Radical OOP — Objects all the way down

/**
 * FolderOrchestrator (Layer 4) — ASYNC lives here
 * 
 * Web4 P7: Async ONLY in Layer 4
 */
export class FolderOrchestrator {
  
  /**
   * Load folder from path — Layer 4 handles async
   * 
   * Returns void — view subscribes to model changes reactively
   */
  folderLoad(path: string, view: FolderOverView): void {
    const ior = new IOR<DefaultFolder>().init("ior:file://" + path);
    
    // Async resolution — fire and forget
    // When resolved, ISR replaces IOR in model → triggers view update
    ior.resolve().then(function onFolderResolved(folder: DefaultFolder): void {
      view.folderSet(folder);  // Sync call to Layer 5
    });
  }
}

/**
 * FolderOverView (Layer 5) — NO ASYNC, purely reactive
 * 
 * Web4 P7: Views are SYNC — they react to model changes
 */
@customElement('folder-over-view')
export class FolderOverView extends UcpView<FolderModel> {
  
  /** The folder being displayed */
  @state()
  private folder: DefaultFolder | null = null;
  
  /** Layer 4 orchestrator */
  private orchestrator = new FolderOrchestrator();
  
  /**
   * Trigger folder load — SYNC, delegates to Layer 4
   */
  folderLoadTrigger(path: string): void {
    this.orchestrator.folderLoad(path, this);  // Sync call to Layer 4
  }
  
  /**
   * Set folder — called by orchestrator when resolved
   * 
   * This is SYNC — just sets state, triggers re-render
   */
  folderSet(folder: DefaultFolder): void {
    this.folder = folder;
    // LitElement @state() triggers requestUpdate() automatically
  }
  
  /**
   * Render children — polymorphic dispatch, no if/else type checks
   * 
   * Web4: Each child knows how to render itself via its view component
   */
  private childrenRender(): TemplateResult[] {
    if (!this.folder) return [];
    
    const parts: TemplateResult[] = [];
    
    // UcpModel proxy handles ISR — we just iterate
    for (const child of this.folder.model.children) {
      // Polymorphic: Each type has its own view tag
      // UcpView.tagFor() returns the correct custom element tag
      const tag = UcpView.tagFor(child);  // 'file-item-view' or 'folder-item-view' or 'ior-loading-view'
      parts.push(html`<${tag} .model=${child.model}></${tag}>`);
    }
    
    return parts;
  }
}

/**
 * IOR is also a renderable type — has its own view
 * 
 * Web4: Even "loading" state is an object (IOR) with a view
 */
@customElement('ior-loading-view')
export class IORLoadingView extends UcpView<IORModel> {
  render(): TemplateResult {
    return html`
      <div class="loading-item">
        <span class="spinner"></span>
        Loading ${this.model.targetType}...
      </div>
    `;
  }
}
```

**Key Web4 Principles Applied**:
- **P7**: `await` ONLY in Layer 4 (`FolderOrchestrator.folderLoad`)
- **P7**: Layer 5 is SYNC — `folderLoadTrigger()` delegates, `folderSet()` reacts
- **P4**: `ior.resolve()` is a method on an IOR instance, not a standalone function
- **P4**: `UcpView.tagFor(child)` returns the polymorphic view tag
- **P34**: IOR is a first-class object with its own view (`ior-loading-view`)
- **Kernel knows nothing about File/Folder**: Just handles IOR strings → IOR objects → instances

### What Layer 2 Code Looks Like (Option C)

**DefaultFolder.ts — PURE, NO TYPE GUARDS**:

```typescript
export class DefaultFolder extends UcpComponent<FolderModel> implements FolderJs {
  
  /**
   * Get children — returns array that may contain:
   * - FileJs instances (resolved)
   * - IOR<FileJs> objects (resolving)
   * 
   * Kernel's UcpModel proxy handles the transition automatically.
   */
  get children(): (FileJs | IOR<FileJs>)[] {
    return this.model.children;  // Kernel handles ISR
  }
  
  /**
   * Add a child — SYNC, PURE
   * 
   * Accepts:
   * - FileJs instance (already resolved)
   * - string (IOR string — kernel will handle)
   */
  childAdd(child: FileJs | string): void {
    this.model.children.push(child);
    this.model.modifiedAt = Date.now();
    
    // If resolved instance, set parent immediately
    if (typeof child !== 'string' && !(child instanceof IOR)) {
      child.parentSet(this);
    }
    // If string or IOR, kernel's ISR will handle parentSet when resolved
  }
  
  /**
   * Get resolved children only — for operations that need instances
   * 
   * Uses filter with instanceof check (NOT type narrowing — runtime check)
   */
  get resolvedChildren(): FileJs[] {
    const resolved: FileJs[] = [];
    for (const child of this.model.children) {
      // Skip IOR objects and strings (not yet resolved)
      if (typeof child === 'string') continue;
      if (child instanceof IOR) continue;
      resolved.push(child);
    }
    return resolved;
  }
}
```

### View Code (Layer 5) — Polymorphic Rendering

```typescript
/**
 * FolderOverView — Web4 Radical OOP View
 * 
 * NO if/else type checks — polymorphic dispatch via UcpView.tagFor()
 */
@customElement('folder-over-view')
export class FolderOverView extends UcpView<FolderModel> {
  
  /** The folder instance (not just model) */
  private folder: DefaultFolder | null = null;
  
  render(): TemplateResult {
    return html`
      <nav class="breadcrumb">
        ${this.breadcrumbRender()}
      </nav>
      
      <div class="children-list">
        ${this.childrenRender()}
      </div>
    `;
  }
  
  /**
   * Render children — POLYMORPHIC, no type checks
   * 
   * Each child type (FileJs, FolderJs, IOR) has its own registered view.
   * UcpView.tagFor() returns the correct custom element tag.
   * 
   * Web4 P4: No arrow functions, no if/else type discrimination
   */
  private childrenRender(): TemplateResult[] {
    if (!this.folder) return [];
    
    const parts: TemplateResult[] = [];
    
    for (const child of this.folder.model.children) {
      // Polymorphic dispatch: child.viewTag or UcpView.tagFor(child)
      // Each type registers its view: FileJs → 'file-item-view', IOR → 'ior-loading-view'
      parts.push(this.childRender(child));
    }
    
    return parts;
  }
  
  /**
   * Render single child — polymorphic via registered views
   */
  private childRender(child: FileJs | IOR<FileJs>): TemplateResult {
    // UcpView.tagFor() uses TypeRegistry to find the view for this type
    // FileJs → 'file-item-view' or 'folder-item-view' (based on isFolder)
    // IOR<T> → 'ior-loading-view'
    const viewTag = UcpView.tagFor(child);
    
    // Lit dynamic tag rendering
    return html`<${unsafeStatic(viewTag)} .instance=${child}></${unsafeStatic(viewTag)}>`;
  }
}
```

### IOR is Thenable — Await Directly

**Key Feature**: The IOR class implements `then()` making it thenable:

```typescript
class IOR<T> {
  /** Store the resolution promise */
  private resolutionPromise: Promise<T> | null = null;
  
  /**
   * Make IOR thenable — can be awaited directly
   * 
   * Usage: const instance = await child;  // If child is IOR
   */
  then<TResult1 = T, TResult2 = never>(
    onfulfilled?: ((value: T) => TResult1 | PromiseLike<TResult1>) | null,
    onrejected?: ((reason: unknown) => TResult2 | PromiseLike<TResult2>) | null
  ): Promise<TResult1 | TResult2> {
    if (!this.resolutionPromise) {
      this.resolutionPromise = this.resolveAndReplace();
    }
    return this.resolutionPromise.then(onfulfilled, onrejected);
  }
}

// Usage in Layer 4 orchestrator (when explicit await needed):
class FolderOrchestrator {
  
  /**
   * Await all children to be resolved
   * 
   * Uses IOR.then() — can await any child regardless of type
   * Resolved instances return immediately (Promise.resolve)
   */
  async childrenAwait(folder: DefaultFolder): Promise<FileJs[]> {
    const resolved: FileJs[] = [];
    
    for (const child of folder.model.children) {
      // Every item is awaitable: IOR has then(), instances wrap in Promise.resolve()
      const instance = await Promise.resolve(child);
      resolved.push(instance);
    }
    
    return resolved;
  }
}

// In Layer 5 views: NO await needed — polymorphic rendering handles it
// IOR renders as <ior-loading-view>, FileJs renders as <file-item-view>
// When IOR resolves → self-replaces → parent re-renders automatically
```

### Why Option C is Best

| Aspect | Option A | Option B | **Option C** |
|--------|----------|----------|--------------|
| **Layer 2 Complexity** | High (type guards) | Low | **Low** |
| **Code Duplication** | Medium | High (orchestrators) | **None** |
| **Kernel Changes** | None | None | **Yes (planned in URP)** |
| **Web4 P34 Compliance** | Partial | Partial | **Full** |
| **DRY** | Violated (childReferences) | Partial | **Full** |
| **Per-Component Work** | Every method | Orchestrator per component | **None** |

### The Key Insight

> "The one layer of references look ahead loading is also not specific for folder or file only. It's the **default behaviour for model**."
> 
> — TRON, 2025-12-22

**Option C embraces this**: The kernel's ISR mechanism handles ALL reference resolution universally. Components just access `model.children` and the kernel does the rest.

---

### Recommendation

**Option C is the correct Web4 architecture because**:

1. **Universal Mechanism**: One kernel solution handles ALL components
2. **Layer 2 Stays Pure**: No type narrowing, no async callbacks
3. **P34 Full Compliance**: IOR is everywhere, self-replacing
4. **DRY**: No orchestrators duplicating kernel functionality
5. **Already Planned**: URP PDCA has the implementation tasks

**Decision**: Implement URP (kernel ISR) first, then Option C is automatic.

## Implementation (2025-12-31)

### Implemented Components

| Component | File | Description |
|-----------|------|-------------|
| **UcpModel ISR** | `layer3/UcpModel.ts` | Proxy detects IOR strings, creates self-replacing IOR objects |
| **IOR.initWithParent()** | `layer4/IOR.ts` | Stores parent reference for self-replacement |
| **IOR.resolveAndReplace()** | `layer4/IOR.ts` | Resolves and replaces self in parent model |
| **Array Proxy** | `layer3/UcpModel.ts` | Wraps arrays to handle IOR string elements |
| **IORLoadingView** | `layer5/views/IORLoadingView.ts` | Shows spinner for resolving IORs |
| **UcpView.tagFor()** | `layer5/views/UcpView.ts` | Polymorphic view dispatch based on type |
| **ViewRegistry** | `layer5/views/ViewRegistry.ts` | Registers component → view mappings |
| **FolderOverView** | `layer5/views/FolderOverView.ts` | Updated with polymorphic child rendering |

### Key Changes

1. **UcpModel.modelProxyCreate()** now intercepts IOR strings on property access and creates self-replacing IOR objects
2. **UcpModel.arrayProxyCreate()** handles Collection<T> with IOR string elements
3. **IOR.initWithParent()** stores reference to parent model for self-replacement
4. **IOR.resolveAndReplace()** fire-and-forget resolution that updates parent model when done
5. **UcpView.tagFor()** returns correct view tag based on instance type or IOR resolving state
6. **FolderOverView.renderLazyChildren()** polymorphically renders children regardless of resolution state
