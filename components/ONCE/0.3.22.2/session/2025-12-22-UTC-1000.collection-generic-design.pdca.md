## Problem

Current implementation uses type aliases:
```typescript
export type Collection<T> = T[];
export type LazyCollection<T> = (T | IORType<T> | string)[];
```

User expected proper OOP generic interface:
```typescript
interface Collection<T extends LazyReference> extends Iterable<T> {
  readonly length: number;
  [index: number]: T;
}
```

## Design Options

### Option A: Current (Type Alias)
```typescript
export type Collection<T> = T[];
export type LazyCollection<T> = (T | IOR<T> | string)[];
```

**Pros**: Simple, array-compatible  
**Cons**: Not a proper interface, two separate types, no methods

### Option B: Generic Interface with LazyReference Constraint
```typescript
interface Collection<T> extends Iterable<LazyReference<T>> {
  readonly length: number;
  [index: number]: LazyReference<T>;
  
  // Collection methods
  at(index: number): LazyReference<T>;
  resolveAll(): Promise<T[]>;
  push(...items: LazyReference<T>[]): number;
}
```

**Pros**: Proper OOP, single type, methods for resolution  
**Cons**: Arrays don't implement this interface directly

### Option C: Collection IS LazyReference Array
```typescript
// LazyReference<T> already includes T | IOR<T> | string | null
interface Collection<T> extends Array<LazyReference<T>> {
  resolveAll(): Promise<T[]>;
}
```

**Pros**: Extends Array, inherits all array methods  
**Cons**: Need to create Collection instances, not just use arrays

### Option D: LazyReference as Generic Constraint
```typescript
interface Collection<T extends LazyReference<unknown>> extends Iterable<T> {
  readonly length: number;
  [index: number]: T;
}

// Usage:
children: Collection<LazyReference<FileSystemNode>>
```

**Pros**: Type constraint is explicit  
**Cons**: Verbose, LazyReference<unknown> is awkward

## Recommendation

**Option C** (Collection extends Array) provides:
- Proper interface semantics
- Full array compatibility
- Single type (no LazyCollection needed)
- Optional `resolveAll()` method for bulk resolution

```typescript
/**
 * Collection<T> - Lazy collection extending Array
 * 
 * Elements are LazyReference<T>: can be T, IOR<T>, or string.
 * Extends Array for full compatibility.
 */
export interface Collection<T> extends Array<LazyReference<T>> {
  /**
   * Resolve all IOR strings/objects to instances
   * @returns Promise with array of resolved instances
   */
  resolveAll(): Promise<(T | null)[]>;
}
```

### Usage Would Be:
```typescript
// FolderModel
children: Collection<FileSystemNode>;  // Simple!

// Assignment still works (array is assignable)
folder.model.children = ["ior:scenario:uuid1", file2];

// Resolution
const resolved = await folder.model.children.resolveAll();
```

## Implementation (Completed)

**Decision**: Option C implemented with refinement.

### Final Design

```typescript
// Collection<T> = Lazy (for IOR-backed references)
interface Collection<T> extends Array<LazyReference<T>> {}

// Array<T> = Simple (for local-only collections)
actionsDiscover(): ActionMetadata[]  // Not Collection<ActionMetadata>
```

### Key Insight

Not ALL collections need lazy loading:
- `children: Collection<FileSystemNode>` → Lazy (IOR-backed)
- `actionsDiscover(): ActionMetadata[]` → Simple (local)
- `peers: OncePeerModel[]` → Simple (not IOR-backed)
