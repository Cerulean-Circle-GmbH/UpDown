## 1. OBJECTIVE

Transform `View` from a compile-time-only interface to a runtime-existing JsInterface class, enabling:
- Runtime type introspection (`View.implementations`)
- View registration via static initialization blocks
- Consistent pattern with `Component extends JsInterface`

---

## 2. CONTEXT

### 2.1 The Misunderstanding (Corrected)

**Previous WRONG assumption:**
> "Views must extend LitElement, can't also extend JsInterface"
> This was treating `extends` and `implements` as the same thing.

**Correct understanding:**
- `extends` = single inheritance chain (blocked for multiple parents)
- `implements` = contract compliance (NOT blocked - can implement multiple!)

### 2.2 Framework Independence Requirement

**UcpView must be framework-independent!** Support for:
- Lit (LitElement)
- React
- Vue
- Angular
- Vanilla JS

**Correct hierarchy:**
```typescript
// View.ts - JsInterface base with component reference
abstract class View extends JsInterface {
  abstract get component(): Component;  // Back-reference to owning component
  abstract get model(): TModel;
  // ...
}

// UcpView.ts - Framework-INDEPENDENT abstract base
abstract class UcpView implements View {
  // Shared Web4 view logic (no framework dependencies)
}

// LitUcpView.ts - Lit-specific adapter
abstract class LitUcpView extends LitElement implements View {
  // Lit-specific rendering
}
```

## 3. PLAN

### 3.1 Framework Adapter Options Analysis

**Option A: `LitUcpView extends LitElement implements View`**
```typescript
abstract class View extends JsInterface { ... }
abstract class UcpView implements View { ... }  // Framework-independent
class LitUcpView extends LitElement implements View { ... }  // Parallel
```

**Option B: `LitUcpView extends LitElement implements UcpView`**
```typescript
abstract class View extends JsInterface { ... }
abstract class UcpView implements View { ... }  // Extended contract
class LitUcpView extends LitElement implements UcpView { ... }
```

| Aspect | Option A (`implements View`) | Option B (`implements UcpView`) |
|--------|------------------------------|----------------------------------|
| Contract | Base View contract | Full UcpView contract (richer) |
| Shared code | None via implements | None via implements |
| Web4 extensions | Must be in View | Can be in UcpView |
| Flexibility | More flexible | Stricter contract |
| DRY | Each adapter re-implements | Each adapter re-implements |

**Decision: Option A** вЂ” Put all Web4 contracts in `View`, keep `UcpView` as framework-independent abstract base.

**Key insight:** `implements` only checks shape вЂ” does NOT inherit implementations. Both options require re-implementing in each framework adapter.

### 3.2 Component Back-Reference

**Every View must reference its owning Component:**

```typescript
abstract class View<TModel extends Model> extends JsInterface {
  // Back-reference to owning component
  abstract get component(): Component;
  abstract set component(c: Component);
  
  // ... other members
}
```

This enables:
1. View can call component methods
2. View can access component's model via component
3. Component lifecycle affects view lifecycle

### 3.3 Key Insight: `implements` Works with Abstract Classes

TypeScript allows `implements` with abstract classes. The `implements` keyword:
1. Checks that the class has all required members (compile-time)
2. Does NOT copy static members (runtime - we handle separately)
3. Works as a "shape check" for the class

#### 3.4.2 NEW: `layer3/UcpView.ts` (Framework-independent abstract base)

```typescript
/**
 * UcpView.ts - Framework-independent View base class
 * 
 * Abstract base for views that don't need DOM rendering.
 * Use cases: server-side rendering, testing, non-DOM environments.
 * 
 * For Lit/browser views, use LitUcpView instead.
 * 
 * @ior ior:esm:/ONCE/{version}/UcpView
```

## 9. NOTES

### Why `implements` Works

TypeScript `implements` keyword:
1. Checks shape at compile time (all abstract members implemented)
2. Does NOT require extending the class
3. Allows multiple implements: `class X implements A, B, C`

This is why:
```typescript
class LitUcpView extends LitElement implements View
```
Works perfectly - LitUcpView gets LitElement's DOM features AND satisfies View's contract.

### Static Registration Caveat

`implements` does NOT copy static members. So:
- LitUcpView does NOT automatically have `View.type` or `View.implementations`
- We manually call `View.implementationRegister(LitUcpView)` in static block
- This is intentional - keeps inheritance chain clean

### Framework Independence

**Why two view base classes?**

1. **UcpView (layer3)** вЂ” Framework-independent
   - For server-side rendering
   - For testing without DOM
   - For non-browser environments (Node.js)
   - No DOM dependencies

2. **LitUcpView (layer5)** вЂ” Lit-specific
   - For browser DOM rendering
   - Uses LitElement for web components
   - Shadow DOM, CSS, etc.

### Component Back-Reference

**Why views need component reference?**

1. View can call component methods
2. View can trigger component actions (not just read model)
3. Component lifecycle affects view
4. Bidirectional binding possible
