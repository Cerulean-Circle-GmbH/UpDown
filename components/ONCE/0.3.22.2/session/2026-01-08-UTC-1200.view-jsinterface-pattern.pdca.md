## 1. OBJECTIVE

Transform `View` from a compile-time-only interface to a runtime-existing JsInterface class, enabling:
- Runtime type introspection (`View.implementations`)
- View registration via static initialization blocks
- Consistent pattern with `Component extends JsInterface`

---

## 2. CONTEXT

### 2.1 The Misunderstanding (Corrected)

**Previous WRONG assumption:**
> "Views must extend LitElement, can't also extend JsInterface"
> This was treating `extends` and `implements` as the same thing.

**Correct understanding:**
- `extends` = single inheritance chain (blocked for multiple parents)
- `implements` = contract compliance (NOT blocked - can implement multiple!)

```typescript
// THIS WORKS in TypeScript:
class UcpView extends LitElement implements View {
  // LitElement provides DOM rendering
  // View (abstract class) provides type contract
}
```

## 3. PLAN

### 3.1 Pattern Comparison: Component vs View

| Aspect | Component (Phase 2) | View (Phase 3) |
|--------|---------------------|----------------|
| Base | `extends JsInterface` | `extends JsInterface` |
| Implementation | `extends Component` | `implements View` (note: implements!) |
| Why | Single inheritance OK | LitElement requires extends |
| Registration | `Component.implementationRegister()` | `View.implementationRegister()` |
| Auto-register | `static start()` | `static { }` block |

### 3.2 Key Insight: `implements` Works with Abstract Classes

TypeScript allows `implements` with abstract classes. The `implements` keyword:
1. Checks that the class has all required members (compile-time)
2. Does NOT copy static members (runtime - we handle separately)
3. Works as a "shape check" for the class

```typescript
// View.ts - JsInterface base
export abstract class View<TModel = any> extends JsInterface {
  // Abstract members that implementers must provide
  abstract get model(): TModel;
  abstract set model(m: TModel);
  abstract refresh(): void;
  abstract add(childView: View<any>): void;
  abstract childRemove(childView: View<any>): void;
  abstract render(): any;
}

// UcpView.ts - Lit adapter that implements View
export abstract class UcpView<TModel = any> extends LitElement implements View<TModel> {
  // ES2022 static block: runs when class loads
  static {
    View.implementationRegister(UcpView);
  }
  
  // Must implement all View abstract members
  get model(): TModel { ... }
  set model(m: TModel) { ... }
  refresh(): void { ... }
  add(childView: View<any>): void { ... }
  childRemove(childView: View<any>): void { ... }
  abstract render(): any;  // Still abstract - subclasses provide
}
```

### 3.3 File Changes

#### 3.3.1 NEW: `layer3/View.ts` (JsInterface base)

```typescript
/**
 * View.ts - Runtime JsInterface for Views
 * 
 * ✅ Web4 Principle 35: JsInterface for Runtime Interfaces
 * 
 * This is the RUNTIME representation of View.interface.ts.
 * TypeScript interfaces are erased at runtime. View extends JsInterface
 * to exist at runtime for:
 * - TypeRegistry: runtime type introspection
 * - Implementation lookup: `View.implementations` → all view classes
 * - RelatedObjects: interface-based lookup
 * 
 * Pattern:
 * - View.interface.ts: Compile-time contract (kept for documentation)
 * - View.ts (this file): Abstract class extends JsInterface
 * - UcpView.ts: Lit adapter implements View
 * 
 * @ior ior:esm:/ONCE/{version}/View
 * @pdca session/2026-01-08-UTC-1200.view-jsinterface-pattern.pdca.md
 */

import { JsInterface } from './JsInterface.js';
import type { Model } from './Model.interface.js';

/**
 * View - Runtime interface for all Web4 views
 * 
 * Extends JsInterface for runtime type introspection.
 * Framework adapters (Lit, Vanilla) implement this.
 * 
 * Layer 5 - All methods are SYNCHRONOUS (Web4 P7)
 */
export abstract class View<TModel extends Model = Model> extends JsInterface {
  
  // ═══════════════════════════════════════════════════════════════
  // CLASS LEVEL (static, from JsInterface)
  // ═══════════════════════════════════════════════════════════════
  
  // Inherited from JsInterface:
  // static get type(): TypeDescriptor
  // static get implementations(): InterfaceConstructor[]
  // static implementationRegister(impl): void
  
  // ═══════════════════════════════════════════════════════════════
  // ABSTRACT MEMBERS (implementers must provide)
  // ═══════════════════════════════════════════════════════════════
  
  /**
   * Model getter - TypeScript accessor (P16)
   */
  abstract get model(): TModel;
  
  /**
   * Model setter - triggers view refresh
   */
  abstract set model(m: TModel);
  
  /**
   * Refresh view when model changes - SYNCHRONOUS! (P7)
   * Called by UcpModel.viewsNotify()
   */
  abstract refresh(): void;
  
  /**
   * Add child view (P22: Collection<T>)
   */
  abstract add(childView: View<any>): void;
  
  /**
   * Remove child view
   * Named childRemove to avoid conflict with DOM Element.remove()
   */
  abstract childRemove(childView: View<any>): void;
  
  /**
   * Render the view
   * Lit: returns TemplateResult
   * Vanilla: manipulates DOM directly
   */
  abstract render(): any;
}
```

## 9. NOTES

### Why `implements` Works

TypeScript `implements` keyword:
1. Checks shape at compile time (all abstract members implemented)
2. Does NOT require extending the class
3. Allows multiple implements: `class X implements A, B, C`

This is why:
```typescript
class UcpView extends LitElement implements View
```
Works perfectly - UcpView gets LitElement's DOM features AND satisfies View's contract.

### Static Registration Caveat

`implements` does NOT copy static members. So:
- UcpView does NOT automatically have `View.type` or `View.implementations`
- We manually call `View.implementationRegister(UcpView)` in static block
- This is intentional - keeps inheritance chain clean
