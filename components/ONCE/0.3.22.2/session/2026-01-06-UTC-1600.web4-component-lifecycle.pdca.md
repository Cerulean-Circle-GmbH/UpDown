## 2. CONTEXT

### 2.1 Current State

The `isInitialized` getter in `UcpComponent` tracks only whether `init()` was called. This is a single boolean, but the actual lifecycle is much richer:

```
╔═══════════════════════════════════════════════════════════╗
║  CLASS LEVEL (static, once per class)                     ║
╠═══════════════════════════════════════════════════════════╣
║  IOR Reference                                            ║
║      ↓                                                    ║
║  COMPILE CHECK (if source changed)                        ║
║      ↓                                                    ║
║  LOAD (import module, .type.json)                         ║
║      ↓                                                    ║
║  START ─┬─ TypeRegistry registration                      ║
║         ├─ JsInterface registration (implements)          ║
║         ├─ VIEW classes load                              ║
║         ├─ VIEW registration with class                   ║
║         └─ CSS preload and apply                          ║
╚═══════════════════════════════════════════════════════════╝
                        ↓
╔═══════════════════════════════════════════════════════════╗
║  INSTANCE LEVEL (per instance)                            ║
╠═══════════════════════════════════════════════════════════╣
║  NEW (instance as blackbox)                               ║
║      ↓                                                    ║
║  INIT (scenario injection, UcpModel proxy wired)          ║
║      ↓                                                    ║
║  READY (display views, receive updates)                   ║
║      ↓                                                    ║
║  RUNNING (scenario updates to peers)                      ║
║      ↓                                                    ║
║  STOP (cease operations)                                  ║
║      ↓                                                    ║
║  SHUTDOWN (cleanup instance resources)                    ║
╚═══════════════════════════════════════════════════════════╝
                        ↓
╔═══════════════════════════════════════════════════════════╗
║  CLASS LEVEL (static, once per class - teardown)          ║
╠═══════════════════════════════════════════════════════════╣
║  STOP (static) - unregister views                         ║
║      ↓                                                    ║
║  UNLOAD - unregister from TypeRegistry                    ║
╚═══════════════════════════════════════════════════════════╝
```

### 2.2 JsInterface Pattern (File/DefaultFile)

TypeScript interfaces are erased at runtime. The JsInterface pattern solves this:

```
File.interface.ts     ← Compile-time contract (ERASED)
        ↓
FileJs.ts             ← Abstract class extends JsInterface, implements File
        ↓
DefaultFile.ts        ← Concrete class implements FileJs
```

Key elements:
1. **JsInterface** base class - provides `type` (TypeDescriptor) and `implementations` registry
2. **FileJs** - runtime-existing interface, declares abstract methods
3. **DefaultFile** - implements FileJs, declares `static implements() { return [FileJs]; }`
4. **static start()** - auto-registers with JsInterfaces via TypeRegistry

### 2.3 TypeRegistry Role

```typescript
TypeRegistry.register(classConstructor, astData)
  → Creates TypeDescriptor from .type.json
  → Stores in typeMap, nameMap, iorMap
  → Auto-discovers implements from AST
  → Registers with interface's implementations list
```

### 3.1 Lifecycle State Enums

**Class-level** (static, tracked on constructor):

```typescript
enum ClassLifecycleState {
  UNLOADED = 'unloaded',           // Not loaded
  LOADING = 'loading',              // Module import in progress
  LOADED = 'loaded',                // Module imported, .type.json loaded
  STARTING = 'starting',            // static start() in progress
  STARTED = 'started',              // TypeRegistry + Views + CSS ready
  STOPPING = 'stopping',            // static stop() in progress
  STOPPED = 'stopped',              // Unregistered, ready to unload
  ERROR = 'error'
}
```

**Instance-level** (tracked on instance):

```typescript
enum InstanceLifecycleState {
  INSTANTIATED = 'instantiated',    // new() called, blackbox
  INITIALIZING = 'initializing',    // init() in progress
  INITIALIZED = 'initialized',      // init() complete, model wired
  READY = 'ready',                  // Fully operational, views bound
  UPDATING = 'updating',            // Model update in progress
  SYNCING = 'syncing',              // Syncing scenario with peers
  STOPPING = 'stopping',            // Instance shutdown in progress
  STOPPED = 'stopped',              // Instance resources released
  ERROR = 'error'
}
```

### 3.2 Component (JsInterface for Component)

Apply the JsInterface pattern (no "Js" suffix — no browser conflict):

```
Component.interface.ts    ← Compile-time contract (ERASED)
        ↓
Component.ts              ← Abstract class extends JsInterface
        ↓
UcpComponent.ts           ← Base implementation extends Component
        ↓
DefaultWeb4TSComponent.ts ← Concrete implementation
```

### 3.3 View (JsInterface for View)

```
View.interface.ts         ← Compile-time contract (ERASED)
        ↓
View.ts                   ← Abstract class extends JsInterface
        ↓
AbstractView.ts           ← Base implementation extends View
        ↓
Concrete views            ← OncePeerDefaultView, etc.
```

Note: FileJs uses "Js" suffix only because browser has native `File` class.
`View` and `Component` have no such conflict.

### 3.4 Class-Level Lifecycle (static methods)

```typescript
// UcpComponent.ts

/** Class-level lifecycle state (on constructor function) */
private static classState: ClassLifecycleState = ClassLifecycleState.UNLOADED;

/**
 * LOAD phase - called by Loader when module is imported
 * Triggered automatically by ES module system
 */
static async load(): Promise<void> {
  this.classState = ClassLifecycleState.LOADING;
  // 1. Load .type.json for this component (AST metadata)
  // 2. Import dependencies
  // 3. Register with TypeRegistry (minimal descriptor)
  this.classState = ClassLifecycleState.LOADED;
}

/**
 * START phase - called after load, before any instances
 * 
 * This is where VIEWS and CSS are loaded — once per class!
 */
static start(): void {
  this.classState = ClassLifecycleState.STARTING;
  
  // 1. Build full TypeDescriptor from AST
  TypeRegistry.instance.register(this, this.typeData);
  
  // 2. Register with JsInterfaces (from static implements())
  const interfaces = this.implements?.() ?? [];
  for (const iface of interfaces) {
    iface.implementationRegister(this);
  }
  
  // 3. Load view classes (dynamic import)
  this.viewsLoad();
  
  // 4. Register views with this component class
  this.viewsRegister();
  
  // 5. Preload and apply CSS
  this.cssPreload();
  
  this.classState = ClassLifecycleState.STARTED;
}

/**
 * STOP phase - called before unload
 */
static stop(): void {
  this.classState = ClassLifecycleState.STOPPING;
  // 1. Unregister views
  // 2. Unregister from TypeRegistry
  // 3. Cleanup static resources (CSS removed if last user)
  this.classState = ClassLifecycleState.STOPPED;
}

/**
 * Load view classes for this component (called from start())
 */

}

/**
 * Register loaded views (called from start())
 */

}

/**
```

### 3.5 Instance-Level Lifecycle

```typescript
// UcpComponent.ts

private instanceState: InstanceLifecycleState = InstanceLifecycleState.INSTANTIATED;

/**
 * Check if init() has been called
 */
get isInitialized(): boolean {
  return this.instanceState !== InstanceLifecycleState.INSTANTIATED &&
         this.instanceState !== InstanceLifecycleState.INITIALIZING;
}

/**
 * Check if instance is fully operational
 */
get isReady(): boolean {
  return this.instanceState === InstanceLifecycleState.READY;
}

  this.instanceState = InstanceLifecycleState.INITIALIZING;
  
  // Get default scenario with model from child class

    scenario = this.scenarioDefault();

  this.viewsBind();
  
  this.instanceState = InstanceLifecycleState.READY;
```

### 3.7 Node.js Type System Loading

Understand how Node.js loads classes at runtime:

```typescript
// Loader.ts (conceptual)

async loadComponent(ior: string): Promise<typeof UcpComponent> {
  // 1. Parse IOR to get module path
  const modulePath = this.iorToPath(ior);
  
  // 2. Check if compile needed (tsconfig, source changes)
  if (await this.compileCheck(modulePath)) {
    await this.compile(modulePath);
  }
  
  // 3. Dynamic import
  const module = await import(modulePath);
  
  // 4. Load .type.json (AST metadata)
  const typeData = await this.loadTypeJson(modulePath);
  
  // 5. Register with TypeRegistry
  TypeRegistry.instance.register(module.default, typeData);
  
  // 6. Call static start()
  module.default.start();
  
  return module.default;
}
```

### 5.3 State Transition Fix
**Problem:** `CREATED → STARTING` was invalid after adding component lifecycle states.

**Root Cause:** Server startup uses `CREATED → STARTING` directly (legacy flow), but new component lifecycle requires `CREATED → INITIALIZING → INITIALIZED → STARTING`.

**Fix:** Added `STARTING` as valid transition from `CREATED` for backward compatibility:
```typescript
[LifecycleState.CREATED]: [LifecycleState.INITIALIZING, LifecycleState.STARTING],
```

### 5.4 Phase 3 View Pattern (CORRECTED — NOT Blocked!)

**⚠️ CORRECTION: Previous analysis was WRONG**

The `extends` vs `implements` distinction was misunderstood:
- `extends` = single inheritance chain (blocked for multiple parents)
- `implements` = contract compliance (NOT blocked — can implement multiple!)

```typescript
// THIS WORKS in TypeScript:
class UcpView extends LitElement implements View {
  // LitElement provides DOM rendering
  // View (abstract JsInterface class) provides type contract
}
```

**Fractal PDCA:** [2026-01-08-UTC-1200.view-jsinterface-pattern.pdca.md](./2026-01-08-UTC-1200.view-jsinterface-pattern.pdca.md)

**Phase 2 (Component) — extends pattern:**

```

║  PHASE 2: Component — extends JsInterface                                 ║
╠═══════════════════════════════════════════════════════════════════════════╣
║                                                                           ║
║  JsInterface                                                              ║
║       ↑ extends                                                           ║
║  Component                    ← static type, implementations              ║
║       ↑ extends                                                           ║
║  UcpComponent                 ← inherits static members                   ║
```

**Phase 3 (View) — implements pattern:**

```
╔═══════════════════════════════════════════════════════════════════════════╗
║  PHASE 3: View — implements JsInterface class                             ║

║                                                                           ║
║  JsInterface                                                              ║
║       ↑ extends                                                           ║
║  View (NEW)                   ← abstract class, has static registry       ║
║       ↓ implements                                                        ║
║  UcpView extends LitElement implements View                               ║
║       └── static { View.implementationRegister(UcpView); }                ║
║                                                                           ║
╚═══════════════════════════════════════════════════════════════════════════╝
```

**Solution: View as Abstract JsInterface Class**

```typescript
// layer3/View.ts - NEW
export abstract class View<TModel = any> extends JsInterface {
  // Inherited: static type, implementations, implementationRegister()
  
  abstract get model(): TModel;
  abstract set model(m: TModel);
  abstract refresh(): void;
  abstract add(childView: View<any>): void;
  abstract childRemove(childView: View<any>): void;
  abstract render(): any;
}

// layer5/views/UcpView.ts - UPDATED
export abstract class UcpView<TModel = any> extends LitElement implements View<TModel> {
  
  // ES2022 static block: auto-registers when module loads
  static {
    View.implementationRegister(UcpView);
  }
  
  // Implements all View abstract members...
}
```

**Why this works:**
1. `View extends JsInterface` — has runtime type registry
2. `UcpView extends LitElement` — has DOM rendering
3. `UcpView implements View` — TypeScript checks contract compliance
4. `static { }` block — manually registers with View.implementations

## 7. RESEARCH FINDINGS

### 7.1 JsInterface Pattern Summary

```
┌─────────────────────────────────────────────────────────────┐
│  COMPILE TIME                    RUNTIME                    │
├─────────────────────────────────────────────────────────────┤
│  File.interface.ts               (erased)                   │
│       ↓                                                     │
│  FileJs extends JsInterface ───→ FileJs.type (TypeDescriptor)
│       ↓                          FileJs.implementations     │
│  DefaultFile implements FileJs → DefaultFile.type           │
│                                  registered in FileJs.impls │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 TypeDescriptor Contents

```typescript
TypeDescriptor {
  name: string;              // Class name
  implements: Set<Class>;    // Interfaces this class implements
  implementingClasses: Set;  // Classes implementing this interface
  properties: Map;           // Property descriptors from AST
  methods: Map;              // Method descriptors from AST
}
```

### 7.3 Node.js Class Loading Flow

```
CLASS LEVEL (once per class):
1. import('./Component.js')     → ES module loaded
2. Component.load()             → Load .type.json (AST metadata)
3. TypeRegistry.register()      → TypeDescriptor created
4. Component.start()            → JsInterface + Views + CSS

INSTANCE LEVEL (per instance):
5. new Component()              → Instance created (blackbox)
6. component.init(scenario)     → Model wired, views bound
7. component.ready              → Fully operational
```

Key insight: Views and CSS are **class-level** concerns, loaded once in `static start()`.
By the time `new Component()` is called, views and CSS are already available.
