## P — PLAN

### Problem Statement

During Path Calculation Consolidation, `(this.model as any)` casts were introduced because the generic `TModel extends Model` doesn't include path properties. This violates Web4 type safety.

**Total `as any)` casts found**: 136 across 10 files in layer2

### Issues in `web4-component-anatomy-details.md`

The documentation shows these incorrect accessor patterns:

```typescript
// WRONG: Uses string and any cast
get componentRoot(): string {
  return (this.model as any)?.componentRoot ?? '';
}

get targetComponentRoot(): string {
  return (this.model as any)?.targetComponentRoot ?? this.componentRoot;
}
```

### Questions for TRON

**Q1: Path types — Should they be `Folder` (runtime `FolderJs`) instead of `string`?**

```typescript
// Current (string):
get componentRoot(): string { ... }
get targetComponentRoot(): string { ... }
get projectRoot(): string { ... }

// Proposed (Folder):
get componentRoot(): FolderJs { ... }
get targetComponentRoot(): FolderJs { ... }
get projectRoot(): FolderJs { ... }
```

If `Folder`, how to handle lazy initialization? Folder needs async creation.

**Q2: What about CLI model paths?**

`CLIModel.projectRoot` is currently `string`. Should it also be `FolderJs`?

## Proposed Solutions

### Solution A: Extend Model Interface (Quick Fix)

Create `PathModel` interface extending `Model`:

```typescript
interface PathModel extends Model {
  componentRoot?: string;
  projectRoot?: string;  // Deprecated - use CLI
  // etc.
}
```

Then: `UcpComponent<TModel extends PathModel>`

**Pros**: Quick, minimal changes
**Cons**: Still uses strings, not Folder objects

### Solution B: Folder-Based Paths (Web4 Radical OOP)

Path properties return `FolderJs` instances:

```typescript
interface PathModel extends Model {
  componentRoot?: FolderJs;
  targetComponentRoot?: FolderJs;
}

// Accessor:
get componentRoot(): FolderJs | undefined {
  return this.model?.componentRoot;
}
```

**Pros**: Proper OOP, type-safe
**Cons**: Requires changes to all path consumers

### Solution C: Hybrid (String + Folder Accessor)

Keep model properties as strings, but provide Folder accessors:

```typescript
// Model keeps string
componentRoot: string;

// Accessor returns Folder
get componentRootFolder(): FolderJs {
  return new DefaultFolder().init({ path: this.model?.componentRoot });
}
```
