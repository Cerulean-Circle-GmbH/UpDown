## Context: Web4 File Browser PWA Architecture

**Goal**: Build a Unit-based Web4 radical OOP P2P file browser as a PWA.

**Current Problem**: The File and Folder Models contain `LazyReference<T>` types which forces type narrowing complexity throughout Layer 2 (implementation) methods.

**This PDCA**: Evaluates an alternative where `LazyReference` is **contained to Layer 4** (orchestrators only), keeping Layer 2/3 purely synchronous with resolved `Reference<T>` types.

### Web4 Layer Architecture Recap

```
┌─────────────────────────────────────────────────────────────────────┐
│ LAYER 5: VIEWS & CLI                                                │
│ - Lit WebComponents (<folder-default-view>, <file-item-view>)       │
│ - CLI commands                                                      │
│ - Receives ONLY resolved instances                                  │
├─────────────────────────────────────────────────────────────────────┤
│ LAYER 4: ORCHESTRATORS (ASYNC ONLY)                                 │
│ - FileOrchestrator, FolderOrchestrator                              │
│ - Contains LazyReference<T> types                                   │
│ - Resolves IOR strings → IOR objects → Instances                    │
│ - Calls Layer 2/3 methods with resolved references                  │
├─────────────────────────────────────────────────────────────────────┤
│ LAYER 3: INTERFACES (SYNC)                                          │
│ - File, Folder interfaces (compile-time)                            │
│ - FileJs, FolderJs JsInterfaces (runtime)                           │
│ - Models use Reference<T> NOT LazyReference<T>                      │
├─────────────────────────────────────────────────────────────────────┤
│ LAYER 2: IMPLEMENTATION (SYNC)                                      │
│ - DefaultFile, DefaultFolder                                        │
│ - All methods assume resolved references                            │
│ - NO type narrowing needed                                          │
├─────────────────────────────────────────────────────────────────────┤
│ LAYER 1: INFRASTRUCTURE                                             │
│ - IOR, PersistenceManager, UcpComponent                             │
│ - IOR.resolve() returns Promise<T>                                  │
└─────────────────────────────────────────────────────────────────────┘
```

### Option A vs Option B Comparison

| Aspect | Option A (Current) | Option B (This PDCA) |
|--------|-------------------|---------------------|
| **LazyReference in Model** | `FolderModel.children: Collection<File>` (can be IOR strings) | `FolderModel.children: Reference<FileJs>[]` (always resolved) |
| **Type Narrowing** | Required in every Layer 2 method | NOT required — types are always resolved |
| **Four-Case Callback Pattern** | Required (`null`, `string`, `IOR<T>`, `T`) | NOT required — only `T` case exists |
| **Async Handling** | Layer 2 contains `.then()` callbacks | Layer 2 is purely synchronous |
| **Orchestrator Role** | Minimal (just routing) | Substantial (resolution + delegation) |
| **Model Complexity** | Higher (LazyReference types) | Lower (Reference types only) |
| **Code Duplication** | Low (one implementation) | Higher (orchestrator mirrors methods) |

### Why Orchestrators DON'T Implement File/Folder

| Aspect | FileJs/FolderJs | FileOrchestrator/FolderOrchestrator |
|--------|-----------------|-------------------------------------|
| **Purpose** | Represents a file/folder | Manages async operations on file/folder |
| **Methods** | Sync (assumes resolved) | Async (resolves before delegating) |
| **Inheritance** | `extends JsInterface` | Standalone class (wrapper) |
| **Model** | Has `model: FileModel` | Wraps a `DefaultFile` |
| **Runtime Type** | `instanceof FileJs` = true | `instanceof FileJs` = false |
| **RelatedObjects** | Can be registered | CANNOT be registered |

**Key Insight**: Orchestrators are NOT files — they are **managers** that:
1. Hold IOR strings (LazyReferences)
2. Resolve them asynchronously
3. Delegate to the wrapped sync instance

### Model Changes (Option B)

#### Current (Option A): LazyReference in Model

```typescript
// layer3/FolderModel.interface.ts (CURRENT)
interface FolderModel extends FileModel {
  children: Collection<File>;  // Can be string | IOR<FileJs> | FileJs | null
  parent: LazyReference<Folder>;
  linkTarget: LazyReference<Folder>;
}

// layer3/Folder.interface.ts (CURRENT)
interface Folder extends File, Container<File> {
  readonly children: File[];  // FILTERED to only resolved instances
  readonly childReferences: Collection<File>;  // RAW — DRY violation!
}
```

#### Proposed (Option B): Resolved References Only in Model

```typescript
// layer3/FolderModel.interface.ts (OPTION B)
interface FolderModel extends FileModel {
  children: FileJs[];  // ALWAYS resolved instances
  parent: FolderJs | null;  // Resolved or null
  linkTarget: FolderJs | null;  // Resolved or null
}

// layer3/Folder.interface.ts (OPTION B)
interface Folder extends File, Container<File> {
  readonly children: FileJs[];  // NO filtering needed
  // NO childReferences — DRY!
}

// layer4/FolderOrchestrator.ts (NEW)
class FolderOrchestrator {
  private folder: DefaultFolder;
  
  // IOR strings stored ONLY in orchestrator
  private childIORs: string[] = [];
  
  async load(path: string): Promise<void> {
    const scenario = await IOR.load<FolderModel>("ior:file://" + path);
    this.childIORs = scenario.model.childIORs || [];
    this.folder = DefaultFolder.fromScenario(scenario);
    
    // Resolve all children
    for (const ior of this.childIORs) {
      const child = await IOR.resolve<FileJs>(ior);
      this.folder.childAdd(child);  // Pure, sync, type-safe
    }
  }
  
  get children(): FileJs[] {
    return this.folder.children;  // Already resolved
  }
}
```

### Persistence Scenario Structure (Option B)

The persisted scenario stores IOR strings, NOT resolved instances:

```json
{
  "ior": {
    "uuid": "folder-uuid-123",
    "component": "Folder",
    "version": "0.3.22.1"
  },
  "owner": "system",
  "model": {
    "uuid": "folder-uuid-123",
    "name": "layer2",
    "path": "/EAMD.ucp/components/ONCE/0.3.22.1/src/ts/layer2",
    "isFile": false,
    "isFolder": true,
    "childIORs": [
      "ior:file:///EAMD.ucp/components/ONCE/0.3.22.1/src/ts/layer2/DefaultFile.ts",
      "ior:file:///EAMD.ucp/components/ONCE/0.3.22.1/src/ts/layer2/DefaultFolder.ts",
      "ior:file:///EAMD.ucp/components/ONCE/0.3.22.1/src/ts/layer2/UcpComponent.ts"
    ]
  }
}
```

**Key Distinction**:
- `model.childIORs`: IOR strings (for persistence and lazy load)
- `model.children` (at runtime): Resolved `FileJs[]` instances

## A - ACT (Analysis)

### Pros of Option B (Layer 4 Containment)

1. **Type Safety by Design**: Layer 2/3 code CANNOT have unresolved references
2. **Simpler Implementation**: No type guards, no Four-Case Callbacks
3. **Clear Layer Separation**: Async is Layer 4, sync is Layer 2/3
4. **DRY**: One `children` property, not two
5. **Testability**: Layer 2 is pure, easy to test synchronously

### Cons of Option B

1. **Code Duplication**: Orchestrators mirror Layer 2 method signatures
2. **More Files**: FileOrchestrator + FolderOrchestrator in Layer 4
3. **Persistence Split**: Model has both `childIORs` (persisted) and `children` (runtime)
4. **Migration Effort**: Significant refactor of current code

### Pros of Option A (Current Approach)

1. **Single Model**: `children` property serves both persistence and runtime
2. **Fewer Files**: No separate orchestrators needed
3. **ISR Pattern**: Already documented and partially implemented
4. **Gradual Migration**: Can migrate method by method

### Cons of Option A

1. **Type Narrowing Everywhere**: Every method needs Four-Case pattern
2. **DRY Violation**: `children` + `childReferences` duplication
3. **Complexity in Layer 2**: Async callbacks mixed with sync code
4. **Error Prone**: Easy to forget type guards

## Recommendation

**TRON to choose**:
- **Option A**: Continue with current approach (Four-Case Callback Pattern in Layer 2)
- **Option B**: Refactor to contain LazyReference in Layer 4 orchestrators

Both options are viable. The choice depends on:
- **Prefer Option A** if: ISR pattern is core to Web4 philosophy, want single-model simplicity
- **Prefer Option B** if: Type safety is paramount, prefer clear layer separation
