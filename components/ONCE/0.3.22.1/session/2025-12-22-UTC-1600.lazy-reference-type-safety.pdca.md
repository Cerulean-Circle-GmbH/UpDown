# Lazy Reference Type Safety PDCA

**Goal**: Ensure type safety when accessing lazy references in File/Folder components.

**Status**: 🔴 **PLAN — AWAITING REVIEW**

**Parent**: [§/.../ffm-implementation.pdca.md](./2025-12-22-UTC-0500.ffm-implementation.pdca.md) | [GitHub](https://github.com/Cerulean-Circle-GmbH/UpDown/blob/dev/web4v0100/components/ONCE/0.3.22.1/session/2025-12-22-UTC-0500.ffm-implementation.pdca.md)

## Problem Statement

### User Feedback (Quoted)

> "the issue is not that the methods are resolved or not resolved. the issue is that the types used in the methods do not guarantee that its resolved. so we need to call the guardReady function or we need to call the function as a .then like callback when we are sure the resolve has happened and use types that do not allow LazyReferences anymore so we do not get compile errors."

### The Real Issue: Type Safety

The problem is NOT runtime behavior — it's **compile-time type safety**.

```typescript
// FolderModel.interface.ts
children: Collection<File>;  // Collection<LazyReference<File>> internally

// LazyReference<T> = string | IOR<T> | T | null
// So: children[0] might be "ior:..." string, NOT a File instance!
```

When code assumes `.children[i]` is a `File`, TypeScript allows it because `Collection<File>` hides the `LazyReference` wrapper. But at runtime, it might still be a string.

**12 compilation errors** occurred when we changed `children: File[]` to `children: Collection<File>` because existing code does:

```typescript
for (const child of this.model.children) {
  console.log(child.model.uuid);  // ❌ Property 'model' does not exist on type 'string'
}
```

---

## Two Solutions

### Solution A: Guard Pattern (Sync Methods)

Use `guardReady()` as first line — if not resolved, log warning and return `this`.

```typescript
// DefaultFolder.ts
open(): this {
  if (!this.guardReady()) return this;  // Skip if children not resolved
  
  // Now safe: guardReady() guarantees all children are instances
  for (const child of this.children) {
    child.model.isOpen = true;  // ✅ Safe
  }
  return this;
}
```

**Type Guarantee**: `guardReady()` is a runtime check. TypeScript still sees `LazyReference<File>`, but we KNOW at runtime they are resolved.

**Problem**: TypeScript doesn't narrow the type after `guardReady()`. We still get compile errors.

---

### Solution B: Resolved Callback Pattern (Async Flow)

Use `.then()` callback with a type that GUARANTEES resolution.

```typescript
// Define a type that CANNOT be lazy
interface ResolvedFolder {
  children: File[];  // NOT Collection — guaranteed resolved instances
}

// Layer 4 Orchestrator
async openFolder(folder: DefaultFolder): Promise<void> {
  // 1. Await resolution
  await folder.model.awaitResolved();
  
  // 2. Get RESOLVED view of the folder
  const resolved = folder.asResolved();  // Returns ResolvedFolder
  
  // 3. Pass to sync callback — type guarantees resolution
  folder.onOpened(resolved);
}

// DefaultFolder.ts — sync callback
onOpened(resolved: ResolvedFolder): void {
  for (const child of resolved.children) {
    child.model.isOpen = true;  // ✅ Type safe — children is File[]
  }
}
```

**Type Guarantee**: `ResolvedFolder` uses `File[]` not `Collection<File>`, so TypeScript enforces that only resolved instances are passed.

---

## Recommended Approach: Hybrid

| Scenario | Pattern | Type |
|----------|---------|------|
| **Sync methods (Layer 2)** | `guardReady()` + type assertion | `this.children as File[]` |
| **Async callbacks (Layer 4→2)** | Resolved interface | `ResolvedFolder.children: File[]` |
| **Views (Layer 5)** | `isReady` check + loading state | Show spinner when not ready |

## Plan

### Step 1: Define Resolved Interfaces (1h)

**Create**: `src/ts/layer3/ResolvedFolder.interface.ts`

```typescript
import type { File } from './File.interface.js';

/**
 * ResolvedFolder - Folder with all children guaranteed resolved
 * 
 * Used in async callbacks after awaitResolved() completes.
 * Unlike Folder (which uses Collection<File>), this uses File[]
 * guaranteeing all children are instances, not IOR strings.
 */
export interface ResolvedFolder {
  readonly uuid: string;
  readonly name: string;
  readonly path: string;
  readonly children: File[];  // Guaranteed instances
  readonly parent: File | null;  // Guaranteed instance or null
}
```

- [ ] Create `ResolvedFolder.interface.ts`
- [ ] Create `ResolvedFile.interface.ts` (for linkTarget)

### Step 2: Add Type-Safe Accessor to DefaultFolder (1h)

**Add method**: `asResolved(): ResolvedFolder`

```typescript
// DefaultFolder.ts

/**
 * Get resolved view of this folder
 * 
 * PRECONDITION: model.isResolved === true
 * Call only after await model.awaitResolved()
 * 
 * @throws Error if called when not resolved
 */
asResolved(): ResolvedFolder {
  if (!this.model.isResolved) {
    throw new Error(
      `${this.constructor.name}.asResolved() called before resolution. ` +
      `Use await model.awaitResolved() first.`
    );
  }
  
  return {
    uuid: this.uuid,
    name: this.name,
    path: this.path,
    children: this.model.children as File[],  // Safe: isResolved guarantees
    parent: this.model.parent as File | null,
  };
}
```

- [ ] Add `asResolved()` to DefaultFolder
- [ ] Add `asResolved()` to DefaultFile
- [ ] Document precondition in JSDoc

### Step 3: Update Folder Methods with Guard or Callback (2h)

**Categorize 22 `.children` usages in DefaultFolder.ts:**

| Usage Type | Count | Pattern |
|------------|-------|---------|
| Loop iteration | ~10 | Guard + type assertion |
| Single access | ~5 | Guard + type assertion |
| Model assignment | ~5 | No change needed |
| Serialization | ~2 | Guard + type assertion |

**Example migration:**

```typescript
// BEFORE (compile error with Collection<File>)
get children(): FileJs[] {
  const resolved: FileJs[] = [];
  for (const child of this.model.children) {
    if (child !== null && typeof child === 'object' && 'model' in child) {
      resolved.push(child as FileJs);
    }
  }
  return resolved;
}

// AFTER (type-safe)
get children(): Collection<FileJs> {
  return this.model.children;
}

childFindByName(name: string): FileJs | null {
  if (!this.guardReady()) return null;
  
  for (const child of this.children as FileJs[]) {  // Safe: guardReady() checked
    if (child.name === name) return child;
  }
  return null;
}
```

### Step 4: Update Views with Loading State (1h)

**FolderOverView.ts:**

```typescript
render(): string {
  if (!this.component.isReady) {
    return `<folder-over-view class="loading">
```
