## Overview

This PDCA documents the CSS loading verification after significant IOR Infrastructure changes (I.1-I.9).

**Initial Finding**: CSS loading APPEARED working but had hidden issues.  
**Actual Finding**: CSS loading is **BROKEN** due to two critical bugs discovered:

1. **IOR relative URL bug**: `/path` treated as local string instead of remote URL
2. **Orchestrator JSON parsing**: Double-parsing already-parsed JSON objects
3. **Service Worker cache**: Browser serving stale cached code

---

## P - PLAN

### Root Cause Analysis (2025-12-20 Deep Investigation)

**Bug 1: IOR Relative URL Handling**
- **Issue**: `new IOR().init('/asset-manifest')` treated `/asset-manifest` as a LOCAL string value
- **Root Cause**: `isIorStringFormat()` only recognized `ior:`, `http://`, `https://`, `wss://`, `ws://`
- **Fix**: Added `str.startsWith('/')` to recognize relative URLs as remote references
- **Status**: ✅ FIXED in IOR.ts

**Bug 2: Orchestrator Double JSON Parsing**
- **Issue**: `IOR.load()` returns parsed JSON object, but orchestrator called `JSON.parse()` again
- **Root Cause**: `JSON.parse()` on an object coerces to string `"[object Object]"` which fails
- **Error**: `SyntaxError: Unexpected token '/', "/asset-manifest" is not valid JSON`
- **Fix**: Check `typeof result === 'string'` before parsing
- **Status**: ✅ FIXED in BrowserOnceOrchestrator.ts

**Bug 3: Service Worker Caching**
- **Issue**: Browser serves cached old JS even after server restart
- **Root Cause**: Service Worker intercepts all requests and serves stale files
- **Status**: ⚠️ Requires manual cache clear for testing

**IOR.ts - isIorStringFormat():**

```typescript
private isIorStringFormat(str: string): boolean {
    return str.startsWith('ior:') || 
           str.startsWith('http://') || 
           str.startsWith('https://') ||
           str.startsWith('wss://') ||
           str.startsWith('ws://') ||
           str.startsWith('/');  // NEW: Relative URLs are remote references
}
```

**IOR.ts - parseRelativeUrl():**
```typescript
private parseRelativeUrl(path: string): this {
    const isBrowser = typeof window !== 'undefined' && window.location;
    let protocol = 'https';
    let host = 'localhost';
    let port = 443;
    
    if (isBrowser) {
        protocol = window.location.protocol.replace(':', '');
        host = window.location.hostname;
        port = parseInt(window.location.port) || (protocol === 'https' ? 443 : 80);
    }
    
    this.model.protocol = protocol;
    this.model.host = host;
    this.model.port = port;
    this.model.path = path;
    this.model.iorString = `${protocol}://${host}:${port}${path}`;
    return this;
}
```

## A - ACT

### ✅ Resolution Summary

The CSS loading issue has been **FULLY RESOLVED**. Two critical bugs were fixed:

1. **IOR relative URL handling** — Fixed `isIorStringFormat()` to recognize `/path` as remote
2. **Orchestrator double parsing** — Fixed to handle both string and object responses from IOR.load()

### Learnings

**L20: IOR Relative URLs Need Browser Context**
- Relative URLs (`/path`) need window.location to get current host/port
- In Node.js, defaults to `localhost:443`
- In browser, reads from `window.location.{protocol,hostname,port}`

**L21: IOR.load() Returns Parsed JSON**
- Don't double-parse: `IOR.load<any>()` returns object, not string
- Always check `typeof result === 'string'` before `JSON.parse()`

**L22: Service Worker Caches Aggressively**
- SW intercepts ALL requests, including JS module imports
- Changes to server files don't propagate until SW is updated/cleared
- Testing requires explicit cache clearing strategy
