## Overview

**Core Vision**: IOR = Reference — One unified concept for local AND remote objects.

Every resource access (files, CSS, WebSocket, REST, scenarios) goes through IOR.
IOR internally resolves protocol chains and delegates to the appropriate Loader.

**Key Insight (TRON 2025-12-20)**: IOR and Reference are the SAME thing.
- Both can be local or remote
- Artificial separation is confusing
- Consolidate: Reference = interface, IOR = layer4 implementation

```
┌─────────────────────────────────────────────────────────────────┐
│            Reference (interface) = IOR (implementation)         │
├─────────────────────────────────────────────────────────────────┤
│  State: UNRESOLVED ──→ RESOLVING ──→ RESOLVED                   │
│                                                                 │
│  new IOR().init("ior:wss:https://host:port/ws").resolve()       │
│                           ↓                                     │
│  Protocol Chain: wss → https                                    │
│                           ↓                                     │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│  │ WebSocket   │  │  HTTPS      │  │  Scenario   │             │
│  │  Loader     │→ │  Loader     │→ │  Loader     │             │
│  └─────────────┘  └─────────────┘  └─────────────┘             │
│                           ↓                                     │
│  Nested References: Immediately converted to local              │
│  Next Level: Prefetched async (non-blocking)                    │
└─────────────────────────────────────────────────────────────────┘
```

## P - PLAN

### **Phase I.1: Reference/IOR Consolidation** ← REVISED

---

#### **🔴 TRON Feedback (2025-12-20)**:

> "what is the difference between an IOR: Internet Object Reference and a Reference Interface and a DefaultReference. this should be all the same. its just artificial separation and very confusing. so an IOR can be local or remote exactly like a DefaultReference. when an IOR is yet remote, and will be accessed it needs to resolve itself to a local reference. so if the object referenced contains remote references, then the references are immediately converted to local references. when accessing these references, the next level of references is prefetched."
>
> "consolidate it all - Reference as an Interface and IOR as the implementation in layer4. try to do the remote reference resolution as much as possible async nonblocking and let the IOR track if it is during resolution or already resolved."

---

#### **Architectural Insight**:

**IOR = Reference** — They are the SAME concept:
- Both can be local OR remote
- Both need resolution to get the actual object
- The artificial separation is confusing

**Consolidated Design**:
```
layer3/Reference.interface.ts    ← Interface (local or remote)
layer4/IOR.ts                    ← Implementation (async resolution)
```

**Resolution Flow**:
```
┌─────────────────────────────────────────────────────────────────┐
│  Reference (interface)                                          │
│  ├── isLocal: boolean                                           │
│  ├── isRemote: boolean                                          │
│  ├── isResolving: boolean    ← NEW: async resolution tracking   │
│  ├── isResolved: boolean     ← NEW: resolution complete         │
│  ├── value: T | null         ← sync access (null if remote)     │
│  └── resolve(): Promise<T>   ← async, non-blocking              │
└─────────────────────────────────────────────────────────────────┘
                               ↓
┌─────────────────────────────────────────────────────────────────┐
│  IOR (layer4 implementation)                                    │
│  ├── When remote IOR resolves:                                  │
│  │   1. Object loaded                                           │
│  │   2. All nested remote refs → converted to local refs        │
│  │   3. Next level prefetched (non-blocking)                    │
│  └── State: UNRESOLVED → RESOLVING → RESOLVED                   │
└─────────────────────────────────────────────────────────────────┘
```

**Prefetch Pattern**:
```typescript
// When IOR resolves an object with nested references:
const user = await userRef.resolve();
// user.address: Reference<Address> ← immediately becomes local
// user.orders: Reference<Order[]>  ← prefetch starts (non-blocking)
```

### **Phase I.6: Unit/File/Folder as Standard Scenarios**

**Vision**: Every file touched by StaticFileRoute becomes a Unit scenario

```
File Request: /ONCE/0.3.22.1/src/ts/layer5/views/css/ItemView.css
           ↓
StaticFileRoute:
  1. Serve file content
  2. Create Unit scenario (if not exists):
     {
       ior: { uuid: "...", class: "Unit", ... },
       model: {
         componentType: "CSSFile",
         componentIor: "ior:esm:/ONCE/0.3.22.1/css/ItemView",
         contentHash: "sha256:...",
         ...
       }
     }
  3. ServiceWorker caches Unit
           ↓
Next Request: ServiceWorker serves from cache
```

**FileModel + UnitModel Integration**:
```typescript
interface FileModel extends Model {
    // ... existing fields ...
    unit?: Reference<UnitModel>;  // Optional linked Unit
}

interface FolderModel extends Model {
    // ... existing fields ...
    unit?: Reference<UnitModel>;  // Folder as Unit for sync
}
```
