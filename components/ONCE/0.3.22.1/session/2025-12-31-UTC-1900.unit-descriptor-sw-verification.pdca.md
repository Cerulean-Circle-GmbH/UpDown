### P.1b: JS Units with TS Origin — FUTURE ENHANCEMENT

**Observation**: `UnitModel` already has an `origin` field:

```typescript
// UnitModel.interface.ts lines 39-40:
/** IOR string: Source file/folder reference (GitTextIOR format) */
origin: string;
```

**Enhancement Plan**:

When creating a Unit for a `.js` file in `dist/`, set its `origin` to the source `.ts` file:

| JS File (definition) | TS File (origin) |
|---------------------|------------------|
| `dist/ts/layer2/DefaultFile.js` | `src/ts/layer2/DefaultFile.ts` |
| `dist/ts/layer4/IOR.js` | `src/ts/layer4/IOR.ts` |

**Implementation** (future PDCA):

```typescript
// In UnitDiscoveryService.unitCreate() — extend for JS files
if (definition.filename.endsWith('.js') && definition.relativePath.startsWith('dist/ts/')) {
  // Calculate origin: dist/ts/... → src/ts/...
  const originPath = definition.relativePath
    .replace('dist/ts/', 'src/ts/')
    .replace('.js', '.ts');
  
  scenario.model.origin = `/EAMD.ucp/components/${componentName}/${version}/${originPath}`;
}
```

**Benefits**:
1. **Traceability**: JS → TS source mapping
2. **Debugging**: Know which TS to edit when debugging JS
3. **Incremental builds**: Detect TS changes → regenerate JS units
4. **Source maps alignment**: Complements TypeScript source maps

**Priority**: 🔮 **FUTURE** — not blocking for v0.3.22.1
