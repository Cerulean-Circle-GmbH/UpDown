```

│ FILE BROWSER PWA — COMPLETE USER JOURNEY                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│ 1. User opens: https://host:42000/EAMD.ucp/components/ONCE/0.3.22.1 │
│                                                                     │
│ 2. Server serves SPA HTML with <folder-over-view>                   │
│                                                                     │

│    - Fetch scenario JSON (?format=json)                             │
│    - Instantiate DefaultFolder                                      │
```

## Option A: Four-Case Callback Pattern in Layer 2

### Architecture

```
Layer 2 (DefaultFolder)
├── model.children: Collection<LazyReference<FileJs>>
├── Every method has type guards + Four-Case callbacks
└── .then() callbacks inside Layer 2 methods
```

### Web4 Principle Compliance

| Principle | Status | Issue |
|-----------|--------|-------|
| **P4b: Polymorphism** | ⚠️ PARTIAL | Type checking (`typeof`, `instanceof`) instead of polymorphic dispatch |
| **P7: Async Only in Layer 4** | ❌ VIOLATED | `.then()` callbacks inside Layer 2 methods |
| **P8: DRY** | ❌ VIOLATED | `children` + `childReferences` duplication |
| **P34: ONE LAYER LOOKAHEAD** | ✅ OK | Pattern supports it |
| **P35: JsInterface** | ✅ OK | FileJs/FolderJs implemented |

### Code Example (Violations Highlighted)

```typescript
// Layer 2 — DefaultFolder.ts
childAdd(child: LazyReference<FileJs>): void {
  if (child === null) { ... }                    // ❌ Type checking, not polymorphism
  if (typeof child === 'string') {
    IOR.resolve(child).then(this.onChildAdd);    // ❌ Async (.then) in Layer 2!
    return;
  }
  if ('isResolving' in child) { ... }            // ❌ More type checking
  // Finally the real case
  this.onChildAdd(child);
}
```

### Verdict: ❌ NOT RECOMMENDED

- Violates **P7** (async in Layer 2)
- Violates **P4b** (type checking instead of polymorphism)
- Violates **P8** (DRY — dual children properties)

## Option B: Per-Component Orchestrators in Layer 4

### Architecture

```
Layer 4 (FolderOrchestrator)
├── Holds IOR strings
├── Resolves before delegating to Layer 2
└── Wraps DefaultFolder

Layer 2 (DefaultFolder)
├── model.children: FileJs[] (always resolved)
├── Pure sync methods
└── No type guards needed
```

### Web4 Principle Compliance

| Principle | Status | Issue |
|-----------|--------|-------|
| **P4b: Polymorphism** | ✅ OK | No type checking in Layer 2 |
| **P7: Async Only in Layer 4** | ✅ OK | Orchestrator handles async |
| **P8: DRY** | ❌ VIOLATED | Orchestrator duplicates Layer 2 method signatures |
| **P34: ONE LAYER LOOKAHEAD** | ⚠️ COMPLEX | Orchestrator must manage lookahead |
| **P35: JsInterface** | ⚠️ ISSUE | Orchestrator doesn't implement FileJs |

### Code Example

```typescript
// Layer 4 — FolderOrchestrator.ts
class FolderOrchestrator {
  private folder: DefaultFolder;
  private childIORs: string[];
  
  async childAdd(ior: string): Promise<void> {        // ✅ Async in Layer 4
    const resolved = await IOR.resolve<FileJs>(ior);
    this.folder.childAdd(resolved);                    // ✅ Sync call to Layer 2
  }
  
  // ❌ Duplicates every method from DefaultFolder
  async childRemove(ior: string): Promise<boolean> { ... }
  async childGet(uuid: string): Promise<FileJs | null> { ... }
  async move(newPath: string): Promise<void> { ... }
}
```

### Verdict: ⚠️ ACCEPTABLE BUT SUBOPTIMAL

- ✅ P7 compliant (async in Layer 4)
- ❌ Violates P8 (DRY — every component needs orchestrator)
- ⚠️ Kernel doesn't help — each component reinvents resolution

## Option C: Kernel ISR (Self-Replacing IOR)

### Architecture

```
Kernel (UcpModel proxy + IOR)
├── Automatically converts IOR strings → IOR objects → instances
├── Universal for ALL components
└── Fire-and-forget resolution with self-replacement

Layer 4 (FolderOrchestrator — THIN)
├── Only for complex multi-component operations
├── Triggers initial folder load
└── Delegates to kernel for resolution

Layer 2 (DefaultFolder)
├── model.children: (FileJs | IOR<FileJs>)[]
├── Pure sync methods
└── No type guards — polymorphic via views

Layer 5 (FolderOverView)
├── Sync only — reacts to model changes
├── Polymorphic rendering: UcpView.tagFor(child)
└── IOR has its own view: <ior-loading-view>
```

### Web4 Principle Compliance

| Principle | Status | Notes |
|-----------|--------|-------|
| **P4a: Objects over functions** | ✅ OK | All methods on objects, no standalone functions |
| **P4b: Polymorphism** | ✅ OK | `UcpView.tagFor(child)` dispatches polymorphically |
| **P7: Async Only in Layer 4** | ✅ OK | Orchestrator triggers, kernel resolves in background |
| **P8: DRY** | ✅ OK | Kernel handles resolution universally |
| **P26: No Factory Functions** | ✅ OK | `new IOR().init()` pattern |
| **P27: Web Components ARE OOP** | ✅ OK | Views render polymorphically |
| **P28: JsInterface Pattern** | ✅ OK | IOR is a first-class type with its own view |
| **P34: IOR Unified Entry Point** | ✅ OK | All resolution via IOR |
| **P34: ONE LAYER LOOKAHEAD** | ✅ OK | Kernel handles automatically |
| **P35: JsInterface** | ✅ OK | FileJs, FolderJs, IOR all have runtime existence |

### Code Example (Fully Compliant)

```typescript
// Layer 4 — FolderOrchestrator.ts (THIN)
export class FolderOrchestrator {
  
  /**
   * Trigger folder load — async lives here
   */
  folderLoad(path: string, view: FolderOverView): void {
    const ior = new IOR<DefaultFolder>().init("ior:file://" + path);
    
    // Kernel handles resolution, self-replacement, view update
    ior.resolve().then(function onResolved(folder: DefaultFolder): void {
      view.folderSet(folder);  // ✅ Sync callback to Layer 5
    });
  }
}

// Layer 5 — FolderOverView.ts (SYNC ONLY)
@customElement('folder-over-view')
export class FolderOverView extends UcpView<FolderModel> {
  
  private orchestrator = new FolderOrchestrator();
  
  folderLoadTrigger(path: string): void {
    this.orchestrator.folderLoad(path, this);  // ✅ Sync call
  }
  
  folderSet(folder: DefaultFolder): void {
    this.folder = folder;  // ✅ Sync, reactive
  }
  
  childrenRender(): TemplateResult[] {
    const parts: TemplateResult[] = [];
    for (const child of this.folder.model.children) {
      // ✅ Polymorphic dispatch — no type checking!
      const tag = UcpView.tagFor(child);
      parts.push(html`<${unsafeStatic(tag)} .instance=${child}></${unsafeStatic(tag)}>`);
    }
    return parts;
  }
}

// IOR has its own view — polymorphic loading state
@customElement('ior-loading-view')
export class IORLoadingView extends UcpView<IORModel> {
  render(): TemplateResult {
    return html`<div class="loading-item"><span class="spinner"></span></div>`;
  }
}
```

### Verdict: ✅ RECOMMENDED

- ✅ Fully P7 compliant (async only in Layer 4)
- ✅ Fully P4b compliant (polymorphic, no type checks)
- ✅ Fully P8 compliant (DRY — kernel handles all)
- ✅ Fully P34 compliant (IOR everywhere, ONE LAYER)
- ✅ Already planned (URP PDCA has implementation tasks)

## Final Recommendation

### 🅲 OPTION C: Kernel ISR (Self-Replacing IOR)

**Why Option C is the correct Web4 architecture**:

1. **Universal Mechanism**: Kernel handles resolution for ALL components — no per-component orchestrators
2. **Layer 2 Stays Pure**: No async, no type guards, no callbacks in business logic
3. **Polymorphic Rendering**: IOR is a first-class type with its own view — no if/else
4. **P34 Full Compliance**: IOR everywhere, self-replacement, ONE LAYER LOOKAHEAD automatic
5. **Already Planned**: URP PDCA defines the kernel implementation tasks
6. **Future-Proof**: New components get ISR for free — just define model with Reference<T>

### Implementation Order

```
1. URP.2: Self-replacing IOR in UcpModel.get (1.5h)
2. URP.3: IOR.initWithParent() + resolveAndReplace() (1h)
3. URP.4: Array proxy for Collection<IOR> (30m)
4. URP.5: Simplify DefaultFolder.resolve() (30m)
5. C.1: Update FolderModel.children type (30m)
6. C.2: Add IORLoadingView component (30m)
7. C.3: Update FolderOverView to use polymorphic rendering (1h)
```

**Total**: ~5.5h of kernel + component work
