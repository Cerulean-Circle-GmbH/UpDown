### Root Cause Analysis

**File**: `src/ts/layer4/FileOrchestrator.ts` line 451-458

```typescript:451:460:src/ts/layer4/FileOrchestrator.ts
async folderLoadFromServer(folderPath: string): Promise<{ contentType: string; data: string }> {
    const { IOR } = await import('./IOR.js');
    
    // Use ?format=json to bypass SPA catch-all route
    const fetchUrl = folderPath + (folderPath.includes('?') ? '&' : '?') + 'format=json';
    
    const ior = await new IOR().init(fetchUrl);
    const data = await ior.load<string>();
    
    // ❌ BUG: data might be an object, not a string!
    const isJson = data.trim().startsWith('{') || data.trim().startsWith('[');
    const contentType = isJson ? 'application/json' : 'text/html';
    
    return { contentType, data };
}
```

**Flow**:
1. `IOR.init('/EAMD.ucp/components/ONCE/0.3.22.1/src/assets?format=json')`
2. `IOR.parseRelativeUrl()` parses path parts: `['EAMD.ucp', 'components', 'ONCE', ...]`
3. **Bug**: Sets `model.uuid = 'ONCE'` (third segment!)
4. `IOR.load()` calls `resolve()` which calls `cacheCheck()`
5. `cacheCheck()` finds cached `ONCE.component.json` by UUID "ONCE"
6. Returns `scenario.model` which is **already a parsed JSON object**
7. `data.trim()` fails: `TypeError: data.trim is not a function`

**Console Evidence**:
```
💾 [IOR] Cache hit: ONCE  ← Wrong UUID match!
[FolderOverView] Load error: TypeError: data.trim is not a function
```

```typescript
async folderLoadFromServer(folderPath: string): Promise<{ contentType: string; data: any }> {
    const { IOR } = await import('./IOR.js');
    
    const fetchUrl = folderPath + (folderPath.includes('?') ? '&' : '?') + 'format=json';
    const ior = await new IOR().init(fetchUrl);
    const rawData = await ior.load<string | object>();
    
    // Handle both string (fresh fetch) and object (cached)
    if (typeof rawData === 'object') {
        // Already parsed - return directly
        return { contentType: 'application/json', data: rawData };
    }
    
    // String - determine type
    const isJson = rawData.trim().startsWith('{') || rawData.trim().startsWith('[');
    const contentType = isJson ? 'application/json' : 'text/html';
    
    return { contentType, data: rawData };
}
```

**Pros**: Simple, handles both cases  
**Cons**: Doesn't fix the root UUID collision problem

### Option B: Bypass IOR Cache for Directory Listings (Better)

**File**: `src/ts/layer4/FileOrchestrator.ts`

```typescript
async folderLoadFromServer(folderPath: string): Promise<{ contentType: string; data: string }> {
    // Direct fetch without IOR caching - directory listings shouldn't be cached
    const fetchUrl = folderPath + (folderPath.includes('?') ? '&' : '?') + 'format=json';
    
    const response = await fetch(fetchUrl);
    if (!response.ok) {
        throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
    
    const data = await response.text();
    const isJson = data.trim().startsWith('{') || data.trim().startsWith('[');
    const contentType = isJson ? 'application/json' : 'text/html';
    
    return { contentType, data };
}
```

**Pros**: Avoids UUID collision, directory listings are dynamic  
**Cons**: ❌ Violates P2P abstraction (views never fetch directly)

### Option C: Fix parseRelativeUrl() UUID Detection (Root Fix)

**File**: `src/ts/layer4/IOR.ts` line 1034-1044

The current logic assumes all paths follow `/{component}/{version}/{uuid}/...` but file browser paths are `/{EAMD.ucp}/components/{component}/{version}/{path}/...`.

```typescript
private parseRelativeUrl(path: string): this {
    // ... existing code ...
    
    // Parse path parts
    const pathParts = path.split('/').filter(p => p.length > 0);
    
    // DON'T assume position = semantic meaning
    // UUID is ONLY valid if it looks like a UUID (36 chars, dashes)
    const uuidPattern = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
    const maybeUuid = pathParts[2] || '';
    
    this.model.uuid = uuidPattern.test(maybeUuid) ? maybeUuid : '';  // Only set if valid UUID
    // ... rest unchanged ...
}
```

**Pros**: Fixes root cause, doesn't affect valid UUID paths  
**Cons**: More complex change
