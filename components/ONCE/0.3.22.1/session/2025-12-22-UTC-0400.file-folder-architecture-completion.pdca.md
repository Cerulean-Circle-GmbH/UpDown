### Unit-Aware File Operations

Files create Units on **first touch**. On subsequent touches, they **rediscover** their existing Unit.

**Critical Operation: Unit-Aware Move**

```typescript
// Moving a file MUST update:
// 1. The file's model.path
// 2. The Unit's model.references[].path
// 3. All symlinks pointing to this file (in §/scenarios/type/, §/scenarios/domain/, etc.)

async pathMove(newPath: string): Promise<void> {
  const oldPath = this.model.path;
  
  // 1. Update file path
  this.model.path = newPath;
  
  // 2. Update Unit references (if Unit exists)
  const unit = await this.unitGet();
  if (unit) {
    for (const ref of unit.model.references) {
      if (ref.path === oldPath) {
        ref.path = newPath;
      }
    }
    await unit.save();
  }
  
  // 3. Update symlinks in §/scenarios/type/
  await this.symlinkPathsUpdate(oldPath, newPath);
  
  await this.save();
}
```

### TRON Feedback (2025-12-22-UTC-0600)

> "Reference<...> can be the local resolved instance (in our case DefaultFile or DefaultFolder) or the IOR or the full persistent scenario. ### Reference<T> Can Be THREE States (IOR = Internet Object REFERENCE)

```typescript
// Reference<T> holds THREE possible states:
type Reference<T> = 
  | string               // 1. IOR string — "ior:scenario:{uuid}" (can load scenario)
  | Scenario<Model>      // 2. Scenario JSON — { ior, uuid, model, owner } (can instantiate)
  | T                    // 3. Instance — DEREFERENCED (DefaultFile, DefaultFolder)
  | null;                // Not yet set

// DEREFERENCING CHAIN:
// IOR string → Scenario → Instance
//    ↓            ↓           ↓
//  "ior:..."   { model }   DefaultFile (DEREFERENCED)
```

### ONE LAYER LOOKAHEAD: Folder Resolved = Children Resolved

```
┌─────────────────────────────────────────────────────────────────────────────┐
│              ONE LAYER LOOKAHEAD — FOLDER RESOLUTION PATTERN                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  KEY INSIGHT: A folder is NOT "resolved" until its children ARE resolved!  │
│  Getting a folder = prefetch its children. ONE LAYER LOOKAHEAD.            │
│                                                                             │
│  1. REQUEST RootFolder → Resolve folder AND its children (lookahead)        │
│     ┌───────────────────────────────────────────────────────────────────┐   │
│     │ RootFolder (RESOLVED)                                             │   │
│     │   children: [DefaultFile✓, DefaultFolder✓, DefaultFile✓]          │   │
│     │             ↑ PREFETCHED — all are UcpComponent instances         │   │
│     │                                                                   │   │
│     │   BUT: DefaultFolder.children = ["ior:...", "ior:..."]            │   │
│     │                                  ↑ UNRESOLVED — still IOR strings │   │
│     │                                  (grandchildren not prefetched)   │   │
│     └───────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  2. USER SELECTS child folder (via ItemView or variable assignment)        │
│     ┌───────────────────────────────────────────────────────────────────┐   │
│     │ ChildFolder (NOW RESOLVING...)                                    │   │
│     │   children: ["ior:..."] → resolve → [DefaultFile✓, DefaultFile✓]  │   │
│     │             ↑ PREFETCHING children (one layer lookahead)          │   │
│     │                                                                   │   │
│     │   DefaultFile.parent = ChildFolder ✓                              │   │
│     │   (grandchildren still IOR strings)                               │   │
│     └───────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  TRIGGER FOR NEXT LAYER:                                                    │
│  - User clicks ItemView (selection)                                         │
│  - Assigning child to variable for operation                                │
│  - Any access that requires the child's children                            │
│                                                                             │
│  PRINCIPLE: Folder "resolved" means children are instances, not IORs.       │
│  BENEFIT: Immediate rendering without loading spinners for visible items.   │
```

### IOR String vs Scenario JSON: Two Stages of Resolution

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ IOR String                      │ Scenario JSON                            │
├─────────────────────────────────┼──────────────────────────────────────────┤
│ "ior:scenario:abc-123-..."      │ { ior: "...", uuid: "abc-123-...",       │
│                                 │   model: { name: "file.ts", ... },       │
│                                 │   owner: "base64..." }                   │
├─────────────────────────────────┴──────────────────────────────────────────┤
│                                                                             │
│ STAGE 1: IOR String                                                         │
│   - Compact reference (just enough to LOCATE the scenario)                  │
│   - Requires network/storage fetch to get full data                         │
│   - kernel.loadScenario(ior) → Scenario                                     │
│                                                                             │
│ STAGE 2: Scenario JSON                                                      │
│   - Full data (enough to INSTANTIATE the component)                         │
│   - Contains model, owner, uuid                                             │
│   - kernel.instantiate(scenario) → Instance                                 │
│                                                                             │
│ STAGE 3: Instance (DEREFERENCED)                                            │
│   - Live UcpComponent with model data                                       │
│   - Ready to use, call methods, render                                      │
│                                                                             │
│ Reference<T> can hold ANY of these three + null!                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Implementation: ONE LAYER LOOKAHEAD Resolution

```typescript
// In DefaultFolder.ts — ONE LAYER LOOKAHEAD pattern

// Model stores IOR strings (unresolved grandchildren)
// FolderModel.children: Collection<string>  // e.g., ["ior:scenario:uuid1", ...]

// Cache stores resolved child instances
private childrenCache: Map<string, FileSystemNode> = new Map();

/**
 * Resolve this folder — includes ONE LAYER LOOKAHEAD for children
 * 
 * A folder is NOT "resolved" until its children are instances!
 * This is called automatically when folder is accessed.
 */
async resolve(): Promise<this> {
  // Resolve all children to instances (ONE LAYER LOOKAHEAD)
  await this.childrenResolve();
  return this;
}

/**
 * Resolve child IORs to UcpComponent instances
 * Called as part of folder resolution (ONE LAYER LOOKAHEAD)
 * 
 * NOTE: Children's children remain as IOR strings (not resolved yet)
 */
async childrenResolve(): Promise<Collection<FileSystemNode>> {
  for (const iorString of this.model.children as string[]) {
    // Skip already resolved
    const uuid = iorString.replace('ior:scenario:', '');
    if (this.childrenCache.has(uuid)) continue;
    
    // Resolve IOR → Scenario → UcpComponent instance
    const scenario = await new IOR<Scenario<FileModel | FolderModel>>()
      .initRemote(iorString)
      .resolve();
    
    if (scenario) {
      // Create instance — but DON'T resolve grandchildren yet!
      const child = this.instanceFromScenario(scenario);
      child.parentSet(this);
      this.childrenCache.set(uuid, child);
      // child.model.children = ["ior:..."] — STILL unresolved (lazy)
    }
  }
  return Array.from(this.childrenCache.values());
}

/**
 * Select a child — triggers NEXT LAYER resolution
 * 
 * Called when:
 * - User clicks ItemView
 * - Child assigned to variable for operation
 * - Any access requiring child's children
 */
async childSelect(uuid: string): Promise<Reference<FileSystemNode>> {
  const child = this.childrenCache.get(uuid);
  if (!child) return null;
  
  // If child is a folder, resolve ITS children (next layer lookahead)
  if (child instanceof DefaultFolder) {
    await child.resolve();  // This prefetches grandchildren
  }
  
  return child;
}

/**
 * Create File or Folder instance from scenario
 * Instance is created but children NOT resolved (lazy)
 */
private instanceFromScenario(scenario: Scenario<any>): FileSystemNode {
  if ('folderName' in scenario.model) {
    const folder = new DefaultFolder();
    folder.init();
    Object.assign(folder.model, scenario.model);
    // folder.model.children = IOR strings — NOT resolved!
    return folder;
  } else {
    const file = new DefaultFile();
    file.init();
    Object.assign(file.model, scenario.model);
    return file;
  }
}
```

### Web4TSComponent: Single Source of Truth for Component Anatomy

**Responsibility**: Web4TSComponent recognizes components and their units by naming convention.

```typescript
// Web4TSComponent is responsible for:
// 1. Recognizing component files by naming convention
// 2. Creating Units for each file on first touch
// 3. Creating BOTH symlink paths immediately:
//    - Current: §/scenarios/type/{CurrentComponent}/{version}/...
//    - Future:  §/scenarios/type/{FutureComponent}/{SAME version}/... (if extractable)
// 4. Maintaining unit consistency across all paths
// 5. VERSION INHERITANCE: Future component gets SAME version as parent
```

**Naming Convention for Extractable Components**:
- `Default{ComponentName}.ts` in `ParentComponent/{version}/` → Future `{ComponentName}/{version}/`
- Example: `DefaultFile.ts` in `ONCE/0.3.22.1/` → Future `File/0.3.22.1/`
- Example: `DefaultFolder.ts` in `ONCE/0.3.22.1/` → Future `Folder/0.3.22.1/`

**Version Inheritance Rule**:
- Extracted components INHERIT the version from the parent component at extraction time
- After extraction, they evolve independently with their own versioning

NOTE: §/scenarios/ is THE ONLY location for scenarios.

```typescript
  childAdd(child: FileSystemNode): void;
  childRemove(child: FileSystemNode): boolean;
  
  // NEW: Replace fs.* sync calls (stats via this.model.*)
  exists(): boolean;              // fs.existsSync()
  get isFile(): boolean;          // Always false (Folder is not a file)
  get isDirectory(): boolean;     // Always true (Folder is a directory)
  // Stats: this.model.createdAt, this.model.modifiedAt, this.model.isLink
  create(): Promise<void>;        // fs.mkdirSync() + Unit creation
  delete(recursive?: boolean): Promise<void>;  // Unit-aware delete
  list(): Collection<FileSystemNode>; // fs.readdirSync()
  linkTo(target: string): Promise<Folder>; // fs.symlinkSync()
  
  // Unit-aware tree operations
  pathMove(newPath: string): Promise<void>; // Recursively updates all Units!
  
  // Tree navigation (P4a: No arrow functions)
  findByPath(path: string): Reference<FileSystemNode>;
  walkDepthFirst(visitor: FileSystemNodeVisitor): void;
  walkBreadthFirst(visitor: FileSystemNodeVisitor): void;
}

// P4a compliant visitor class
interface FileSystemNodeVisitor {
  visit(node: FileSystemNode): void;
}
```
