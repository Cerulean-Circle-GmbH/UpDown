## Context

**Goal**: Build a Unit-based Web4 radical OOP P2P file browser as a PWA.

**Problem**: Options A and B both have drawbacks:
- **Option A**: Four-Case Callback Pattern pollutes Layer 2 with type narrowing
- **Option B**: Per-component orchestrators (FileOrchestrator, FolderOrchestrator) duplicate kernel functionality

**This PDCA**: Option C uses the **kernel's universal ISR mechanism** (Self-Replacing IOR objects) so Layer 2 is pure AND no per-component orchestrators are needed.

### The Universal ISR Mechanism (from URP PDCA)

**Key Insight**: The kernel already has a plan for universal lazy reference handling via **Self-Replacing IOR Objects**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    KERNEL ISR (SELF-REPLACING IOR)                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  model.children = ['ior:scenario:file-1', 'ior:scenario:folder-2']  │
│                                                                     │
│  Step 1: First Access                                               │
│  ─────────────────────                                              │
│  const child = folder.model.children[0];                            │
│    │                                                                │
│    ├── UcpModel.get() intercepts                                    │
│    ├── Detects 'ior:...' string                                     │
│    ├── Creates IOR object: new IOR().initRemote(iorString)          │
│    ├── Sets parent reference: ior.initWithParent(model, 'children', 0)│
│    ├── Replaces string with IOR in array: children[0] = ior        │
│    ├── Triggers resolution: ior.resolveAndReplace() (fire-and-forget)│
│    └── Returns IOR object (sync)                                    │
│                                                                     │
│  Step 2: Background Resolution                                      │
│  ─────────────────────────────                                      │
│  ior.resolveAndReplace():                                           │
│    │                                                                │
│    ├── const instance = await this.resolveInstance()                │
│    ├── SELF-REPLACEMENT: children[0] = instance                     │
│    └── UcpModel.set triggers view re-render                         │
│                                                                     │
│  Step 3: Subsequent Access                                          │
│  ─────────────────────────                                          │
│  const child = folder.model.children[0];                            │
│    │                                                                │
│    └── Returns DefaultFile instance directly (already resolved)     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### How Option C Differs from Option B

**Option B**: Per-component orchestrators hold IOR strings and resolve them:

```typescript
// Option B: FolderOrchestrator (separate class)
class FolderOrchestrator {
  private childIORs: string[] = [];  // IORs stored HERE
  private folder: DefaultFolder;
  
  async load(path: string): Promise<void> {
    const scenario = await IOR.load(path);
    this.childIORs = scenario.model.childIORs;
    this.folder = new DefaultFolder().init(scenario);
    
    // Orchestrator resolves each child
    for (const ior of this.childIORs) {
      const child = await IOR.resolve(ior);
      this.folder.childAdd(child);
    }
  }
}
```

**Option C**: Kernel handles resolution universally, no orchestrator needed:

```typescript
// Option C: Kernel does it all
async function loadFolder(path: string): Promise<DefaultFolder> {
  // IOR.instantiate() + UcpModel proxy handles everything
  return await IOR.resolve<DefaultFolder>("ior:file://" + path);
}

// In Layer 5 view:
const folder = await loadFolder("/EAMD.ucp/components/ONCE/0.3.22.1");

// First access to children triggers ISR automatically:
for (const child of folder.model.children) {
  if (child instanceof IOR) {
    // Still resolving — show loading spinner
    await child;  // Can await the IOR directly
  } else {
    // Already resolved — use instance
    renderChild(child);
  }
}
```

### What Layer 2 Code Looks Like (Option C)

**DefaultFolder.ts — PURE, NO TYPE GUARDS**:

```typescript
export class DefaultFolder extends UcpComponent<FolderModel> implements FolderJs {
  
  /**
   * Get children — returns array that may contain:
   * - FileJs instances (resolved)
   * - IOR<FileJs> objects (resolving)
   * 
   * Kernel's UcpModel proxy handles the transition automatically.
   */
  get children(): (FileJs | IOR<FileJs>)[] {
    return this.model.children;  // Kernel handles ISR
  }
  
  /**
   * Add a child — SYNC, PURE
   * 
   * Accepts:
   * - FileJs instance (already resolved)
   * - string (IOR string — kernel will handle)
   */
  childAdd(child: FileJs | string): void {
    this.model.children.push(child);
    this.model.modifiedAt = Date.now();
    
    // If resolved instance, set parent immediately
    if (typeof child !== 'string' && !(child instanceof IOR)) {
      child.parentSet(this);
    }
    // If string or IOR, kernel's ISR will handle parentSet when resolved
  }
  
  /**
   * Get resolved children only — for operations that need instances
   * 
   * Uses filter with instanceof check (NOT type narrowing — runtime check)
   */
  get resolvedChildren(): FileJs[] {
    const resolved: FileJs[] = [];
    for (const child of this.model.children) {
      // Skip IOR objects and strings (not yet resolved)
      if (typeof child === 'string') continue;
      if (child instanceof IOR) continue;
      resolved.push(child);
    }
    return resolved;
  }
}
```

### View Code (Layer 5) — Consumer Choice

```typescript
@customElement('folder-over-view')
export class FolderOverView extends UcpView<FolderModel> {
  
  render(): TemplateResult {
    return html`
      <nav class="breadcrumb">
        ${this.renderBreadcrumb()}
      </nav>
      
      <div class="children-list">
        ${this.renderChildren()}
      </div>
    `;
  }
  
  private renderChildren(): TemplateResult[] {
    const parts: TemplateResult[] = [];
    
    for (const child of this.folder.model.children) {
      // Consumer choice: handle IOR vs resolved
      if (child instanceof IOR) {
        // Still resolving — show loading state
        parts.push(html`
          <div class="loading-item">
            <span class="spinner"></span>
            Loading...
          </div>
        `);
      } else if (typeof child === 'string') {
        // IOR string not yet accessed — trigger resolution
        // (This shouldn't happen if accessed via proxy, but handle it)
        parts.push(html`<div class="loading-item">Loading...</div>`);
      } else {
        // Resolved FileJs instance
        if (child.isFolder) {
          parts.push(html`<folder-item-view .model=${child.model}></folder-item-view>`);
        } else {
          parts.push(html`<file-item-view .model=${child.model}></file-item-view>`);
        }
      }
    }
    
    return parts;
  }
}
```

### IOR is Thenable — Await Directly

**Key Feature**: The IOR class implements `then()` making it thenable:

```typescript
class IOR<T> {
  /** Store the resolution promise */
  private resolutionPromise: Promise<T> | null = null;
  
  /**
   * Make IOR thenable — can be awaited directly
   * 
   * Usage: const instance = await child;  // If child is IOR
   */
  then<TResult1 = T, TResult2 = never>(
    onfulfilled?: ((value: T) => TResult1 | PromiseLike<TResult1>) | null,
    onrejected?: ((reason: unknown) => TResult2 | PromiseLike<TResult2>) | null
  ): Promise<TResult1 | TResult2> {
    if (!this.resolutionPromise) {
      this.resolutionPromise = this.resolveAndReplace();
    }
    return this.resolutionPromise.then(onfulfilled, onrejected);
  }
}

// Usage in Layer 5:
for (const child of folder.model.children) {
  if (child instanceof IOR) {
    const resolved = await child;  // Await the IOR directly!
    renderChild(resolved);
  } else {
    renderChild(child);
  }
}
```

### The Key Insight

> "The one layer of references look ahead loading is also not specific for folder or file only. It's the **default behaviour for model**."
> 
> — TRON, 2025-12-22

**Option C embraces this**: The kernel's ISR mechanism handles ALL reference resolution universally. Components just access `model.children` and the kernel does the rest.

### Recommendation

**Option C is the correct Web4 architecture because**:

1. **Universal Mechanism**: One kernel solution handles ALL components
2. **Layer 2 Stays Pure**: No type narrowing, no async callbacks
3. **P34 Full Compliance**: IOR is everywhere, self-replacing
4. **DRY**: No orchestrators duplicating kernel functionality
5. **Already Planned**: URP PDCA has the implementation tasks

**Decision**: Implement URP (kernel ISR) first, then Option C is automatic.
