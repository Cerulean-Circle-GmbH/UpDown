## Problem Statement (TRON 2026-01-02)

> "Start migrating with focus on the ONCE runtime scenario housekeeping. Remove the stopped servers scenarios and assure the scenarios are always reflecting the real online state of once peers after the start of a new peer. Peers that shutdown just mark their scenarios as shutdown."

### Current Behavior Issues

1. **Scenarios deleted on shutdown** — Currently, stopped/shutdown peer scenarios are immediately deleted by `performHousekeeping()`. This loses historical state.

2. **Stale scenarios persist** — When a peer crashes without graceful shutdown, its scenario remains in `RUNNING` state, misleading other peers.

3. **29 sync fs calls in ServerHierarchyManager** — Uses `fs.existsSync`, `fs.unlinkSync`, `fs.lstatSync`, `fs.readdirSync` directly instead of File/Folder OOP.

### Target Behavior

1. **Mark as shutdown, don't delete** — Peers set `model.state = LifecycleState.SHUTDOWN` and save scenario
2. **Housekeeping verifies online state** — On primary start, ping each `RUNNING` scenario and mark unreachable as `SHUTDOWN`
3. **Clean old shutdown scenarios** — Delete scenarios marked `SHUTDOWN` for >24h (configurable)
4. **Use File/Folder OOP** — Migrate fs calls to use DefaultFile/DefaultFolder methods
