## Problem Statement

### Current State

```typescript
// FileSystemNode.type.ts
export type FileSystemNode = DefaultFile | DefaultFolder;  // UNION TYPE

// DefaultFile.ts
export class DefaultFile extends UcpComponent<FileModel> {}

// DefaultFolder.ts  
export class DefaultFolder extends UcpComponent<FolderModel> 
  implements Tree<FileSystemNode>, Container<FileSystemNode> {}
```

### TRON Requirement

> "Folder is a file that can add files from OOP perspective, while file is a tree leaf."

**Target State**: `Folder extends File` (proper inheritance, not union type)

---

## The Runtime Problem

### TypeScript vs Runtime

| Aspect | TypeScript | JavaScript Runtime |
|--------|------------|-------------------|
| `Folder extends File` | ✅ Works perfectly | ❌ Class already extends UcpComponent |
| `folder instanceof File` | N/A (compile-time) | ❌ Returns false if using union type |
| Interface checks | ✅ Type guards | ❌ Interfaces don't exist |

### Current Architecture Issue

```typescript
// Current:
class DefaultFile extends UcpComponent<FileModel> {}
class DefaultFolder extends UcpComponent<FolderModel> {}

// Problem: DefaultFolder CANNOT also extend DefaultFile
// JavaScript only allows single inheritance!
```

## Analysis: Options for "Folder extends File"

### Option A: Abstract File Interface (JsInterface Pattern)

```typescript
// layer3/File.ts
export abstract class File extends JsInterface {
  abstract get model(): FileModel;
  abstract get path(): string;
  abstract exists(): boolean;
  // ... common file operations
}

// layer2/DefaultFile.ts
export class DefaultFile extends UcpComponent<FileModel> implements File {
  static start(): void {
    super.start();
    File.implementationRegister(DefaultFile);
  }
}

// layer2/DefaultFolder.ts
export class DefaultFolder extends UcpComponent<FolderModel> implements File {
  static start(): void {
    super.start();
    File.implementationRegister(DefaultFolder);
  }
}
```

**Pros**: Uses existing JsInterface pattern  
**Cons**: `folder instanceof File` still returns false

### Option B: Mixin Pattern

```typescript
// FileMixin contains shared functionality
function FileMixin<T extends Constructor<UcpComponent>>(Base: T) {
  return class extends Base {
    get isFile(): boolean { return true; }
    // ... common methods
  };
}

export class DefaultFile extends FileMixin(UcpComponent<FileModel>) {}
export class DefaultFolder extends FileMixin(UcpComponent<FolderModel>) {}
```

**Pros**: Code reuse  
**Cons**: Complex, still not true inheritance, P4a violation (functions)

### Option C: File as Base, Folder Extends It

```typescript
// DefaultFile.ts
export class DefaultFile extends UcpComponent<FileModel> {
  get isFolder(): boolean { return false; }
  // File-specific methods
}

// DefaultFolder.ts
export class DefaultFolder extends DefaultFile {  // <-- True inheritance!
  get isFolder(): boolean { return true; }  // Override
  // Folder-specific methods
  
  // PROBLEM: model is FileModel, not FolderModel!
}
```

**Pros**: True OOP inheritance, `folder instanceof DefaultFile` works  
**Cons**: Model type mismatch (FolderModel vs FileModel)

### Option D: Unified FileSystemModel

```typescript
// FileSystemModel = union of File + Folder properties
export interface FileSystemModel extends Model {
  // Common
  path: string;
  name: string;
  isLink: boolean;
  
  // File-specific (optional for folders)
  content?: Reference<Blob>;
  mimetype?: string;
  size?: number;
  
  // Folder-specific (optional for files)
  children?: Collection<FileSystemNode>;
}

export class DefaultFile extends UcpComponent<FileSystemModel> {
  get isFolder(): boolean { return false; }
}

export class DefaultFolder extends DefaultFile {  // TRUE INHERITANCE
  get isFolder(): boolean { return true; }
  get children(): FileSystemNode[] { /* ... */ }
}
```

**Pros**: True inheritance, model works for both  
**Cons**: Model is less type-safe (optional fields)

## Recommendation: Option A (JsInterface Pattern) ✅ TRON APPROVED

### Why Option A

1. **Runtime Interface**: `File` exists at runtime as a JsInterface class
2. **RelatedObjects Compatible**: `image.relatedObjects(File, imageFile)` works
3. **Implementation Tracking**: `File.implementations` returns `[DefaultFile, DefaultFolder]`
4. **Web4 Pattern**: Already used for `PersistenceManager`, `View`, etc.
5. **No Model Changes**: FileModel and FolderModel remain separate

### TRON Use Case (Quoted)

> "image1.png is an image"
> "there is an Image class that renders an `<image-default-view>`"
> "the image file shall be stored in `image.relatedObjects(File, imageFile)`"
> "so File must be an interface of DefaultFile at runtime"

### Target State

**Interface Hierarchy:**
```
JsInterface
  └── File (runtime interface)

Tree<T>
  └── Container<T> extends Tree<T>
```

**Class Implementation:**
```typescript
// layer3/File.ts — EXISTS AT RUNTIME!
export abstract class File extends JsInterface {
  abstract get path(): string;
  abstract get name(): string;
  abstract get isFile(): boolean;
  abstract get isFolder(): boolean;
  abstract exists(): boolean;
  abstract delete(): Promise<void>;
}

// layer2/DefaultFile.ts — LEAF node (no children)
export class DefaultFile extends UcpComponent<FileModel> implements File {
  static start(): void {
    File.implementationRegister(DefaultFile);
  }
}

// layer2/DefaultFolder.ts — SIMPLIFIED: Container<File>, File
export class DefaultFolder extends UcpComponent<FolderModel> 
  implements Container<File>, File {
  // Container extends Tree → no need to list Tree separately
  // File → implements JsInterface for runtime registration
  static start(): void {
    File.implementationRegister(DefaultFolder);
  }
}

// Usage in Image component
image.relatedObjectRegister(File, imageFile);  // ✅ Works at runtime!
const file = image.relatedObjectLookupFirst(File);  // ✅ Returns imageFile
```

**Why This Works:**
- `Container<File> extends Tree<File>` → implementing Container satisfies Tree
- Using `File` (JsInterface) as type parameter → semantically clean
- `Collection<File>` can hold both DefaultFile and DefaultFolder
