## 🔴 PROBLEM STATEMENT

### Current Architecture (Problematic)

```
┌─────────────────────────────────────────────────────────┐
│                    CURRENT (COUPLED)                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  LitElement  ←── UcpView ←── DefaultItemView            │
│      ↑              ↑              ↑                    │
│   (Lit lib)    (Web4 class)   (Lit component)          │
│                                                         │
│  ❌ UcpView EXTENDS LitElement                          │
│  ❌ View/ItemView are interfaces (not JsInterface)      │
│  ❌ Framework lock-in (Lit required for all views)      │
│  ❌ Cannot implement View as JsInterface (single inherit)│
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Problems

1. **Framework Lock-in**: UcpView extends LitElement → all views tied to Lit
2. **JsInterface Impossible**: TypeScript single inheritance blocks View as JsInterface
3. **Testing Issues**: Lit decorators fail in tsx environment
4. **Portability Lost**: Can't use React, Vue, or vanilla JS for views
5. **Web4 Principle Violation**: Views should be framework-independent

## 🟢 TARGET ARCHITECTURE

### Delegation Pattern (Decoupled)

```

│                    TARGET (DECOUPLED)                    │
```

## 🔴 CORE FUNDAMENTALS

### 0. Rendering: Let the Framework Handle It!

**Keep it simple:**
- **Lit views**: Lit's `render()` returns `TemplateResult` → Lit manages DOM
- **Vanilla views** (DEFERRED): `render()` returns `void` → view manages shadow DOM directly
- **Abstract views**: NO render() method! Just provide data via `scenario.model`

```typescript
// ═══════════════════════════════════════════════════════════════
// Lit Adapter - Lit handles the DOM
// ═══════════════════════════════════════════════════════════════
@customElement('lit-once-peer-item-view')
class LitOncePeerItemView extends LitElement {
  @property({ attribute: false }) scenario: Scenario<OncePeerItemViewModel>;
  
  // Lit's render() returns TemplateResult - Lit manages DOM
  render(): TemplateResult {
    const m = this.scenario.model;
    return html`
      <div class="item-view">
        <i class="fa ${m.icon}"></i>
        <span>${m.name}</span>
      </div>
    `;
  }
}

// ═══════════════════════════════════════════════════════════════
// Vanilla Adapter (DEFERRED) - View manages DOM directly
// ═══════════════════════════════════════════════════════════════
@customElement('web4-once-peer-item-view')
class DefaultOncePeerItemView extends HTMLElement {
  private scenario: Scenario<OncePeerItemViewModel>;
  
  // Vanilla render() returns void - we manage shadow DOM
  render(): void {
    const m = this.scenario.model;
    // Template engine fills shadow DOM from external template
    this.templateEngine.render(this.shadowRoot, m);
  }
}
```

**The Abstract layer just provides data access - NO render() abstraction!**

### 1. ALL Views Work on `scenario.model` - ALWAYS AND ONLY

```typescript
// ✅ CORRECT: Views ONLY read/write scenario.model
abstract class AbstractItemView<T extends ItemViewModel> extends AbstractView<T> {
  
  // The scenario containing the model
  protected scenario: Reference<Scenario<T>> = null;
  
  // Model is ALWAYS accessed via scenario
  get model(): T {
    return this.scenario!.model;
  }
  
  // ItemViewModel is a GETTER INTERFACE - not storage!
  get name(): string { return this.model.name; }
  set name(v: string) { this.model.name = v; }  // Triggers ModelProxy
}
```

### 2. `ItemViewModel` is a GETTER INTERFACE (Not Storage)

```typescript
// ItemViewModel defines the contract for model access
// It does NOT store data - scenario.model does!
interface ItemViewModel extends Model {
  name: string;
  description: string;
  icon: string;
  badge: Reference<string | number>;
}

// The actual data lives in scenario.model
// Views provide getters/setters that delegate to scenario.model
```

### 3. ModelProxy Enables Rerender & Peer Sync

```typescript
// ModelProxy wraps scenario.model for:
// 1. Automatic view re-render on change
// 2. Scenario sync to peers via WebSocket

class UcpController<T> {
  private modelProxy: T;  // Proxy wrapping scenario.model
  
  init(scenario: Scenario<T>): this {
    // Create proxy that intercepts all property changes
    this.modelProxy = this.proxyCreate(scenario.model);
    return this;
  }
  
  private proxyCreate(model: T): T {
    return new Proxy(model, {
      set: (target, prop, value) => {
        target[prop] = value;
        this.viewsUpdateAll();      // Re-render views
        this.scenarioBroadcast();   // Sync to peers
        return true;
      }
    });
  }
}
```

### Class Relationships

```typescript
// ═══════════════════════════════════════════════════════════════
// LAYER 3: JsInterface hierarchy (Abstract classes)
// ALL views work on scenario.model - ALWAYS AND ONLY
// ═══════════════════════════════════════════════════════════════

abstract class AbstractView<TModel extends Model> extends JsInterface {
  // Scenario is the source of truth (NOT a local field!)
  protected scenario: Reference<Scenario<TModel>> = null;
  protected childViews: AbstractView<unknown>[] = [];
  
  // Model getter delegates to scenario.model
  get model(): TModel {
    return this.scenario!.model;
  }
  
  // Refresh triggers re-render (framework handles how)
  abstract refresh(): void;
  
  // ❌ NO abstract render() here!
  // Lit views: Lit's render() returns TemplateResult
  // Vanilla views: render() returns void, manages DOM directly
  // Each framework handles rendering its own way
}

abstract class AbstractItemView<T extends ItemViewModel> extends AbstractView<T> {
  // ItemViewModel is a GETTER INTERFACE - delegates to scenario.model
  get name(): string { return this.model.name; }
  set name(v: string) { this.model.name = v; }  // Triggers ModelProxy
  
  get description(): string { return this.model.description; }
  set description(v: string) { this.model.description = v; }
  
  get icon(): string { return this.model.icon; }
  set icon(v: string) { this.model.icon = v; }
  
  get badge(): Reference<string | number> { return this.model.badge; }
  set badge(v: Reference<string | number>) { this.model.badge = v; }
}

// ═══════════════════════════════════════════════════════════════
// LAYER 5: Lit Adapter - Lit handles DOM
// Custom element: <lit-item-view>
// ═══════════════════════════════════════════════════════════════

@customElement('lit-item-view')  // ← 'lit-' prefix!
class LitItemView extends LitElement {
  // Scenario is passed in - Lit view reads scenario.model
  @property({ attribute: false }) scenario: Scenario<ItemViewModel>;
  
  // Lit's render() returns TemplateResult - Lit manages DOM
  render(): TemplateResult {
    if (!this.scenario) return html``;
    
    const m = this.scenario.model;  // Read from scenario.model
    
    // Lit renders (inline template - accepted limitation for Lit)
    return html`
      <div class="item-view">
        <i class="fa ${m.icon}"></i>
        <span class="name">${m.name}</span>
        <span class="description">${m.description}</span>
        ${m.badge ? html`<span class="badge">${m.badge}</span>` : nothing}
      </div>
    `;
  }
}

// ═══════════════════════════════════════════════════════════════
// LAYER 5: Concrete Web4 Implementation (business logic)
// ═══════════════════════════════════════════════════════════════

class OncePeerItemView extends AbstractItemView<OncePeerItemViewModel> {
  
  get name(): string { return this.modelField?.name ?? 'ONCE Peer'; }
  set name(v: string) { if (this.modelField) this.modelField.name = v; }
  
  get description(): string { return this.modelField?.connectionStatus ?? ''; }
  set description(v: string) { /* read-only for peer */ }
  
  get icon(): string { 
    return this.modelField?.isConnected ? 'fa-server' : 'fa-server-slash'; 
  }
  set icon(v: string) { /* computed from state */ }
  
  get badge(): Reference<string | number> { 
    return this.modelField?.clientCount ?? null; 
  }
  set badge(v: Reference<string | number>) { /* computed */ }
  
  refresh(): void {
    for (const child of this.childViews) {
      child.refresh();
    }
  }
  
  // ❌ NO render() method in Web4 view!
  // Rendering is handled by the framework adapter (Lit/Vanilla)
  // This class just provides data via scenario.model and business logic

    // Navigate to peer details
  }
  
  handlePeerMenu(): void {

    // Show action menu
  }
}
```

```
┌───▼───────┐    ┌────────▼────┴───┐
│LitItemView│    │DefaultItemView  │     ← PARALLEL ALTERNATIVES
│(LitElement)│   │(Vanilla/ONCE)   │       (not inheritance!)
│           │    │                 │
│ @click →  │    │ onclick →       │
│ Lit events│    │ DOM events      │
└───────────┘    └─────────────────┘
     ↓                  ↓
  Browser            Browser
  (with Lit)         (vanilla JS)
```

### Naming Pattern

| Layer | Pattern | Custom Element | Description |
|-------|---------|----------------|-------------|
| **Abstract** | `Abstract{X}View` | — | Web4 JsInterface class (business logic) |
| **Concrete** | `Once{X}View` | — | ONCE-specific Web4 implementation |
| **Lit Adapter** | `Lit{X}View` | `<lit-{x}-view>` | LitElement wrapper (parallel) |
| **Vanilla Adapter** | `Default{X}View` | `<web4-{x}-view>` | DOM wrapper, ONCE template (DEFERRED) |

**Custom Element Examples:**
```typescript
// Lit adapter for OncePeerItemView
@customElement('lit-once-peer-item-view')
class LitOncePeerItemView extends LitItemView { ... }

// Vanilla adapter (DEFERRED) - would be:
@customElement('web4-once-peer-item-view')  
class DefaultOncePeerItemView extends HTMLElement { ... }
```

**Focus:** Lit views only for now. Accept Lit's inline template limitation.

### Key Insight: Parallel Adapters

```
AbstractItemView (Web4 business logic)
       │
       ├──── LitItemView (Lit rendering) ──────┐
       │                                       │ PARALLEL
       └──── DefaultItemView (Vanilla DOM) ────┘ ALTERNATIVES
```

Both adapters:
- Receive the same `AbstractItemView` delegate
- Render to DOM using different frameworks
- Handle events differently (Lit vs vanilla)
- Are interchangeable in the browser

## 📝 DESIGN DECISIONS

### D1: ViewRenderData vs Template Strings

**Question**: What should UcpView.render() return?

**Options**:
1. **Data object** (`ViewRenderData`) - framework-agnostic
2. **Template string** - portable but limited
3. **Virtual DOM** - heavy, reinvents React

**Decision**: **Data object (ViewRenderData)**

```typescript
interface ViewRenderData {
  type: string;
  attributes: Record<string, unknown>;
  children: ViewRenderData[];
  events?: Record<string, string>; // event name → method name
}
```

Rationale:
- Framework-agnostic
- Adapters convert to Lit html``, React JSX, etc.
- Testable without browser
- Serializable (can send over WebSocket for SSR)

### D2: CSS Loading Location

**Question**: Where should CSS loading happen?

**Options**:
1. **UcpView** - current location
2. **LitViewAdapter** - framework-specific
3. **CSSLoader singleton** - separate concern

**Decision**: **LitViewAdapter** (framework-specific)

Rationale:
- UcpView becomes framework-independent
- Lit uses `adoptedStyleSheets`, React uses CSS modules, etc.
- Each adapter handles its framework's CSS mechanism

### D3: Event Handling

**Question**: How do Web4 views handle events?

**Decision**: **ViewRenderData.events** + Adapter binding

```typescript
// Web4 view defines events
class DefaultItemView extends ItemView {
  templateRender(): ViewRenderData {
    return {
      type: 'item',
      events: {
        'click': 'handleClick',     // method name on this view
        'contextmenu': 'handleMenu'
      },
      // ...
    };
  }
  
  handleClick(): void {
    // Action invocation
  }
}

// Lit adapter binds them
class LitItemViewAdapter extends LitElement {
  render() {
    const data = this.delegate.render();
    return html`
      <div @click=${() => this.delegate.handleClick()}>
        ...
      </div>
    `;
  }
}
```
