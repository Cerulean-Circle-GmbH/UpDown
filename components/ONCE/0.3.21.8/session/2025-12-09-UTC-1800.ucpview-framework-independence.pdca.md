### Problems

1. **Framework Lock-in**: UcpView extends LitElement ‚Üí all views tied to Lit
2. **JsInterface Impossible**: TypeScript single inheritance blocks View as JsInterface
3. **Testing Issues**: Lit decorators fail in tsx environment
4. **Portability Lost**: Can't use React, Vue, or vanilla JS for views
5. **Web4 Principle Violation**: Views should be framework-independent

### 1. ALL Views Work on `scenario.model` - ALWAYS AND ONLY

```typescript
// ‚úÖ CORRECT: Views ONLY read/write scenario.model
abstract class AbstractItemView<T extends ItemViewModel> extends AbstractView<T> {
  
  // The scenario containing the model
  protected scenario: Reference<Scenario<T>> = null;
  
  // Model is ALWAYS accessed via scenario
  get model(): T {
    return this.scenario!.model;
  }
  
  // ItemViewModel is a GETTER INTERFACE - not storage!
  get name(): string { return this.model.name; }
  set name(v: string) { this.model.name = v; }  // Triggers ModelProxy
}
```

### 3. ModelProxy Enables Rerender & Peer Sync

```typescript
// ModelProxy wraps scenario.model for:
// 1. Automatic view re-render on change
// 2. Scenario sync to peers via WebSocket

class UcpController<T> {
  private modelProxy: T;  // Proxy wrapping scenario.model
  
  init(scenario: Scenario<T>): this {
    // Create proxy that intercepts all property changes
    this.modelProxy = this.proxyCreate(scenario.model);
    return this;
  }
  
  private proxyCreate(model: T): T {
    return new Proxy(model, {
      set: (target, prop, value) => {
        target[prop] = value;
        this.viewsUpdateAll();      // Re-render views
        this.scenarioBroadcast();   // Sync to peers
        return true;
      }
    });
  }
}
```

abstract class AbstractView<TModel extends Model> extends JsInterface {
  // Scenario is the source of truth (NOT a local field!)
  protected scenario: Reference<Scenario<TModel>> = null;
  protected childViews: AbstractView<unknown>[] = [];
  
  // Model getter delegates to scenario.model
  get model(): TModel {
    return this.scenario!.model;
  }
  
  // Refresh triggers re-render (framework handles how)
  abstract refresh(): void;
  
  // ‚ĚĆ NO abstract render() here!
  // Lit views: Lit's render() returns TemplateResult
  // Vanilla views: render() returns void, manages DOM directly
  // Each framework handles rendering its own way
}

abstract class AbstractItemView<T extends ItemViewModel> extends AbstractView<T> {
  // ItemViewModel is a GETTER INTERFACE - delegates to scenario.model
  get name(): string { return this.model.name; }

  get description(): string { return this.model.description; }

## ūüďĚ DESIGN DECISIONS

### D1: ViewRenderData vs Template Strings

**Question**: What should UcpView.render() return?

**Options**:
1. **Data object** (`ViewRenderData`) - framework-agnostic
2. **Template string** - portable but limited
3. **Virtual DOM** - heavy, reinvents React

**Decision**: **Data object (ViewRenderData)**

```typescript
interface ViewRenderData {
  type: string;
  attributes: Record<string, unknown>;
  children: ViewRenderData[];
  events?: Record<string, string>; // event name ‚Üí method name
}
```

Rationale:
- Framework-agnostic
- Adapters convert to Lit html``, React JSX, etc.
- Testable without browser
- Serializable (can send over WebSocket for SSR)

### D2: CSS Loading Location

**Question**: Where should CSS loading happen?

**Options**:
1. **UcpView** - current location
2. **LitViewAdapter** - framework-specific
3. **CSSLoader singleton** - separate concern

**Decision**: **LitViewAdapter** (framework-specific)

Rationale:
- UcpView becomes framework-independent
- Lit uses `adoptedStyleSheets`, React uses CSS modules, etc.
- Each adapter handles its framework's CSS mechanism

### D3: Event Handling

**Question**: How do Web4 views handle events?

**Decision**: **ViewRenderData.events** + Adapter binding

```typescript
// Web4 view defines events
class DefaultItemView extends ItemView {
  templateRender(): ViewRenderData {
    return {
      type: 'item',
      events: {
        'click': 'handleClick',     // method name on this view
        'contextmenu': 'handleMenu'
      },
      // ...
    };
  }
  
  handleClick(): void {
    // Action invocation
  }
}

// Lit adapter binds them
class LitItemViewAdapter extends LitElement {
  render() {
    const data = this.delegate.render();
    return html`
      <div @click=${() => this.delegate.handleClick()}>
        ...
      </div>
    `;
  }
}
```
