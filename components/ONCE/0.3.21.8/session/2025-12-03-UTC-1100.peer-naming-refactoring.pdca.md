## 🎯 PLAN

### Problem Statement

1. **Naming Convention Violation**: Current methods use "server/client" terminology which is outdated:
   - `startServer` → should be `peerStart`
   - `stopServer` → should be `peerStop`
   - `shutdownAll` → should be `peerStopAll`
   - `startClientServer` → should be removed (replaced by auto-port `peerStart`)

2. **Regression Bug**: `once startServer` in 0.3.21.8 throws error when port 42777 is occupied, instead of auto-fallback to 8080+ like 0.3.21.5 did.

3. **Missing Tests**: No Tootsie tests verify the peer start/stop lifecycle with auto-port fallback.

### Expected Behavior (from 0.3.21.5)

```
# First peer (becomes primary on 42777)
$ once startServer
🟢 Started as PRIMARY SERVER on port 42777

# Second peer (auto-finds 8080)
$ once startServer  
🔵 Started as CLIENT SERVER on port 8080

# Third peer (auto-finds 8081)
$ once startServer
🔵 Started as CLIENT SERVER on port 8081
```

### Web4 Naming Convention (Principle 16)

| Old Name | New Name | Reason |
|----------|----------|--------|
| `startServer` | `peerStart` | Peer-to-peer, no hierarchy |
| `stopServer` | `peerStop` | Consistent naming |
| `shutdownAll` | `peerStopAll` | `{noun}{verb}` pattern |
| `startClientServer` | REMOVED | `peerStart` auto-detects |
| `getServers` | `peersDiscover` | Discovery pattern |
