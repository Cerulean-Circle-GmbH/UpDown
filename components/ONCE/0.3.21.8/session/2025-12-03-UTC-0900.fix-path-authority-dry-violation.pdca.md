## **🔍 PROBLEM ANALYSIS**

### **The Violation**

`ONCETestCase.ts` (lines 36-59) reimplements path calculations:

```typescript
// ❌ DRY VIOLATION - Reimplements DefaultCLI's Path Authority
protected getONCEVersion(): string {
  const currentFilePath = fileURLToPath(import.meta.url);
  const testDir = path.dirname(currentFilePath);
  const testRoot = path.dirname(testDir);
  const versionDir = path.dirname(testRoot);
  return path.basename(versionDir);
}

protected getComponentRoot(): string {
  const currentFilePath = fileURLToPath(import.meta.url);
  // ... more path navigation
}

protected getTestDataDir(): string {
  return path.join(this.getComponentRoot(), 'test', 'data');
}
```

### **Path Authority Principle (Web4)**

From `path-authority-architecture.test.ts`:
> - DefaultCLI calculates ALL paths (Path Authority)
> - Component uses ONLY model state (NO path calculations)
> - Model-driven program flow (behavior determined by model state)
> - Environment-agnostic (NO process.cwd() or env vars in Component)

**Who Has Path Authority:**
- `DefaultCLI.calculateProjectRootInternal()` - SINGLE SOURCE OF TRUTH
- Sets: `projectRoot`, `componentRoot`, `testDataDir`, `isTestIsolation`
- All components MUST use `this.model.*` not calculate paths

### **Test Isolation Definition**

From `DefaultCLI.ts` (lines 149-164):
```typescript
// Test isolation: When cwd contains '/test/data'
// projectRoot = path up to and including test/data
// Example: .../ONCE/0.3.21.8/test/data becomes project root
```

### **Root Cause**

**DEEPER ISSUE:** Path info is recalculated everywhere instead of being available once.

The problem is NOT that TestScenario lacks fields. Adding path fields to every scenario would **pollute models** with redundant data.

**The Real Problem:**
1. `DefaultCLI` calculates paths (correct - single source of truth)
2. But `Web4TSComponent` doesn't expose these paths for generated components
3. So every component/test reimplements path calculation
4. This causes DRY violations AND model pollution

**What Should Happen:**
1. `DefaultCLI` calculates paths ONCE at startup
2. `Web4TSComponent` stores and exposes these paths globally
3. ALL components (including tests) access via `Web4TSComponent.getProjectRoot()` etc.
4. NO component recalculates paths - NO scenario pollution

## **📋 PLAN**

### **Goal**
Fix Path Authority architecture - single calculation, global access via Web4TSComponent

### **Architectural Fix**

```
┌─────────────────────────────────────────────────────────────┐
│                    DefaultCLI                                │
│  calculateProjectRootInternal() → SINGLE SOURCE OF TRUTH     │
│  Sets: projectRoot, componentRoot, testDataDir, isTestIsolation │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                  Web4TSComponent                             │
│  Stores CLI's calculated paths                               │
│  Exposes: getProjectRoot(), getComponentRoot(), etc.         │
│  Available to ALL components via delegation                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│           Generated Components / Tests                       │
│  Access via: this.web4ts.getProjectRoot()                   │
│  NO recalculation - NO scenario pollution                   │
└─────────────────────────────────────────────────────────────┘
```

### **Approach: Web4TSComponent exposes Path Authority**

1. Web4TSComponent already has `model.projectRoot`, `model.componentRoot`, etc.
2. Add accessor methods: `getProjectRoot()`, `getComponentRoot()`, `getTestDataDir()`, `isTestIsolation()`
3. Generated components access via delegation: `this.web4ts.getProjectRoot()`
4. Tests access via the component they're testing OR via Tootsie's context

## **📝 ADDITIONAL FIX: StaticFileRoute for MIME Types**

### **Problem**
- IORRoute was matching `/dist/ts/layer1/ONCE.js` as IOR pattern (4 path segments)
- Browser received `application/json` instead of `application/javascript`
- Module loading failed with MIME type error

### **Solution**
Created `StaticFileRoute.ts` with:
1. **Priority 5** (higher than IORRoute's 10) - checked first
2. **Preconfigured MIME types** - no default json for everything
3. **Extension matching** - only matches paths with file extensions (.js, .css, .html, etc.)

### **Files Changed**
- `src/ts/layer2/StaticFileRoute.ts` (NEW)
- `src/ts/layer2/ServerHierarchyManager.ts` (import + registration)
