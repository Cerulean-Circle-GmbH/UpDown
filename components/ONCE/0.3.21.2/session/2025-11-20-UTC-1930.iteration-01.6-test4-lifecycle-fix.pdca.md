## **🌐 APPENDIX: ONCE P2P Server Network Architecture**

### **Overview**

ONCE implements a **hierarchical P2P (Peer-to-Peer) server network** where servers can run on:
- **Single host** (localhost) - Multiple servers on different ports
- **Multiple hosts** - Distributed across network (LAN/WAN)
- **Hybrid mode** - Mix of local and remote servers

This architecture enables scalability, fault tolerance, and distributed computing without requiring centralized infrastructure.

### **Server Roles**

#### **1. Primary Server (Port 42777)**

**Responsibilities**:
- **Server Registry**: Maintains list of all connected client servers
- **Message Relay**: Routes messages between servers and browsers
- **Housekeeping**: Cleans up stale/shutdown scenarios on startup
- **Discovery**: Responds to server discovery requests
- **Coordination**: Assigns tasks, manages state synchronization

**Discovery**:
- Only ONE primary per host/domain
- Claims port 42777 (or fails if already taken)
- Runs housekeeping at startup to discover existing servers

**Multi-Host Behavior**:
- Each host can have its own primary on port 42777
- Remote primaries can register as clients to another primary
- Creates hierarchical network: Primary-of-Primaries pattern

#### **2. Client Server (Ports 8080+)**

**Responsibilities**:
- **Application Logic**: Serves web pages, handles requests
- **Registration**: Registers with primary server on startup
- **Heartbeat**: Maintains WebSocket connection to primary
- **Unregistration**: Gracefully unregisters on shutdown

**Discovery**:
- Tries port 42777 first (primary)
- If taken, gets next available port (8080, 8081, ...)
- Connects to primary via WebSocket for registration

**Multi-Host Behavior**:
- Can register to primary on same host OR remote host
- Connection via: `ws://{primaryHost}:42777`
- Uses hostname resolution for cross-host discovery

### **Registration Flow (Multi-Host)**

#### **Scenario 1: Client on Same Host**

```
1. Client starts on Host A:8080
2. Client detects primary at localhost:42777
3. Client opens WebSocket to ws://localhost:42777
4. Client sends:
   {
     type: 'server-registration',
     serverModel: {
       uuid: '4588cf61',
       port: 8080,
       hostname: 'McDonges-3',
       domain: 'box.fritz',
       capabilities: [{capability: 'httpPort', port: 8080}]
     }
   }
5. Primary adds to serverRegistry
6. Primary broadcasts 'server-registered' to all browsers
7. Client receives confirmation
```

#### **Scenario 2: Primary on Remote Host Registers as Client**

```
1. Primary on Host B starts (claims port 42777 locally)
2. Primary on Host B discovers Host A primary via DNS/config
3. Host B primary opens WebSocket to ws://McDonges-3.fritz.box:42777
4. Host B primary sends registration AS A CLIENT:
   {

       port: 42777,  ← Claims to be on port 42777

       domain: 'box.fritz',
       isPrimaryServer: true,  ← Declares itself as primary
       capabilities: [{capability: 'httpPort', port: 42777}]
     }
   }
5. Host A primary adds Host B primary to its registry
6. Now Host A can relay messages to Host B's network
```

#### **Scenario 3: Client on Remote Host**

### **Discovery Mechanisms**

#### **Implemented**

1. **Port-Based Discovery** (Single Host)
   - Check localhost:42777 (primary)
   - If available → become primary
   - If taken → become client on next port

2. **Scenario-Based Discovery** (Housekeeping)
   - Read `scenarios/{domain}/{hostname}/ONCE/{version}/*.scenario.json`
   - Parse state, port, UUID
   - Health-check and register

#### **Not Implemented**

1. **DNS-Based Discovery**
   - mDNS/Bonjour for LAN discovery
   - DNS SRV records for WAN discovery
   - Format: `_once._tcp.{domain}`

2. **Config-Based Discovery**
   - `once.config.json` with known primaries
   - Environment variable: `ONCE_PRIMARY_HOSTS`

3. **Gossip Protocol**
   - Servers share known peers
   - Eventual consistency across network
