## **📋 CMM3 PDCA Structure**

### **P - PLAN**

#### **1. Problem Statement**
Implement the TRUE Radical OOP Web4 architecture refactoring of ONCE v0.3.21.2 through systematic, iterative PDCA cycles. Each iteration must be fully tested, compilable, and stable before proceeding to the next.

#### **2. Current Situation (AS-IS)**
- **Current State**: ONCE v0.3.21.2 has functional architecture with separation of concerns violations
- **Architecture Issues**:
  - Multiple types per file (violates "one type per file")
  - Direct system imports in Layer 2 (violates layer separation)
  - Functional event handlers (violates TRUE Radical OOP)
  - Configuration constants instead of scenarios
  - Switch/case for dispatch instead of convention-over-configuration
  - Inline HTML and CommonJS `__dirname` usage
  - Promise callbacks instead of async/await
  - String literals instead of enums
  - Mixed concerns in interfaces (statistics, etc.)

#### **3. Target Situation (TO-BE)**
- **Target State**: ONCE v0.3.21.2 with TRUE Radical OOP Web4 architecture
- **Architecture Goals**:
  - ✅ All Web4 principles upheld (see checklist)
  - ✅ Complete 5-layer separation
  - ✅ Observer Pattern for lifecycle events
  - ✅ One type per file
  - ✅ Scenario-based configuration
  - ✅ Convention-over-configuration dispatch
  - ✅ Full test coverage (Vitest + Playwright)
  - ✅ Zero regressions

**Why is iterative approach needed?**
- ✅ **Complexity**: Refactoring touches 100+ files
- ✅ **Risk**: Large-bang refactoring causes regressions
- ✅ **Testing**: Need to validate each step
- ✅ **Stability**: Must maintain working system throughout
- ✅ **Learning**: Each iteration provides feedback for next
