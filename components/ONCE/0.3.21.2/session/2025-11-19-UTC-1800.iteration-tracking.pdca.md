**🎯 CRITICAL ARCHITECTURAL ISSUES IDENTIFIED BY TRON**:

ONCE component has **6 fundamental architectural problems** that violate Web4 principles:

**Problem Analysis**:
1. ❌ `ONCEDomain.interface.ts` - Wrong name, should be `ONCE.interface.ts` (core kernel interface)

6. ❌ **NEW**: `ONCEScenarioMessage` and `ONCEMessageTracker` are functional/procedural (violate protocol-less Web4)

**Root Cause**:
ONCE was not designed as a **component itself** - it was designed as a **kernel/runtime**. This violated the **fractal Web4 principle**: "Components are made of components, all the way down."

**Three NEW Web4 Principles Discovered**:

**1. Global ONCE Singleton Pattern**:
```typescript

// Node.js
global.ONCE

// Workers/PWA
self.global.ONCE
```

**2. Protocol-Less Communication (FUNDAMENTAL PARADIGM)**:
- ❌ NO messages, NO protocols, NO message tracking
- ✅ Objects replicate via scenarios (hibernation/rehydration)
- ✅ Instances created with SAME UUID (true clones)
- ✅ Objects have METHODS, not message handlers
- ✅ **Primary Server (42777) is the global P2P registry with IOR failover profiles**
- **Impact**: Delete `ONCEScenarioMessage`, `ONCEMessageTracker`, redesign P2P

**3. IOR-based Method Invocation with Primary Server Failover**:
- IORs are **extended URLs** with protocols and query parameters
- Endpoints → Method calls (like CLI command → method)
- Query parameters → Scenario properties or method parameters
- **Primary Server IOR**: `ior:https://primary.once.network:42777,backup1:42778,backup2:42779/ONCE/0.3.21.2/registry-uuid/listServers`
- **Primary Server = Global P2P Registry** with automatic failover
- Loaders register for IOR protocols, implement Loader interface
- **Impact**: Action endpoints become method invocation layer
