# Component Delegation Fix - CLI ↔ Component Path Authority

**Date**: 2025-11-21  
**Issue**: `Component not initialized` error when starting server  
**Root Cause**: Broken delegation chain between CLI and Component  

---

## The Problem

When manually starting a server with `./once startServer`, the system crashed with:

```
Error: Component not initialized
    at ServerHierarchyManager.getDemoHubHTML
```

This revealed that `ServerHierarchyManager` was trying to access `this.component.model.componentRoot` but it was `undefined`.

---

## Root Cause Analysis

### The Delegation Chain

The system has three initialization paths:

1. **Production CLI Path**: `./once` → `ONCECLI` → `DefaultONCE`
2. **Direct Instantiation**: `new DefaultONCE()` (used in old white-box tests)
3. **Test CLI Path**: `./once test` → `ONCECLI` → `DefaultONCE`

### What Was Broken

In `DefaultONCE` constructor:
```typescript
// ❌ OLD CODE - Calculated paths but set them late
constructor() {
  // ... version calculation ...
  this.model = {
    component: 'ONCE',
    version: versionString,
    // ❌ NO componentRoot, projectRoot here!
  };
  
  // Set backlinks to managers
  this.serverHierarchyManager.component = this; // ✅ Backlink exists
  this.scenarioManager.component = this; // ✅ Backlink exists
}
```

Paths were only calculated later in `getWeb4TSComponent()` (lazy-loaded), so when `ServerHierarchyManager` tried to access them immediately, they were `undefined`.

In `ServerHierarchyManager`:
```typescript
// ❌ OLD CODE - Assumed component.model.componentRoot exists
private getDemoHubHTML(): string {
    const componentRoot = this.component?.model.componentRoot;
    if (!componentRoot) {
        throw new Error('Component not initialized'); // 💥 CRASH!
    }
    // ...
}
```

## The Fix (3 Parts)

### Part 1: Calculate Paths in DefaultONCE Constructor

```typescript
// ✅ NEW CODE in DefaultONCE.ts constructor
constructor() {
  // Calculate version from directory
  const currentFileUrl = new URL(import.meta.url);
  const currentVersionDir = path.resolve(path.dirname(currentFileUrl.pathname), '..', '..', '..');
  const versionString = isVersionDir ? componentDirName : '0.0.0.0';
  
  // ✅ CRITICAL: Calculate paths immediately
  const componentRoot = currentVersionDir;
  const componentsDir = path.dirname(path.dirname(componentRoot));
  const projectRoot = path.dirname(componentsDir);
  const isTestIsolation = projectRoot.includes('/test/data');
  
  this.model = {
    component: 'ONCE',
    version: versionString,
    // ✅ Set path authority immediately
    componentRoot,
    projectRoot,
    targetDirectory: projectRoot,
    targetComponentRoot: componentRoot,
    isTestIsolation
  };
  
  // Managers now have access to paths via backlink
  this.serverHierarchyManager.component = this;
  this.scenarioManager.component = this;
}
```

### Part 2: Add Fallback in ServerHierarchyManager

```typescript
// ✅ NEW CODE - Fallback for when component link is not set
private getDemoHubHTML(): string {
    // Try to use component path if available (DRY principle), else fall back to direct calculation
    const componentRoot = this.component?.model.componentRoot;
    const fullPath = componentRoot
        ? path.join(componentRoot, 'src/view/html/demo-hub.html')
        : path.join(path.dirname(fileURLToPath(import.meta.url)), '../../../src/view/html/demo-hub.html');
    const html = fs.readFileSync(fullPath, 'utf-8');
    // ...
}
```

This ensures backward compatibility even if `component` is not set.

### Part 3: CLI Overrides projectRoot (Web4 Pattern)

```typescript
// ✅ ONCECLI.ts constructor - matches Web4TSComponentCLI pattern
constructor() {
  super();
  this.init();  // ← DefaultCLI calculates ALL paths in model
  
  // Component calculates its own componentRoot from import.meta.url
  this.component = new DefaultONCE();
  
  // ✅ Direct model assignment (Web4 pattern - NO re-init!)
  // Component already set: componentRoot, version (from its own location)
  // CLI provides: projectRoot, targetDirectory (path context for operations)
  this.component.model.projectRoot = this.model.projectRoot;
  this.component.model.targetDirectory = this.model.projectRoot;
  
  this.discoverMethods();
}
```

**Key Insight**: The CLI does **not** create a backlink to itself in the component. It only overrides `projectRoot` and `targetDirectory`.

## How Web4TSComponent Does It

Looking at `Web4TSComponentCLI.ts`:

```typescript
constructor() {
  super();
  this.init();  // ← DefaultCLI calculates ALL paths in model
  
  // ✅ Create component with defaults
  // Component discovers its own componentRoot from import.meta.url
  this.component = new DefaultWeb4TSComponent().init();
  
  // ✅ Direct model assignment (OOP - NO re-init!)
  // Component already set: IOR, owner, version, componentRoot (from its own location)
  // CLI provides: projectRoot, targetDirectory (path context for operations)
  this.component.model.projectRoot = this.model.projectRoot;
  this.component.model.targetDirectory = this.model.projectRoot;
  
  this.discoverMethods();
}
```

**Pattern**:
1. ❌ NO backlink from component to CLI
2. ✅ Component calculates its own `componentRoot` from `import.meta.url`
3. ✅ CLI calculates `projectRoot` via git/directory traversal
4. ✅ CLI overwrites component's `projectRoot` with its own
5. ✅ One-way delegation: CLI → Component (path override)

## Why This Design?

### Path Calculation Responsibilities

| Path | Calculated By | Why? |
|------|---------------|------|
| `projectRoot` | CLI | Needs git/directory traversal, environment awareness |
| `componentRoot` | Component | Knows its own location via `import.meta.url` |
| `version` | Component | Derived from directory name it lives in |
| `testDataDir` | CLI | Project-level concept, not component-level |

### Why No Backlink CLI ← Component?

1. **Separation of Concerns**: Components don't need to know about CLI
2. **Reusability**: Components can be used without CLI (direct instantiation)
3. **DRY**: CLI calculates `projectRoot` once, components inherit it
4. **Clean Architecture**: One-way dependency (CLI → Component)

## All Three Initialization Paths Now Work

### 1. Production CLI (`./once startServer`)
```
./once → ONCECLI.constructor() 
       → this.init() (calculates projectRoot)
       → new DefaultONCE() (calculates componentRoot)
       → override: component.model.projectRoot = cli.model.projectRoot
       → ✅ All paths available immediately
```

### 2. Direct Instantiation (`new DefaultONCE()`)
```
new DefaultONCE() → constructor() calculates componentRoot AND projectRoot
                 → ✅ All paths available immediately
                 → Fallback: ServerHierarchyManager uses fileURLToPath if needed
```

### 3. Test CLI (`./once test`)
```
Same as Production CLI path
✅ All paths available immediately
```

## Lessons Learned

1. **Path Authority**: Calculate paths ONCE, as early as possible
2. **DRY Principle**: Don't recalculate paths - inherit from authority
3. **Fallback Pattern**: Always have a fallback when accessing optional references
4. **Follow Patterns**: Web4TSComponentCLI is the gold standard - match it exactly
5. **Test All Paths**: CLI, direct instantiation, and test CLI must all work
