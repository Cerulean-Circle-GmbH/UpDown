### **Problem Statement**

**Current Issue**: The browser client and demo hub **hardcode `localhost`** in multiple places, ignoring the server's actual hostname and network identity from scenarios.

**Evidence from `curl http://mcdonges-3.fritz.box:42777/demo`**:
- Demo hub HTML is identical whether accessed via `localhost` or `mcdonges-3.fritz.box`
- All URLs hardcode `localhost` instead of using `window.location.hostname`
- Server data includes `host`, `hostname`, `domain`, `ip` but demo hub hardcodes localhost anyway

### **Current Hardcoding Violations**

From `demo-hub.html`, found **8 critical hardcoded localhost instances**:

### **Root Cause Analysis**

**Why does hardcoding localhost matter?**

1. **Remote Access Broken**: When accessing `http://mcdonges-3.fritz.box:42777/demo`, the page loads but:
   - WebSocket connects to `ws://localhost:42777` (FAILS - no server there)
   - Fetch requests go to `http://localhost:42777/servers` (FAILS - wrong machine)
   - Demo hub appears broken even though server is running

2. **Violates Web4 Principles**: 
   - Scenarios contain `"host": "McDonges-3.fritz.box"` but demo hub ignores it
   - Server detection works correctly, but client doesn't use detected values
   - Data is correct in model, but UI hardcodes assumptions

3. **Inconsistent with Scenario-Driven Architecture**:
   - Server uses OS detection → writes scenario with real hostname
   - Demo hub fetches scenario data → ignores hostname, uses localhost
   - Breaking the "scenario is source of truth" principle

### **Solution Design**

**Pattern**: Replace all hardcoded `localhost` with `window.location.hostname`

**Rationale**:
- `window.location.hostname` returns the hostname used to access the page
- If user accesses via `localhost`, it returns `localhost`
- If user accesses via `mcdonges-3.fritz.box`, it returns `mcdonges-3.fritz.box`
- Works for LAN IPs too (192.168.1.100)
- Zero configuration needed

**Implementation Strategy**:
1. Create a JavaScript constant at the top: `const CURRENT_HOST = window.location.hostname;`
2. Replace all `localhost` strings with `${CURRENT_HOST}`
3. Keep fallbacks in client URLs (already implemented)
4. Test both localhost and remote access

### **QA Decisions**

**D1: Use `window.location.hostname` (NOT scenario data for demo hub communication)**
- **Why**: Demo hub needs to talk to the PRIMARY server it was served from
- **Alternative**: Parse scenario data from server to get primary hostname
- **Decision**: `window.location.hostname` is simpler, always correct, and automatic

**D2: Keep `server.host` fallback for client URLs (NOT replace with window.location)**
- **Why**: Client servers may run on different hosts/ports than primary
- **Alternative**: Use `window.location.hostname` for all URLs
- **Decision**: Client servers have their own `host` in scenario, use that first

**D3: Single source of truth pattern**
- **Why**: Reduce duplication, make URL construction consistent
- **Alternative**: Hardcode localhost in each fetch/WebSocket call
- **Decision**: Define `CURRENT_HOST` constant once at top of script
