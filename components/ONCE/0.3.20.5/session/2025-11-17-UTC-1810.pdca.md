## 1f.1 PLAN: Define Problem & Solution

**Problem**: Primary server crash recovery test was skipped because in-process `process.kill()` would kill the entire test runner.

**Root Cause**: Servers running as objects within the test process share the same PID space, making it impossible to simulate a true process crash without affecting the test runner itself.

**Goal**: Implement a multi-process test architecture where servers run as separate OS processes, allowing us to test crash recovery scenarios realistically.

**Success Criteria**:
1. Primary server can be killed via SIGKILL without affecting test process
2. Client servers continue running after primary crash
3. New primary server successfully discovers existing client scenarios
4. All processes clean up properly after test completion

**Design Decisions** (User approved):
-  **Decision 1A**: Use `child_process.spawn()` for true process isolation
- **Decision 2A**: Use HTTP health checks (`/health` endpoint) for process verification
- **Decision 3B**: 15-second timeout with 1-second polling intervals for server readiness checks

**Implementation Plan**:
1. Create `spawn-server.mjs` helper script for spawning independent server processes
2. Implement `spawnServer()` function with JSON-based inter-process communication
3. Implement `checkServerHealth()` for HTTP-based readiness verification
4. Implement `waitFor()` helper for condition-based waiting with timeout
5. Replace skipped in-process test with multi-process architecture
6. Add `ONCE_DOMAIN` environment variable support for test domain consistency
7. Track all spawned processes for proper cleanup

## 1f.4 ACT: Standardize & Improve

### What Worked Well:
1. **Multi-process architecture**: `child_process.spawn()` provided true process isolation
2. **JSON IPC**: Single-line JSON output on stdout was reliable and easy to parse
3. **HTTP health checks**: `/health` endpoint provided robust readiness verification
4. **Environment variable override**: `ONCE_DOMAIN` allowed consistent test scenarios
5. **Process tracking**: `serverInstances[]` array ensured complete cleanup

### Issues & Resolutions:
1. **Issue**: Domain mismatch between test and spawned servers (`box.fritz.McDonges-3` vs `local.once`)
   - **Resolution**: Added `ONCE_DOMAIN` env var support, passed `detectedDomain` to spawned servers

2. **Issue**: Scenario directory not found errors in tests
   - **Resolution**: Changed cleanup to preserve directory structure, only delete files

3. **Issue**: Test hangs due to uncleaned client servers
   - **Resolution**: Added `serverInstances[]` tracking and comprehensive `afterEach` cleanup

4. **Issue**: Slow test execution (34s for one test)
   - **Resolution**: Implemented domain detection caching, increased timeout to 40s

### Future Improvements:
1. **Performance**: Optimize `beforeEach` hook to reduce per-test overhead
2. **Timeouts**: Make timeouts configurable via environment variables
3. **Logging**: Add debug mode for multi-process test diagnostics
4. **Reusability**: Extract spawn helpers into shared test utilities

### Standards Established:
- Use `child_process.spawn()` for all multi-process server tests
- Output JSON on single line for inter-process communication
- Use HTTP health checks before test assertions
- Track all spawned processes in cleanup arrays
- Set realistic timeouts (30-40s for multi-server tests)
- Cache expensive operations (like domain detection) across tests
