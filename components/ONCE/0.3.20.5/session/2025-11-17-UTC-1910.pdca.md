# PDCA: Correct Hostname/Domain in Scenarios and Demo Hub URLs

**Date**: 2025-11-17 UTC 19:10  
**Version**: 0.3.20.5  
**Requirement**: Servers must use actual hostname (e.g., "McDonges-3.fritz.box") in scenarios and UI, not "localhost"  
**Type**: CMM3 Defect Fix

## PLAN

### Problem Statement

Despite implementing the correct directory structure (`scenarios/box/fritz/McDonges-3/ONCE/`), the system is still creating scenarios in the old location (`scenarios/local/once/localhost/ONCE/`) with incorrect data:

1. **Scenarios contain wrong values**:
   - `"domain": "local.once"` instead of `"box.fritz"`
   - `"host": "localhost"` instead of `"McDonges-3.fritz.box"`
   - `"hostname": "localhost"` instead of `"McDonges-3"`
   - `"ip": "127.0.0.1"` instead of actual network IP

2. **Demo hub hardcodes localhost**:
   - "Open Client" buttons: `http://localhost:${port}/once`
   - openAllClients(): `http://localhost:${port}/once`
   - openHealth(): `http://localhost:${port}/health`

3. **Server data doesn't include host/hostname**:
   - `fetchServerStatus()` only extracts `port` and `uuid`
   - `handleServerRegistered()` only extracts `port` and `uuid`
   - Missing: `host`, `hostname`, `domain`, `ip`

### Root Cause Analysis

**Why are scenarios created with localhost?**

Looking at scenario file `8211b87e-5b3a-4ff3-bd37-5d9e76f8020c.scenario.json`:
```json
{
  "state": {
    "domain": "local.once",
    "host": "localhost",
    "hostname": "localhost"
  }
}
```

This happens when servers are spawned in **test/demo mode** using `child_process.spawn()`. The spawned Node.js processes detect `os.hostname()` as `"localhost"` instead of the actual FQDN because:

1. **Test environment**: The `spawn-server.mjs` helper spawns servers as child processes
2. **OS hostname detection**: `os.hostname()` returns `"localhost"` in the spawned process context
3. **Detection timing**: The hostname is detected in the constructor before any network interfaces are initialized

The **actual machine** (`McDonges-3.fritz.box`) has the correct detection in non-spawned processes, but spawned servers always get `localhost`.

**Why does demo hub use localhost?**

The demo hub hardcodes `localhost` in:
1. URL construction: `http://localhost:${server.port}/once`
2. Server data extraction: Only stores `port` and `uuid`, not `host`

### Solution Design

#### Option A: Fix Server Data Flow (RECOMMENDED)
1. Update `fetchServerStatus()` to extract `host`, `hostname`, `domain`, `ip` from server registry
2. Update `handleServerRegistered()` to store full server model data
3. Update demo hub URL construction to use `server.host` or fall back to `localhost`
4. Update "Open Client" links to use actual hostname
5. Add IP address display to server cards

**Pros**: 
- Correct data flow from server model → UI
- Supports remote access (if firewall allows)
- Shows actual network topology

**Cons**:
- URLs like `http://McDonges-3.fritz.box:8080/once` may not resolve in all browsers
- Requires DNS or `.local` mDNS resolution

#### Option B: Document Localhost Limitation
- Keep `localhost` for UI URLs
- Document that demo hub only works on local machine
- Accept that scenario files contain `localhost`

**Pros**:
- Simple, no code changes
- Always works (localhost always resolves)

**Cons**:
- Misleading scenario data
- Doesn't support remote access
- Violates "true network state" principle

### Decision: Option A

**Rationale**:
- Scenario files should contain **true** network state
- Demo hub should **attempt** to use real hostnames
- Browser will fall back to localhost if DNS fails
- Aligns with "TRUE Radical OOP" principle (detect reality, don't assume)
