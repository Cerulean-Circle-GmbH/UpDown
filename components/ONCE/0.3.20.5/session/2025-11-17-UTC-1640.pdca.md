### **To TRON: QA Decisions required**

[ ] **Decision 1: Multi-Process Architecture Approach**
   - **Context:** Current test uses `process.kill(primaryPid, 'SIGKILL')` which kills the test process itself because servers run in-process
   - **Options:**
     - **Option A:** Use `child_process.spawn()` to run servers as separate OS processes
       - ✅ Pros: True process isolation, realistic crash simulation, can actually kill server PID
       - ❌ Cons: More complex, requires IPC for test assertions, async communication overhead
     - **Option B:** Use `worker_threads` for thread-level isolation
       - ✅ Pros: Lighter than processes, shared memory possible
       - ❌ Cons: Still same process, cannot use `process.kill()`, less realistic
     - **Option C:** Mock the crash scenario without actual process kill
       - ✅ Pros: Simple, stays in-process
       - ❌ Cons: Not testing real crash recovery, defeats purpose of test
   - **Recommendation:** Option A - Use child_process.spawn() for true process isolation

[ ] **Decision 2: Test Communication Strategy**
   - **Context:** When servers run as separate processes, test needs to verify state
   - **Options:**
     - **Option A:** HTTP health checks to verify server state
       - ✅ Pros: Uses existing /health endpoint, simple, realistic
       - ❌ Cons: Network latency, timing issues
     - **Option B:** File system checks (scenario files)
       - ✅ Pros: Direct verification, no network dependency
       - ❌ Cons: File system race conditions, less direct
     - **Option C:** IPC channels for direct communication
       - ✅ Pros: Fast, reliable, direct state access
       - ❌ Cons: Additional complexity, not how production works
   - **Recommendation:** Option A - HTTP health checks (matches production behavior)

[ ] **Decision 3: Test Timeout Strategy**
   - **Context:** Multi-process tests need appropriate timeouts for startup/shutdown
   - **Options:**
     - **Option A:** 30 seconds total timeout with 2s intervals
       - ✅ Pros: Safe, handles slow systems
       - ❌ Cons: Slow test execution
     - **Option B:** 15 seconds total with 1s intervals
       - ✅ Pros: Balanced, reasonable for CI/CD
       - ❌ Cons: May be tight on slow systems
     - **Option C:** 10 seconds total with 500ms intervals
       - ✅ Pros: Fast feedback
       - ❌ Cons: May cause flaky tests on slow systems
   - **Recommendation:** Option B - 15 seconds with 1s intervals
