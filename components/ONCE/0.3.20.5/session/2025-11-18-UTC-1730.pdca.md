## PLAN

### Problem Statement
The "🔄 Relay via Primary" button is not working on client pages because:
1. `fetchConnectedServers()` is hardcoded to `http://localhost:42777/servers`
2. This fails when browser is on a client server or accessing via FQDN
3. The function should dynamically determine the primary server's address
4. Version is still hardcoded to `0.3.20.3` in relay messages

### Root Cause Analysis
**5 Why Analysis**:
1. **Why** does relay fail? → `connectedServers` array is empty
2. **Why** is it empty? → `fetchConnectedServers()` fetch fails
3. **Why** does fetch fail? → Hardcoded `localhost:42777` doesn't work from remote/client servers
4. **Why** is it hardcoded? → Previous CMM1 hack without proper scenario-based implementation
5. **Why** no scenario? → Browser client doesn't fetch its server's scenario which contains primary server info

**CMM3 Checklist Review**:
- ❌ **Path Authority**: Browser hardcodes URLs instead of getting them from server
- ❌ **DRY**: Duplicates primary server address instead of single source of truth
- ❌ **Environment Agnostic**: Hardcodes localhost instead of dynamic discovery
- ✅ **Git Protocol**: Following proper commit message format
- ❌ **Radical OOP**: Functional approach instead of object-based communication

### Impact Assessment
- **Severity**: HIGH - Core feature (relay pattern) completely broken
- **Scope**: Browser clients on client servers AND remote access to primary
- **User Impact**: Cannot demonstrate relay pattern in demos
- **Technical Debt**: Another hardcoded localhost violation

### Solution Design

#### Option A: Fetch Primary from Current Server's /health Endpoint ✅
**Approach**: 
- Client servers already know their primary (in scenario)
- `/health` endpoint already includes `primaryServer` info (line 136-141 in ServerHierarchyManager.ts)
- Browser fetches `/health` from its connected server to get primary address
- Then fetches `/servers` from that primary

**Pros**:
- ✅ Works for both primary and client server connections
- ✅ Works with FQDN/remote access
- ✅ Uses existing `/health` data structure
- ✅ Single source of truth (server's scenario)
- ✅ TRUE Radical OOP (scenario-driven)

**Cons**:
- Requires two HTTP requests instead of one

#### Option B: Client Servers Proxy /servers Requests to Primary
**Approach**:
- Add `/servers` endpoint to client servers that proxies to primary

**Pros**:
- Single HTTP request from browser

**Cons**:
- ❌ Adds complexity to client servers
- ❌ Violates separation of concerns
- ❌ Redundant code duplication

**Decision**: **Option A** - Use existing `/health` endpoint to discover primary dynamically

**Changes**:

```javascript
// OLD (line 442-464):
async function fetchConnectedServers() {
    const response = await fetch('http://localhost:42777/servers');
    // ...
}

// NEW:
async function fetchConnectedServers() {
    // Step 1: Get primary server info from current server's /health
    const hostname = serverHost.split(':')[0];
    const port = serverHost.split(':')[1] || '8080';
    const healthResponse = await fetch(`http://${hostname}:${port}/health`);
    const healthData = await healthResponse.json();
    
    // Step 2: Determine primary server address
    let primaryHost, primaryPort;
    if (healthData.isPrimaryServer) {
        // We're on the primary
        primaryHost = hostname;
        primaryPort = port;
    } else if (healthData.primaryServer) {
        // We're on a client, get primary from health data
        primaryHost = healthData.primaryServer.host || hostname;
        primaryPort = healthData.primaryServer.port || 42777;
    } else {
        throw new Error('Cannot determine primary server address');
    }
    
    // Step 3: Fetch connected servers from primary
    const response = await fetch(`http://${primaryHost}:${primaryPort}/servers`);
    // ... rest of function
}
```
