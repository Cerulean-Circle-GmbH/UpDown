## **📋 PLAN**

### **1a. Problem Statement**

User reported two critical issues with the browser client (`once-client.html`):

1. **Confusing Connection Status**: Browser at `http://mcdonges-3.fritz.box:8081/once` shows "Connected to mcdonges-3.fritz.box:8081" - this is circular and doesn't show that the **server** is connected to the **primary** on port 42777
2. **Broken Broadcast Buttons**: The "📡 Broadcast to All" button doesn't work anymore

### **1b. Current Behavior**

**Connection Status Display**:
```
🟢 Connected to mcdonges-3.fritz.box:8081 (uuid...)
```
Problem: This only shows browser → server connection, not server → primary connection.

**`fetchServerInfo()` Issues**:
- Line 409: Uses `localhost` instead of actual hostname
- Doesn't fetch or display primary server connection info
- Hardcoded version `0.3.20.3` everywhere (6+ instances)

**Broadcast Function Issues**:
- Hardcoded version `0.3.20.3` in scenario messages
- May not be sending correct message format

### **1c. Expected Behavior**

**Connection Status Should Show**:
```
🟢 Browser → Server: mcdonges-3.fritz.box:8081 (uuid...)
🟢 Server → Primary: mcdonges-3.fritz.box:42777
```

**Broadcast Should**:
- Send correctly formatted scenario messages
- Work for all message types (broadcast, relay, P2P)
- Use correct version (0.3.20.5)

### **1d. Root Cause Analysis**

**Issue 1: Hardcoded localhost**
- Line 409: `http://localhost:${port}/health` ignores actual hostname

**Issue 2: Incomplete Server Info**
- `/health` endpoint returns server's own info but not its primary connection
- Browser client doesn't query for primary connection status

**Issue 3: Hardcoded Version**
- `0.3.20.3` appears in:
  - Title (line 6)
  - Headers (lines 78, 109, 113, 128, 131, 142)
  - Scenario messages (line 268)

**Issue 4: Missing Primary Info Endpoint**
- No way for browser to ask "Who is your primary server?"

### **1e. Goals**

1. Show both browser→server AND server→primary connections
2. Fix hardcoded `localhost` in `fetchServerInfo()`
3. Update all hardcoded versions to dynamic version
4. Verify broadcast buttons work correctly
5. Test end-to-end with real multi-server setup

### **1f. Dependencies & Impact**

**Files to Modify**:
- `src/view/html/once-client.html` - Fix all issues
- `src/ts/layer2/ServerHierarchyManager.ts` - May need to add primary info to `/health` response

**Testing Required**:
- Manual: Open browser clients on different ports
- Manual: Click broadcast buttons
- Automated: Run broadcast-all-servers.test.ts

**Risks**:
- Breaking existing browser client functionality
- Changing `/health` response format might break other consumers

### **1g. CMM3 Violations in Current Code**

1. **Hardcoded Values**: Version strings repeated 6+ times
2. **Hardcoded Infrastructure**: `localhost` instead of scenario-driven hostname
3. **Incomplete Status**: Not showing full connection topology
4. **No Version Injection**: HTML doesn't use dynamic version from server

## **📐 DO**

### **QA Decision 1: How to Show Connection Info?**

**Options**:
- A) Single line showing both connections
- B) Two separate status lines
- C) Expandable connection tree

**Decision**: **B - Two separate lines**
- Clearer for users
- Shows hierarchy (browser → server → primary)
- Easier to style/color code

### **QA Decision 2: How to Get Primary Server Info?**

**Options**:
- A) Add `primaryServer` field to `/health` response
- B) Create new `/primary-info` endpoint
- C) Include in WebSocket connection message

**Decision**: **A - Add to `/health` response**
- Already fetching `/health`
- No extra HTTP request
- Consistent with existing pattern

### **QA Decision 3: Version Injection Strategy?**

**Options**:
- A) Template replacement during HTML serve
- B) Fetch version from `/health` and update DOM
- C) Keep hardcoded (not acceptable)

**Decision**: **B - Fetch and update**
- Already fetching `/health`
- Works with static file serving
- No template engine needed

### **Implementation Plan**

**Step 1: Update `/health` Endpoint**
```typescript
// Add primary server info to health response
primaryServer: this.serverModel.isPrimaryServer ? null : {
    host: this.serverModel.primaryServer?.host || 'localhost',
    port: this.serverModel.primaryServer?.port || 42777,
    connected: !!this.primaryWsClient
}
```

**Step 2: Fix `fetchServerInfo()`**
```javascript
// Use actual hostname from serverHost, not localhost
const hostname = serverHost.split(':')[0];
const port = serverHost.split(':')[1] || '8080';
const response = await fetch(`http://${hostname}:${port}/health`);
```

**Step 3: Update Connection Status Display**
```javascript
// Show both connections
updateConnectionStatus(`🟢 Browser → ${serverHost} (${data.uuid.substring(0,8)}...)`);
if (data.primaryServer && data.primaryServer.connected) {
    addConnectionInfo(`🟢 Server → Primary: ${data.primaryServer.host}:${data.primaryServer.port}`);
}
```

**Step 4: Update All Version References**
```javascript
// Replace all '0.3.20.3' with dynamic version from server
const serverVersion = data.version || '0.3.20.5';
document.title = `ONCE v${serverVersion} Browser Client`;
```

**Step 5: Fix Broadcast Function**
```javascript
// Use dynamic version in scenario messages
version: serverVersion, // Instead of hardcoded '0.3.20.3'
```
