# PDCA: Fix ESM require() causing hostname detection failure

**Date**: 2025-11-18 UTC 12:25  
**Version**: 0.3.20.5  
**Requirement**: Server must detect actual hostname/domain, not fallback values  
**Type**: Critical Bug Fix - ESM/CommonJS Incompatibility

---

## **📋 PLAN**

### **Problem Statement**

User reported scenarios still being saved to `local/once/localhost/` with fallback values even though `os.hostname()` returns FQDN.

**End-to-end test revealed**:
```
os.hostname() = "McDonges-3.fritz.box"  ✅ Correct

But scenario saved with:
domain: "local.once"           ❌ Wrong (should be "box.fritz")
host: "localhost"              ❌ Wrong (should be "McDonges-3.fritz.box")  
hostname: "localhost"          ❌ Wrong (should be "McDonges-3")
ip: "127.0.0.1"               ❌ Wrong (should be actual network IP)
```

### **Root Cause Analysis**

**Debug logging revealed**:
```
🐛 detectHostname() ERROR: ReferenceError: require is not defined
```

**The Issue**: Code uses `require('os')` inside detection functions, but the project uses **ESM** (ES modules), not CommonJS!

```typescript
// BROKEN CODE:
private detectHostname(): string {
    const os = require('os');  // ❌ require() doesn't work in ESM!
    return os.hostname();
}
```

**Why tests passed**: Vitest runs in a different module context that allows `require()`. Real CLI execution is pure ESM.

**Impact**: Every call to `require('os')` throws `ReferenceError`, catch block returns fallback `'localhost'`.

### **Solution Design**

**Fix**: Use proper ESM `import` at top of file instead of `require()` inside functions.

```typescript
// Add to imports:
import os from 'os';

// Use directly:
private detectHostname(): string {
    return os.hostname();
}
```

**Files to fix**:
- `ServerHierarchyManager.ts`: Add `import os from 'os'` and remove all `require('os')` calls

### **Lessons Learned - CMM3 & Web4 Tech Stack**

**What I Did Wrong**:

2. ❌ Didn't do end-to-end testing FIRST when user reported issue
3. ❌ Assumed tests passing meant code was working
4. ❌ Wrote misleading summary claiming "code was already working"

**Web4 Tech Stack Rules** (MUST FOLLOW):
- ✅ **ESM ONLY** - Never use `require()`, always use `import`
- ✅ **Top-level imports** - Import modules at file top, not inside functions
- ✅ **.js extensions** - Always use `.js` in imports even for `.ts` files
- ✅ **Type: module** - package.json must have `"type": "module"`

**What I Should Have Done** (CMM3):
1. ✅ **User reports issue** → Do end-to-end test IMMEDIATELY
2. ✅ **Don't assume** → Test the ACTUAL way users run the code
3. ✅ **Debug systematically** → Add logging to see what's actually happening
4. ✅ **Fix root cause** → Don't just fix symptoms
5. ✅ **Verify fix** → Test end-to-end again after fix
