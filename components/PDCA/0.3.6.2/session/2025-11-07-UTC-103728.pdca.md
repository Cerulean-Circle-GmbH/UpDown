## **📊 SUMMARY**

### **Problem Statement**

**Context:** User identified critical edge case in fixAllPDCAs crawling logic:

**User Feedback:**
> "Thinking about the crawling, I like the way you suggested, but I think we need to add one more reasoning. I think the crawling can stay as it is, but after the rename strip and creationDate, you make a new snapshot. The reason is then the sequence might be different. If rewritePDCA is done before all the files went through creationDate, duallinks might be set that need to change when the creationDate of the next file or the next might change. Does this make sense, or not?"

**The Problem Illustrated:**

```
Initial state (original filenames):
├─ 2025-10-28-UTC-1632-description.pdca.md
├─ 2025-10-29-UTC-2055-description.pdca.md
└─ 2025-10-30-UTC-1200-description.pdca.md

After processing file 1 (strip + creationDate):
├─ 2025-10-27-UTC-1500.pdca.md  ← Moved EARLIER due to creationDate!
├─ 2025-10-29-UTC-2055-description.pdca.md
└─ 2025-10-30-UTC-1200-description.pdca.md

Problem: File 1's "Next PDCA" link now points to 2025-10-29

After processing file 2 (strip + creationDate):
├─ 2025-10-27-UTC-1500.pdca.md
├─ 2025-10-28-UTC-1000.pdca.md  ← NEW position! Should be between!
├─ 2025-10-29-UTC-2055-description.pdca.md (original)
└─ 2025-10-30-UTC-1200-description.pdca.md

WRONG: File 1 still points to 2025-10-29, but should point to 2025-10-28!
```

**Root Cause:**
- `rename creationDate` can move files earlier OR later in chronological order
- `rewritePDCA` generates "Previous PDCA" links based on chronological order
- `fixDualLinks` updates links based on current file structure
- If file order changes DURING processing, early files get wrong links

**Impact:**
- ❌ Broken PDCA chains (wrong Previous/Next links)
- ❌ Files point to wrong neighbors
- ❌ Chain navigation doesn't work

**Solution:** Two-Phase with Re-Snapshot
```
Phase 1: All filename operations (rename strip + creationDate)
  ↓
Phase 1.5: RE-SNAPSHOT (capture new file order with git tracking)
  ↓
Phase 2: All content operations (fixDualLinks, rewritePDCA)
```

### **Phase 2: Two-Phase Implementation** ✅ COMPLETE

**Implementation Summary:**

**Changed:** Refactored `fixAllPDCAs()` method from single-phase to three-phase execution.

**Before (Single Phase):**
```
For each file:
  1. Rename (strip + creationDate)  
  2. Fix content (fixDualLinks/rewritePDCA)
Problem: Early files get wrong links if later files change order
```

**After (Three Phases):**
```
Phase 1: All filename operations
  - For each file: rename strip + creationDate
  
Phase 1.5: Re-snapshot ← NEW!
  - Rescan directory
  - Create new snapshot with updated file order
  - Uses git hash tracking (survives renames)
  
Phase 2: All content operations  
  - For each file: fixDualLinks or rewritePDCA
  - Uses Phase 1.5 snapshot (correct order)
```

**Key Changes:**
1. Split processing loop into two separate phases
2. Added Phase 1.5 re-snapshot between phases
3. Fixed git cwd issue (use file's directory, not projectRoot) for test compatibility
4. Fixed hasDescription regex (don't match `-UTC` in timestamp)
5. Updated console output to show phase labels

**Lines Changed:** ~200 lines in `DefaultPDCA.ts` (lines 5722-5924)

**Commits:**
- Refactor fixAllPDCAs to two-phase with re-snapshot
- Fix git cwd for snapshot compatibility  
- Fix hasDescription regex
- Fix rename creationDate to work in test directories
- Fix rewritePDCA to find chronologically previous PDCA
- Fix TC-FIX-05 and TC-FIX-11 tests

**Additional Fixes (User Request: "Please do both"):**

1. **Fixed `rename creationDate` for test directories:**
   - Problem: Was using `projectRoot` as git cwd, failed in test directories
   - Solution: Use `dir` (file's directory) as git cwd
   - Impact: Tests can now create their own git repos

2. **Fixed TC-FIX-05 timezone handling:**
   - Problem: Git commit dates without explicit timezone were interpreted as local time
   - Solution: Added `+0000` to git commit dates in test
   - Impact: Test now passes consistently

3. **Fixed TC-FIX-11 link verification:**
   - Problem: Test was checking "Next PDCA" links, which `rewritePDCA` doesn't generate
   - Solution: Changed to check "Previous PDCA" links (which `rewritePDCA` does generate)
   - Impact: Test now verifies the actual behavior

4. **Fixed critical bug in `rewritePDCA`:**
   - Problem: Used `findMostRecentPDCAInternal` which finds LATEST file, not previous one
   - Solution: Find file immediately before current file in chronological sorted order
   - Impact: Previous PDCA links now correct after reordering!
   - This was the ROOT CAUSE of why two-phase snapshot was needed!
