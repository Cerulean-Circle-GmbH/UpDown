### **My Answer**
Absolutely correct! The manual cleanup commit (2a7476c0d) should never have been necessary. The bug is that `fixAllPDCAs` leaves old files behind after `rename('strip')` operations. While `git mv` successfully renames files, subsequent operations or restores can resurrect the old files, leaving them in a "deleted" state that requires manual cleanup.

**Learning Applied:** All file lifecycle operations must be programmatic and atomic. `fixAllPDCAs` should handle its own cleanup after Phase 1 operations.

**Objective:** Implement automatic cleanup of old files after strip operation

**Requirements Traceability:** No requirements.md found in component

### **Problem Analysis**

**Current Behavior:**
- `fixAllPDCAs` Phase 1 runs `rename('strip')` on files with descriptions
- `git mv` successfully renames files (e.g., `2025-10-27-UTC-1500.architecture-evolution-analysis.pdca.md` → `2025-10-27-UTC-1500.pdca.md`)
- Old files are left in git's staging area as "deleted" after subsequent operations or hard resets
- Manual `git add -A && git commit` required to clean up

**Root Cause:**
- `fixAllPDCAs` tracks renamed files using git hashes to update `currentPath`
- But it does NOT commit the deletions after the strip phase
- When later operations occur (especially `git restore` for testing), old files are resurrected
- The commit shows BOTH files: old deleted (`-`) and new modified (`+`)

**User Mandate:**
> "Thos deletions should be done in either the rename or in the fixAllPDCAs, but alwys programmatically"
