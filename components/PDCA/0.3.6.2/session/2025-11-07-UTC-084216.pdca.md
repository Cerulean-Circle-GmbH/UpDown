### **TRON Follow-up: Rename Tracking Edge Case (2025-11-07 UTC)**
```quote
This will thus crawl through the folder one PDCA after the other in current time sequence, but what if the creationDate suddenly puts a PDCA after the currently place or before, will the crawler not get confused?
```

**Critical Edge Case Identified:**
```
Initial scan: [070000.pdca.md, 120000.pdca.md, 150000.pdca.md]
Processing 120000.pdca.md:
  → rename('creationDate') discovers git creation was 06:00
  → Renames to: 060000.pdca.md
  
New order: [060000.pdca.md, 070000.pdca.md, 150000.pdca.md]
            ↑ moved BEFORE 070000!

Risk: Crawler could skip 060000 (already past it) or process it twice
```

**TRON Decision: Option A+ - Git Identity Tracking**

**Solution:**
```typescript
// 1. Snapshot at start - collect git object hashes
const fileSnapshots = files.map(f => ({
  originalPath: f,
  gitHash: getGitObjectHash(f)  // Identity survives renames
}));

// 2. Process using git identity (not filename)
for (const snapshot of fileSnapshots) {
  const currentPath = findFileByGitHash(snapshot.gitHash);
  if (currentPath) {
    await processFile(currentPath);
  }
}
```

**Benefits:**
- ✅ Each file processed exactly once (tracked by immutable git hash)
- ✅ Robust against any number of renames (strip, creationDate, manual)
- ✅ No duplicate processing or skipped files
- ✅ Works even if multiple files rename in same batch

**Implementation:** Will use `git log --format=%H -- <file>` to get object hash for identity tracking.
