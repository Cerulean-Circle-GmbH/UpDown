### **Problem Analysis:**

**Current Behavior:**
When `rewritePDCA` processes a corrupted PDCA:
1. Calls `createPDCA` to generate a new clean file (e.g., `temp-2025-11-10.pdca.md`)
2. Merges content from corrupted file into new file
3. Deletes the old corrupted file (e.g., `2025-10-24-UTC-2135.pdca.md`)
4. Renames new file to match original timestamp

**Problem:**
- The git history of the old file ends at the deletion commit
- No record exists linking the new file to the old file's history
- Users lose ability to trace back through the file's evolution before corruption

**Solution:**
Before deleting the old file, get its git object hash and store it in a git note on the new file with key `rewritePDCA.oldFileHash`. This creates a permanent provenance chain.

```typescript
private async preserveOldFileHistory(oldFilePath: string, newFilePath: string): Promise<boolean> {
  // 1. Get git object hash of old file
  const oldFileHash = execSync(
    `git rev-list -1 HEAD -- "${oldRelativePath}"`,
    { cwd: projectRoot, encoding: 'utf-8' }
  ).trim();
  
  // 2. Validate SHA-1 format
  if (!/^[0-9a-f]{40}$/.test(oldFileHash)) {
    throw new Error(`Invalid git hash format: ${oldFileHash}`);
  }
  
  // 3. Add git note to HEAD commit
  execSync(
    `git notes --ref=rewritePDCA.oldFileHash add -m "${oldFileHash}" HEAD`,
    { cwd: projectRoot }
  );
  
  return true;
}
```
