## **📊 SUMMARY**

### **Problem Statement**

**Context:**
- Iteration 1.18 added `createComponentScenario()` to generate `component.json` files during `create` command
- This only affects NEW components created after the fix
- EXISTING components (ONCE 0.3.21.7, etc.) still lack `component.json` files
- These files are required for dynamic `implementationClassName` loading

**Current Gap:**
No CLI command exists to generate `component.json` for existing components without recreating them.

**User Request:**
> "figure out how to generate ca component.scenario.json for components that dp not have one yet. e.g. by introducing web4tscomponent on ONCE latest componentDescriptorUpdate"

**Naming Convention (Radical OOP):**
> "updateComponentDescriptor make it componentDescriptorUpdate that would be more radical OOP and group all actions on componentDescriptors..."

**Pattern:** `{noun}{verb}` groups all operations on the same noun together (e.g., `componentDescriptorUpdate`, `componentDescriptorCreate`, `componentDescriptorValidate`)

**After Iteration 1.19:**
- ✅ `web4tscomponent on ONCE latest componentDescriptorUpdate` generates `ONCE.component.json`
- ✅ Command reuses existing `createComponentScenario()` method
- ✅ Works with any component loaded via `on()` command
- ✅ Creates scenario in `scenarios/{Component}/{Version}/{uuid}.scenario.json`
- ✅ Creates symlink `{Component}.component.json → ../../scenarios/...`

### **Root Cause Analysis**

**Why doesn't this command exist?**
1. `createComponentScenario()` was just added in Iteration 1.18 (today)
2. Previously, NO scenario files were generated at all
3. The feature needs to be exposed as a CLI command for existing components

**Why not just re-run `create`?**
- Would recreate entire component structure
- Risk of overwriting customizations
- Unnecessary file operations

### **Solution Approach**

**Web4 Pattern: Reuse existing methods**
- `componentScenarioCreate()` already exists and works (renamed from `createComponentScenario()` per Principle 16)
- Method requires: `component`, `version`, `componentRoot`
- All available from `this.model.context` (loaded via `on()`)

**Implementation:**
```typescript

  // 1. Validate context (component loaded via on())
  // 2. Extract component metadata from this.model.context
  // 3. Call this.componentScenarioCreate()
  // 4. Report success

```

**Naming Rationale (Radical OOP):**
- ✅ `componentDescriptorUpdate` groups with future methods like `componentDescriptorValidate`
- ✅ Pattern: `{noun}{verb}` - all operations on same noun grouped together
- ❌ `updateComponentDescriptor` spreads related methods across alphabetical listing
