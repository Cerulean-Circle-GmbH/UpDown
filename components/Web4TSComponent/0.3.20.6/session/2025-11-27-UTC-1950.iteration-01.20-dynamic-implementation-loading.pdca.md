## **📊 SUMMARY**

### **Problem Statement**

**Current Situation (Anti-Pattern):**
The CLI uses hardcoded patterns to discover implementation classes:

```typescript
// ❌ HARDCODED ANTI-PATTERN
const classNamesToTry = [
  `Default${componentName}`,  // Standard pattern
  `Node${componentName}`,     // ❌ Hardcoded Node.js-specific
  `${componentName}`,         // Direct name
];
```

**Problems:**
1. ❌ Violates Web4 Principle 1: "Everything is a Scenario"
2. ❌ Hardcoded `Node${Component}` pattern is inflexible
3. ❌ No way to specify custom implementation names (e.g., `BrowserOnce`, `WorkerOnce`)
4. ❌ Ignores `implementationClassName` stored in scenario

**Impact:**
- `upgrade` command fails for components with non-default names
- Cannot support multiple implementations per component
- Configuration must be in code, not in scenario (anti-pattern)

### **Target Situation**

**After Iteration 1.20:**
- ✅ CLI reads `component.json` (symlink to scenario)
- ✅ Extracts `scenario.model.implementationClassName`
- ✅ Uses extracted name to load implementation
- ✅ Fallback: `Default${Component}` if field missing
- ✅ No hardcoded patterns like `Node${Component}`

**Example Flow:**
```typescript
// 1. Read component.json
const scenarioPath = `${componentPath}/${component}.component.json`;
const scenario = JSON.parse(await fs.readFile(scenarioPath, 'utf-8'));

// 2. Extract implementation class name
const className = scenario.model.implementationClassName || `Default${component}`;

// 3. Load the class
const ComponentClass = await import(`${modulePath}`);
const instance = new ComponentClass[className]();
```

### **Root Cause Analysis**

**Why does hardcoded pattern exist?**
1. Historical: Predates `component.json` scenario files
2. No mechanism existed to store implementation class name
3. Convention-over-configuration taken too far (hardcoded instead of discoverable)

**Why is this the right time to fix?**
1. ✅ Iteration 1.18 added scenario file generation
2. ✅ Iteration 1.19 added `componentDescriptorUpdate` command
3. ✅ All components can now have `component.json` files
4. ✅ `implementationClassName` field exists in scenarios

### **Solution Approach**

**Web4 Principle 1: "Everything is a Scenario"**
> Configuration IS scenario state. No separate config files.

**Implementation Strategy:**
1. Locate file where hardcoded `classNamesToTry` exists
2. Add scenario file reading logic
3. Extract `implementationClassName` from scenario
4. Remove hardcoded `Node${Component}` pattern
5. Keep `Default${Component}` as ONLY fallback

```typescript
private async componentScenarioRead(
  componentPath: string,
  componentName: string
): Promise<any | null> {
  try {
    const scenarioPath = join(componentPath, `${componentName}.component.json`);
    if (!existsSync(scenarioPath)) {
      return null; // Scenario file doesn't exist - use fallback
    }
    
    const scenarioJson = await import('fs/promises').then(fs => fs.readFile(scenarioPath, 'utf-8'));
    return JSON.parse(scenarioJson);
  } catch (error) {
    // Error reading/parsing scenario - use fallback
    return null;
  }
}
```
