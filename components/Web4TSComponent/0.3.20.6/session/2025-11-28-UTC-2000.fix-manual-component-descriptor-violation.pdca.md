## PROBLEM STATEMENT

### Critical CMM3 Violation

The assistant manually created files in ONCE 0.3.21.7 instead of using the CMM3 toolchain, causing:

1. **Missing CLI Script**: `scripts/versions/once-v0.3.21.7` does not exist
2. **Wrong File Permissions**: `components/ONCE/0.3.21.7/once` has incorrect permissions
3. **Process Violation**: Manual file creation bypasses CMM3 quality gates and automation
4. **Wrong Version**: Created descriptor in 0.3.21.7 instead of 0.3.21.6

### Root Cause

**Real Root Cause** (corrected per user feedback):

1. **Renamed Implementation Class**: ONCE changed from `DefaultONCE` to `NodeJsOnce`
2. **Missing Feature**: `implementationClassName` field was not yet implemented in component scenarios
3. **Broken Upgrade Tool**: Without `implementationClassName`, `once upgrade nextBuild` could not function because Web4TSComponent CLI couldn't find `NodeJsOnce.js` (it was looking for `DefaultONCE.js`)
4. **Half-Way Manual Workaround**: Assistant tried to manually create files in 0.3.21.7 to work around the broken upgrade process
5. **Result**: 0.3.21.7 is in a broken half-way state - partially created by upgrade tool, partially manual

**Key Insight**: The real problem was the missing `implementationClassName` feature, not process violation. The process violation was a symptom, not the cause.

## ROOT CAUSE ANALYSIS

### Why The Upgrade Failed

1. **Class Rename**: ONCE implementation renamed `DefaultONCE` → `NodeJsOnce`
2. **Missing Metadata**: No `component.json` to tell CLI which class to load
3. **CLI Fallback Logic**: Falls back to `Default{Component}` pattern when no scenario exists
4. **Result**: CLI looked for `DefaultONCE.js` (doesn't exist) instead of `NodeJsOnce.js` (exists)

### Why Manual Creation Failed

1. **Partial Toolchain**: Manual file creation can't replicate all upgrade tool steps
2. **Missing Steps**: 
   - CLI script generation
   - File permission setting
   - Package.json updates
   - README updates
3. **Half-Way State**: Some files from upgrade tool, some manually created = inconsistent state

### CMM3 Principle Violated

**Web4 Principle 0: CMM3 Process Authority**
> "Never manually create artifacts generated by CMM3 tools. Tools ensure consistency, permissions, and process adherence."

## LESSONS LEARNED

### What Went Wrong

1. **Feature First, Then Upgrade**: Missing `implementationClassName` feature broke upgrade tool
2. **Manual Workaround Created Mess**: Half-way manual fixes create inconsistent state
3. **Wrong Version Target**: Tried to work in 0.3.21.7 instead of fixing 0.3.21.6 first

### Correct Approach

1. **Fix Current Version First**: Always add features to CURRENT working version
2. **Create Descriptor Before Upgrade**: Component descriptor must exist before upgrade
3. **Delete Broken State**: Don't try to fix half-way state - delete and recreate
4. **Trust CMM3 Tools**: Upgrade tool handles all complexity (permissions, scripts, symlinks)
5. **Bootstrap Pattern**: One-time bootstrap scenario in current version is acceptable

### Root Cause Insight (User Correction)

**Original (Wrong)**: Process violation caused the problem  
**Corrected (Right)**: Missing feature caused upgrade to fail, which led to manual workaround, which led to process violation

**Key Learning**: Symptoms != Root Cause. The process violation was a symptom of the missing feature.


### New CMM3 Rules

**Rule 1: Feature First, Upgrade Second**
> "Never upgrade a component until all features needed for the upgrade are implemented in the current version."

**Rule 2: Bootstrap Pattern**
> "The ONLY manual creation allowed is a bootstrap scenario in the CURRENT version to enable initial feature implementation. All other artifacts MUST be created by CMM3 tools."

**Rule 3: Delete Broken State**
> "If manual workarounds create a broken half-way state, DELETE completely and recreate using proper CMM3 tools. Never try to 'fix' a corrupted state."
