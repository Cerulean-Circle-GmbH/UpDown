## 5. Key Research Findings

### 5.1 Component.json Files Are NOT Auto-Generated
**CRITICAL DISCOVERY:**
- `component.json` files are **scenario outputs** from `toScenario()`
- They are **NOT automatically written** during component creation
- Web4TSComponent only writes: `package.json`, `tsconfig.json`, `vitest.config.ts`, shell scripts, TypeScript files
- **component.json is a RUNTIME artifact**, not a build artifact

**Evidence:**
- Searched all of Web4TSComponent: NO code writes `*.component.json`
- `toScenario()` returns the scenario object, doesn't persist it
- Component scenarios are persisted by OTHER systems (servers, scenario managers)

### 5.2 Constructor.name Pattern in ONCE
**From ONCE codebase analysis:**
```typescript
// NodeJsOnce.ts line 46

  // Multiple uses of this.constructor.name:
```

**Key Insight:**
- ONCE already uses `this.constructor.name` to get actual class name
- Pattern: `this.constructor.name` → "NodeJsOnce" (not "DefaultONCE")
- This IS the solution for Web4TSComponent!

## 6. Proposed Solution (Revised)

### 6.1 NOT Model Field - Method Instead!
**Why NOT add to model:**
- `implementationClassName` is META-information about the code
- Models are for DOMAIN state, not reflection metadata
- Would pollute every component's model with deployment info

**Better: Component Method Pattern**
```typescript
// In Component.interface.ts
export interface Component<TModel extends Model = Model> {
  // ... existing methods ...
  
  /**
   * Get the actual implementation class name for this component
   * Used by CLI for dynamic imports
   * Default: "Default{ComponentName}"
   * Override: Return this.constructor.name for custom implementations
   */
  getImplementationClassName(): string;
}
```

### 6.2 DefaultCLI Enhancement
```typescript
private async loadComponent(
  componentName: string, 
  version: string,
  componentPath: string
): Promise<Component> {
  // ✅ Strategy 1: Try common patterns
  const classNamesToTry = [
    `Default${componentName}`,  // Standard pattern
    `Node${componentName}`,     // Node.js-specific
    `${componentName}`,         // Direct name
  ];
  
  let ComponentClass: any;
  let modulePath: string;
  
  for (const className of classNamesToTry) {
    modulePath = join(componentPath, 'dist', 'ts', 'layer2', `${className}.js`);
    
    if (existsSync(modulePath)) {
      try {
        const module = await import(modulePath);
        ComponentClass = module[className];
        
        if (ComponentClass) {
          console.log(`   ℹ️  Loaded ${className} from ${path.basename(modulePath)}`);
          break;
        }
      } catch (error) {
        // Try next pattern
        continue;
      }
    }
  }
  
  if (!ComponentClass) {
    throw new Error(`Component class not found for ${componentName}. Tried: ${classNamesToTry.join(', ')}`);
  }
  
  // ... rest unchanged
}
```

## 10. CRITICAL: DRY Violation Discovered

**TRON's Observation:**
> `"componentPath": "/Users/.../Web4TSComponent/0.3.20.2"` is wrong.  
> The intention is `"componentsDirectory": "/Users/.../components"`  
> `componentPath` seems to be a duplicate and violates Web4 DRY

**Analysis:**

From `Web4TSComponentModel.interface.ts:115-121`:
```typescript
/**
 * Calculated path to TARGET component root (this OR context)
 * Example: /Users/.../components/IdealMinimalComponent/0.3.18.2
 */
targetComponentRoot?: string;

/**
 * Alias for targetComponentRoot (backward compatibility)
 * @pdca 2025-11-07-UTC-0000.eliminate-path-duplication-all-cases.pdca.md
 */
componentPath?: string;
```

**Key Findings:**
1. ✅ `componentPath` is ALREADY MARKED as deprecated alias
2. ✅ `targetComponentRoot` is the correct field name
3. ✅ Prior PDCA (2025-11-07) attempted to eliminate this duplication
4. ❌ **BUT**: The field is still in the model and still being used!

**DRY Violations:**
```typescript
// Model has BOTH fields (violation):
model: {
  componentsDirectory: "/Users/.../components",       // ✅ Correct: points to components/
  targetComponentRoot: "/Users/.../Component/0.x.x",  // ✅ Correct: full path to specific version
  componentPath: "/Users/.../Component/0.x.x"         // ❌ DUPLICATE of targetComponentRoot
}
```

**Function Parameters Also Use Wrong Name:**
```typescript
// DefaultWeb4TSComponent.ts line 871
async auditComponentCompliance(componentPath: string): Promise<any>

// DefaultWeb4TSComponent.ts line 1412  
async from(componentPath: string): Promise<this>

// DefaultWeb4TSComponent.ts line 3552
private async analyzeComponentStructure(componentPath: string, ...)

// DefaultCLI.ts line 327
private async loadComponent(componentName, version, componentPath: string)
```

**Correct Pattern Should Be:**
```typescript
// ✅ Use targetComponentRoot everywhere:
async auditComponentCompliance(targetComponentRoot: string)
async from(targetComponentRoot: string)
private async analyzeComponentStructure(targetComponentRoot: string, ...)
private async loadComponent(componentName, version, targetComponentRoot: string)

// ✅ Model should NOT have componentPath at all:
model: {
  componentsDirectory: string;      // Base: /project/components
  targetComponentRoot: string;      // Specific: /project/components/Name/0.x.x
  // componentPath: REMOVE THIS
}
```
