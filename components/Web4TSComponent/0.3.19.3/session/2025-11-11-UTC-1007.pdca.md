## 📋 **1. PLAN**

### 1a. Goal
Analyze Web4 principles, architecture patterns, and tech stack to identify rules that can be enforced via ESLint, preventing violations at build time rather than discovering them during code review.

### 1b. Context
After implementing strict `no-unused-vars` and underscore prohibition rules, we recognize ESLint as a powerful tool for enforcing Web4 architectural principles. The PDCA training system provides comprehensive documentation of Web4 patterns, making this an ideal time to systematically identify enforceable rules.

## 📋 **QA Decisions**

**TEMPLATE VERIFICATION: QA Decisions section verified**

**Q1: Should we implement ALL identified rules immediately, or prioritize incrementally?**  
**Decision:** ✅ Prioritize incrementally  
**Rationale:** Some rules may have false positives or require refinement. Implement high-impact, low-risk rules first (already done: CommonJS, underscore prefix), then add more sophisticated rules in subsequent versions. This allows testing and validation without disrupting existing development.

**Q2: Should rules apply to test files with the same strictness as production code?**  
**Decision:** ❌ NO - Test files get relaxed rules  
**Rationale:** Test files often need `any` types for mocking, may use different patterns for test data setup, and should focus on clarity over strict architectural compliance. Already configured in `eslint.config.mjs` with separate rule set for `**/*.test.ts`.

**Q3: Should we enforce "no flags" (Web4 positional parameters) via ESLint?**  
**Decision:** ❌ NO - Not ESLint's domain  
**Rationale:** This is a CLI design pattern, not TypeScript syntax. ESLint operates on TypeScript AST and cannot detect CLI usage patterns. Better enforced through CLI framework design and code review.

**Q4: Should we try to enforce "empty constructor" pattern via ESLint?**  
**Decision:** ⚠️ DEFER - Complex, high false-positive risk  
**Rationale:** While enforceable (detect constructors with parameters), this rule would need sophisticated understanding of component vs. utility classes. Too many false positives without context awareness. Better enforced through code review and documentation.

**Q5: Should we enforce import.meta.url over __dirname/__filename?**  
**Decision:** ✅ YES - Already done  
**Rationale:** This is a critical Web4 violation (underscore naming + CommonJS pattern). Already enforced via `no-restricted-globals` rule for `__dirname` and `__filename`.

#### **Category B: Naming Conventions**
- ✅ **ENFORCED**: No underscore prefix `_` - Indicates unused, must be removed
- ⚠️ **ENFORCEABLE**: No underscore in ANY identifier (not just prefix)
- ⚠️ **ENFORCEABLE**: PascalCase for classes/interfaces
- ⚠️ **ENFORCEABLE**: camelCase for variables/functions
- ❌ **NOT LINTABLE**: Semantic naming (meaningful names)

#### **Category C: Async/Promise Patterns**
- ✅ **ENFORCED**: No floating promises - Must await or catch
- ✅ **ENFORCED**: Await thenable expressions only
- ✅ **ENFORCED**: No async promise executor
- ✅ **ENFORCED**: Prefer promise reject errors

#### **Category D: TypeScript Strictness**
- ✅ **ENFORCED**: Consistent type imports (`import type`)
- ⚠️ **WARNING**: Explicit function return types
- ⚠️ **WARNING**: No explicit `any` (allowed for reflection/dynamic code)
- ✅ **ENFORCED**: No unused variables/parameters

#### **Category E: Code Quality**
- ✅ **ENFORCED**: No `var` - Use `const`/`let`
- ✅ **ENFORCED**: Prefer `const` over `let`
- ✅ **ENFORCED**: Prefer arrow callbacks
- ✅ **ENFORCED**: Object shorthand syntax
- ✅ **ENFORCED**: Prefer template literals
- ✅ **ENFORCED**: Strict equality (`===`)
- ✅ **ENFORCED**: No debugger statements

#### **Category F: Radical OOP Patterns**
- ❌ **NOT LINTABLE**: Empty constructors (too many false positives)
- ❌ **NOT LINTABLE**: Method chaining return types (requires context)
- ❌ **NOT LINTABLE**: Model-driven state (requires semantic understanding)
- ❌ **NOT LINTABLE**: CLI auto-discovery annotations (documentation pattern)

#### **Category G: Tech Stack Compliance**
- ⚠️ **ENFORCEABLE**: No Jest imports (BANNED - use Vitest)
- ⚠️ **ENFORCEABLE**: No `ts-jest` (BANNED)
- ⚠️ **ENFORCEABLE**: No `jest.config` files
- ❌ **NOT LINTABLE**: Vitest-only testing (requires package analysis)

### Phase 2: Rule Design

#### **New Rule 1: Strict Underscore Prohibition (Anywhere)**

**Current Status:** Partially enforced (prefix only)

**Proposed Enhancement:**
```javascript
{
  selector: 'Identifier[name=/_/]',  // ANY underscore, not just prefix
  message: '🚫 Underscore in ANY identifier is PROHIBITED. Web4 naming: camelCase/PascalCase only. No snake_case, no _prefix, no suffix_.',
}
```

**Rationale:** Web4 strictly forbids underscores in naming. Current rule only catches prefix (`^_`), but `snake_case` and `suffix_` are equally violations.

**Impact:** 
- **Risk:** HIGH - Will flag `__dirname`, `__filename` (already caught), `snake_case` variables
- **Value:** HIGH - Enforces Web4 naming convention completely
- **False Positives:** LOW - Underscore has no legitimate use in Web4

**Examples:**
```typescript
// ❌ VIOLATIONS
const user_name = 'test';          // snake_case
const isValid_ = true;             // suffix
const _helper = () => {};          // prefix (already caught)
const __proto__ = {};              // double underscore

// ✅ CORRECT
const userName = 'test';           // camelCase
const isValid = true;              // camelCase
const helper = () => {};           // camelCase
```

#### **New Rule 2: Ban Jest (CRITICAL - Tech Stack Violation)**

**Status:** NOT enforced

**Proposed Rule:**
```javascript
'no-restricted-imports': [
  'error',
  {
    name: 'jest',
    message: '🚫 Jest is BANNED. Use Vitest instead: import { describe, it, expect } from "vitest"',
  },
  {
    name: '@jest/globals',
    message: '🚫 Jest is BANNED. Use Vitest instead: import { describe, it, expect } from "vitest"',
  },
  {
    name: 'ts-jest',
    message: '🚫 ts-jest is BANNED. Use Vitest with native TypeScript support.',
  },
],
```

**Rationale:** Jest is explicitly BANNED in tech stack documentation. Vitest is the approved testing framework (ESM-native, TypeScript-first).

**Impact:**
- **Risk:** LOW - Jest should not exist in codebase
- **Value:** CRITICAL - Prevents tech debt introduction
- **False Positives:** NONE - Jest is never acceptable

**Examples:**
```typescript
// ❌ VIOLATIONS
import { describe, it, expect } from 'jest';
import '@jest/globals';
import { JestConfigWithTsJest } from 'ts-jest';

// ✅ CORRECT
import { describe, it, expect } from 'vitest';
```

#### **New Rule 3: Enforce Vitest-Only Test Imports**

**Status:** NOT enforced

**Proposed Rule:**
```javascript
// In test file context only
'no-restricted-syntax': [
  'error',
  {
    selector: 'ImportDeclaration[source.value=/^(jest|@jest\\/globals|ts-jest)/]',
    message: '🚫 Jest imports are BANNED. Use Vitest: import { describe, it, expect } from "vitest"',
  },
],
```

**Rationale:** Complement to `no-restricted-imports`, catches dynamic imports and ensures test files use Vitest.

**Impact:**
- **Risk:** LOW - Overlaps with Rule 2
- **Value:** MEDIUM - Additional safety net
- **False Positives:** NONE

#### **New Rule 4: No CamelCase with Underscores (Hybrid Pattern)**

**Status:** NOT enforced

**Proposed Rule:** (Covered by enhanced Rule 1)

**Impact:**
- **Risk:** LOW - Already covered by strict underscore prohibition
- **Value:** N/A - Redundant with Rule 1
- **Decision:** Use enhanced Rule 1 instead
