### **To TRON: QA Decisions Required**

**Q1: Should ESLint run on auto-rebuild (when files change)?**  
**Decision:** ❌ NO - Only on explicit builds  
**Rationale:** Auto-rebuild happens frequently during development (every file save). Running ESLint on every save would slow down the development loop. Explicit builds (with arguments like `verbose`, `force`) are intentional actions where linting is appropriate.

**Q2: Should linting apply to all components from project root?**  
**Decision:** ❌ NO - Per-component only  
**Rationale:** Project-wide linting across all components is extremely slow (~minutes). Per-component linting is fast (~3s) and sufficient for build-time enforcement.

**Q3: Should ESLint configuration use CommonJS or ES modules?**  
**Decision:** ✅ ES Modules (eslint.config.mjs)  
**Rationale:** ESLint 8.x doesn't support ES modules in `.eslintrc.js`, but supports flat config format with `.mjs` extension. This avoids CommonJS regression and maintains TRUE Radical OOP principles.

**Q4: Should `initProject` create ESLint configuration?**  
**Decision:** ✅ YES  
**Rationale:** New projects should inherit linting enforcement from the start. Templates ensure consistency across all Web4 projects.

## **📋 PLAN**

**Objective:** Integrate ESLint into Web4TSComponent build process to automatically enforce TRUE Radical OOP principles (empty constructors, model-driven state, ES modules, method chaining)

**Requirements Traceability:**
- User requested: "write a pdca about how to enforce radical oop with a linter configuration"

- Architectural goal: Prevent regressions to CommonJS and enforce Web4 coding standards

**Implementation Strategy:**

### **Phase 1: Root-Level Configuration**
1. Create `eslint.config.mjs` with flat config format (ES module compatible)
2. Configure Radical OOP rules:
   - `@typescript-eslint/consistent-type-imports` - Force `import type` for type-only imports
   - `@typescript-eslint/no-unused-vars` - Enforce `_` prefix for intentionally unused variables
   - `@typescript-eslint/no-explicit-any` - Discourage `any` types (warning level)
   - `prefer-const` - Enforce immutability where possible
   - `object-shorthand` - Modern ES6+ syntax
3. Create `.eslintignore` to exclude build artifacts and dependencies
4. Update root `package.json` with ESLint dependencies and scripts

### **Phase 2: Per-Component Build Integration**
1. Modify `build.sh` to run ESLint before TypeScript compilation
2. Add `nolint` parameter to skip linting when needed
3. Make linting respect `verbose` flag (detailed vs compact output)
4. Restrict linting to explicit builds (not auto-rebuild)
5. Scope linting to current component only (`src/**/*.ts`)

### **Phase 3: Template Propagation**
1. Update `build.sh.template` with linting integration
2. Create `eslint.config.mjs.template` for new projects
3. Create `.eslintignore.template` for new projects
4. Update `root-package.json.template` with ESLint dependencies

### **Phase 4: initProject Integration**
1. Modify `DefaultWeb4TSComponent.ts` `initProject()` method
2. Create ESLint config files from templates
3. Use existing `syncFileFromTemplate` mechanism

### **Phase 5: Documentation**
1. Create `LINTING.md` with project-wide linting strategy
2. Document per-component vs project-wide linting tradeoffs
3. Explain `nolint` parameter usage

**Expected Output:**
- ESLint enforces Radical OOP on every explicit build
- Build fails with clear error messages on violations
- New components inherit linting via templates
- New projects get ESLint setup via `initProject`
- Fast per-component linting (~3s per component)
- Developers can skip linting with `nolint` parameter
