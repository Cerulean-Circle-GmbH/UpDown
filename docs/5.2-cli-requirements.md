
# 5.2 CLI Command Requirements

## ESM-Only Principle
- All CLI, scripts, and automation must use modern ES Modules (ESM) everywhere. Never use CommonJS (`require`, `module.exports`).
- All imports must use ESM syntax and resolve correctly with `ts-node --esm`.
- Never use `main.ts` as a CLI entry point; always use a dedicated CLI file (e.g., `GitScrumProject.cli.ts`).

## Principles
- All CLI commands must use positional arguments only.
- No Linux-style options (e.g., `--help`, `-v`).
- The first argument is always the class or command (e.g., `help`, `create`).
- The second and subsequent arguments are method parameters.
- The CLI must parse these arguments and invoke the corresponding class method.
- Example: `oosh help` → `OOSH.help()`
- Example: `oosh create ProjectName` → `OOSH.create('ProjectName')`
- All help and documentation must be accessible via class methods, not options.

## Example Usage
- `oosh help` → Calls `OOSH.help()` and prints help to the console.
- `oosh create ProjectName` → Calls `OOSH.create('ProjectName')`.

## Implementation Notes
- The CLI entry point must be a pure loader, delegating to static class methods.
- All shell wrappers must invoke TypeScript entry points using `ts-node` with ESM and correct project root.
- All completions and automation must be code-driven and reflect the current codebase.
